# 三层 AGENT 体系 · 我为什么选三层

> **状态**：stable（迭代中）
> **本文是什么**：聊聊"全局 / 项目 / 模块"三层 AGENT 划分背后的思考。
> **不是什么**：唯一正确答案——这只是我的选择，你的项目可能适合两层 / 四层 / 完全不分层。

---

## 核心想法

**问题**：协作偏好规则放哪？全堆一个文件 → 文件爆炸 / AI 找不到关键规则。完全散到各文件 → 没有"先看哪里"的入口。

**我的解法 · 三层**：
- **全局层**：跨项目通用、与技术栈无关的协作偏好（约 100 行）
- **项目层**：当前项目特定的工作方式 + 配置 + 钩子（约 100 行）
- **模块层**：单个模块的契约 + 指向 spec 的指针（形式可变，不限行数）

**优先级反向**：模块层 > 项目层 > 全局层。**越具体越胜出**——因为一线最了解当下。

---

## 为什么不是两层 / 四层

**两层（全局 + 项目）的问题**：
- 项目内不同模块的契约会撞——比如 combat 模块"无状态纯函数"vs UI 模块"事件驱动"，硬塞同一个项目层文件冲突
- 模块演化历史无处放
- 跨模块改动时不知道该看哪个模块的契约

**四层（全局 + 项目 + 子项目 + 模块）的问题**：
- 子项目层在大多数项目里多余（除非 monorepo 里有多个子项目）
- 用户找规则时要扫 4 层，认知负荷高
- 中间层"该写什么"边界模糊——容易和上下层重复

**三层是个甜蜜点**：覆盖大多数项目结构，认知负荷可控，每层职责清晰。

---

## 三层各自的边界

### 全局层（`<workspace>/AGENT.md`）

**WHAT**：跨项目通用协作偏好
**典型内容**：
- 回复语言 / 路径风格 / 编码格式
- "先读再改"等通用工作原则
- AI 行为规则（信念度标注 / 自检义务 / 工具失败的退化策略）
- 跨项目通用的协作姿态（清晰简洁 / 客观中立 / 直接指出错误）

**禁止内容**：
- 任何具体技术栈名
- 任何具体业务术语
- 任何具体目录 / 文件名

**行数预算**：约 100 行（强信念）。理由：长了 AI 在每次会话开头读它会浪费 context；同时强迫归类抽象。

**配套文件**：当 AGENT.md 接近 100 行预算时，可拆分出：
- `AI_SELF_CHECK.md`：文档健康度自检扫描清单（AGENT §文档健康度扫描 指向它）
- `AI_SOFT_HINTS.md`：AI 行为"软提示"——不是硬规则，是降低犯错概率的提醒

这些配套文件不是独立的"层"，而是全局 AGENT 的工程化延伸——AGENT 用指针引用它们，保持自身精简。

### 项目层（`<workspace>/<project>/AGENT.md`）

**WHAT**：当前项目特定的工作方式
**典型内容**：
- 引擎 / 语言 / 项目类型
- 架构约束 / 状态中枢 / 事件机制
- 项目特定的协作钩子（编译验证命令、lint 命令）
- 指针到 spec / checklist / 维护流程

**禁止内容**：
- 跨项目通用的偏好（应进全局层）
- 单个模块的契约（应进模块层或 spec）

**行数预算**：约 100 行。超了就抽段留指针。

**与全局层的关系**：全局 AGENT 写"协作姿态"（清晰简洁 / 不奉承 / 直接指出错误），项目层 AGENT 不再重复这些，只需在开头注明"遵循工作区根 AGENT.md 的全局协作偏好"。这避免了「全局 AGENT + 项目层 COLLABORATION.md」两份文件各自维护的冗余——**协作姿态是全局偏好，不应按项目拆分。**

### 模块层（`<project>/<module>/README.md` + spec 目录）

**WHAT**：单个模块的契约 + 接口定义

**与原始设计的演变**：原始三层设计设想模块层用 `AGENT.md`，但实战中发现模块契约更适合拆成两块：
- **README**：模块职责概述 + 快速索引（路由器）
- **spec/**：详细契约（权威定义），每个模块一份 spec 文件，可写 200+ 行不受行数预算限制

这个拆分比「一份 AGENT.md 装一切」更合理——spec 可以写长、可以独立 review、可以被其他模块引用时精确指向某个文件。

**典型内容**（README）：
- 模块的 WHAT / WHY / HOW（简明）
- 公共 API 与边界
- 指向 spec/ 中该模块的详细契约
- 指向项目 CHANGELOG 中该模块的历史变更

**禁止内容**：
- 全局或项目层已写的通用规则（不重复）
- 其他模块的契约（应放在那个模块的 README / spec）

**行数预算**：README 50~100 行（路由职责）。spec 按需写，不受限制。

## 三层挂钩的通用模式

一个完整的三层挂钩结构：

| 层 | 写什么 | 行数 |
|---|---|---|
| **全局层** | 思想——跨项目通用原则 | 1~2 行原则 + 1 行链接 |
| **项目层** | 骨架 + 指针——本项目的具体实施 | 1 行钩子指向 checklist / spec |
| **Checklist / spec** | 细节——可随实战不断长的扫描表 / 契约定义 | 不限 |

**好处**：
- 全局层轻量，思想跨项目可复用
- 项目层只需一个钩子，认知负荷小
- 细节层可以随实战不断长，不污染上层
- 模块层保持轻量（README 做路由器）

如果硬塞到单层，全局层会爆预算，且未来其他项目用不上项目特定的细节。如果全 push 到项目层，跨项目时思想消失。三层挂钩让"思想 / 实施 / 细节"各得其所。

---

## 优先级反向（"模块 > 项目 > 全局"）

**为什么反向**：

- 全局层是"我希望所有项目这样"的偏好
- 项目层是"这个项目实际这样"的事实
- 模块层是"这个模块在这个项目里实际这样"的更具体事实

当冲突时，**事实胜出偏好**——一线最了解当下。

**典型例子**：
- 全局层说"路径优先用绝对路径"
- 项目层说"本项目内部链接用相对路径"
- 模块层 spec 说"本模块的资源 preload 用绝对路径"

三个全成立，按"模块覆盖项目，项目覆盖全局"逻辑应用。

---

## 两个正交的"分层"维度

本文的三层是**位置维度**的分层——全局 / 项目 / 模块，按文件所在位置路由。

还有一个**归属维度**的分层——AGENT 侧（画像、心跳、记忆）vs 项目侧（spec、CHANGELOG、checklist）。位置和归属是两个正交的维度：PRODUCER_PROFILE 放在全局层的位置，但它属于 AGENT 侧；项目 CHANGELOG 放在项目层的位置，属于项目侧。

详见 `element-decoupling-and-assembly.md`——那里把框架拆成了"零件市场"，按场景选零件，不按位置全装。

## 给"想搭框架的人"的建议

- **从全局层开始**：先写全局层 30~50 行的"我希望 AI 大致这样工作"。等遇到具体项目了再写项目层
- **项目层不是模板**：不要"复制其他项目的项目层改改"。每个项目的工作方式不同，硬复制反而误导
- **模块层按需建**：不是每个模块都要建。有契约要立 / 有接口要明确 / 有跨模块引用才建
- **模块层载体可变**：小模块可以用 README 单文件；大模块用 README + spec 拆分；不强制 AGENT.md 作为模块层命名。关键是职责被装在哪
- **checklist 是项目层的延伸**：容量不够时把项目层超出的内容抽到 checklist
- **COLLABORATION.md 可以合并到全局 AGENT**：协作姿态（清晰简洁 / 不奉承 / 直接指出错误）本质上是跨项目通用的。如果已经有一份全局 AGENT，没必要再维护一份 COLLABORATION 文件来装同样的内容
- **优先级反向是关键**：很多人会写"必须按全局层执行"——这是反向了，会让一线无法适应
- **先问场景再选层数**：三层是把"位置维度"拆到刚好够用的甜蜜点。但如果你还没有多个项目，从一层开始就够了——详见 `element-decoupling-and-assembly.md` 的场景组合

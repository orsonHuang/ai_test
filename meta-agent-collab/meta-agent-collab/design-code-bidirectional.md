# 设计-代码双向影响 · 设计文本不是"指南"，是协作产物

> **状态**：draft（重塑中）
> **本文是什么**：聊聊"代码实现过程中发现的问题，反过来如何修正设计文本"——双向流动而非单向。
> **不是什么**：DDD / Spec-driven development 教程。

---

## 核心想法

**反直觉的事**：很多人以为设计文本是上游，代码是下游，单向流动。但实际上：**代码实现过程中会暴露设计文本里没想清楚的地方**——这时候应该反向修正设计，而不是"代码硬实现已经写错的设计"。

**好的协作不是"按设计文档逐字实现"，是"代码和设计文本互相校准"**。代码实现是设计意图的最严格 review——任何模糊、矛盾、遗漏都会在写代码时立刻暴露。这种暴露应该 **回流** 到设计文本，让后续工作不再踩同坑。

---

## 为什么单向流动不行

如果只许"设计 → 代码"单向：

1. **设计永远落后于真相**：代码改了、设计没改，过段时间设计文本变成"参考但不准"
2. **新人被误导**：下一个看设计的人按字面理解，撞到代码层面"早就解决但没回流"的坑
3. **设计师失去 feedback**：好的设计应该被代码反复打磨——拒绝 feedback 意味着设计永远停留在"看起来合理"
4. **AI 协作时的特殊问题**：AI 看设计文本制定 plan，如果设计已偏离代码，plan 就会基于错误前提

---

## 双向流动的四种典型场景

**场景 1 · 设计文本表述不一致 → 代码强迫拍板**：
设计文本存在歧义或多种理解方式。代码实现时必须选一种。**不应该是"代码自己挑一种实现"，而是把这个不一致 bubble up 给设计层拍板。**

**场景 2 · 设计文本字面与意图冲突 → 代码触发修订**：
设计文本的字面意思在实际运行中与设计意图冲突。**不应该是"按字面写然后摆烂"，而是回头修设计文本。**

**场景 3 · 设计文本遗漏 → 代码暴露需补充**：
设计文本只规定了核心逻辑，没说边缘场景在处理流程中怎么表现。代码实现时必须决定。**这个决策应该回流到设计文本**——后续同类场景不用重新拍。

**场景 4 · 数值平衡的实测反馈 → 设计文本调整**：
设计文本的数值在实际不同条件下表现差异巨大。**反推到设计文本调整策略**——不是因为代码不行，是因为原始策略在跨场景时不稳定。

## 新增模式：spec 先行开发的正向流动

除了"设计文档 → 代码"单向和"代码 → 设计文档"反向外，还有第三种：**spec 先行**。

spec 先行模式的工作流：
1. 先写 spec（契约定义——接口、行为、约束）
2. spec 作为"施工图纸"指导代码实现
3. 代码实现过程中发现 spec 的遗漏/矛盾 → 回修 spec
4. spec 始终保持为代码的权威来源

spec 和设计文档的区别：
- **设计文档**：解释"为什么这样设计"，给人看
- **spec**：定义"对外的契约是什么"，给 AI 当施工图纸
- 两者可以共存——设计文档讲意图，spec 定契约

spec 先行对 AI 协作的特殊价值：AI 读 spec 比读"解释型设计文档"更容易写出正确代码——因为 spec 是精确的接口定义，不需要 AI 做"从解释推断实现"的二次解读。

## 反向场景：代码悄悄改写了设计

以上场景是"代码暴露了设计问题 → 回修设计"（**正向修正**）。
但实战中出现了另一个方向：**代码实现时无意中扩张了 scope，改写了设计意图——事后才发现**。

这两个方向有本质区别：

| 方向 | 主动方 | 结果 | 处置 |
|------|--------|------|------|
| 正向修正 | 设计层主动修订（由代码暴露触发） | 设计更准确 | 好事，应鼓励 |
| 反向悄改 | 实现时无意扩张，设计被动偏移 | 代码里有"没人拍板的设计" | 必须止血——回滚 or 补拍板 |

### 反向悄改的识别信号

AI 主动加了设计文本里**没有明确要求**的功能 / 条件 / 约束，应当当场标"存疑"并 bubble up：

- "我实现了 X，但设计文本没写 X，只是看起来合理——需要你确认是否在 scope 内"
- 而不是默默实现，等验收时发现

**核心原则**：设计文本的每个功能都应该有明确的拍板来源。代码实现时"看起来合理就加"= 绕过了拍板流程。

---

## 双向流动的触发条件（强信念）

代码实施时遇到下列任一情况 → **必须 bubble up 给设计层**：

1. 设计文本字面有歧义，必须挑一种实现
2. 设计文本字面与意图冲突（实测后才发现）
3. 设计文本遗漏了 UI / HUD / 流程层细节
4. 数值实测发现设计假设不成立
5. 同类问题在多个模块都会撞（说明是设计层缺原则）

**不该 bubble up**（典型）：
- 纯实现细节（变量名 / 文件结构 / 局部 helper）
- 用户已明确"按现状先做，先跑起来"的临时性实现
- 不影响后续其他实现的 corner case

---

## 给"想搭框架的人"的建议

- **设计文本不是 freeze 后不动的**：建立 review 机制，让代码层 push back 容易（一个 issue / 一句对话即可）
- **同步成本要低**：如果改设计文本需要走复杂的 review 流程，没人会回流。让它和代码 commit 一样轻
- **设计原则段比具体规则更值钱**：沉淀的是"原则"（如"所有伤害系数统一用乘法"），比"某技能改为 ×1.5"价值大得多。原则段面向未来所有同类，具体数值只面向当前一个
- **回流要留指针**：设计文本被代码反推修改时，注明"某次实施时反推修订"——让未来读者知道这条原则是有实战来源的，不是凭空想
- **spec 先行可减少双向摩擦**：如果 spec 在代码之前写好，很多"代码暴露的设计问题"会在写 spec 阶段就暴露

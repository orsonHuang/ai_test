# 工作单元收尾的思想

> **状态**：stable（迭代中）
> **本文是什么**：聊聊"为什么完工时需要一个结构化收尾"，以及它怎么挂进三层 AGENT 体系。
> **不是什么**：操作手册（具体步骤在各项目自己的 checklist 中，因为扫描清单是项目特定的）。
> **维护原则**：本文是理念，不是规范。讲为什么这么想，不讲必须怎么做。

---

## 触发这篇的核心问题

实战中发现：AI 完成一个工作单元后，会停在"代码跑通 + 验收通过"的位置，认为事情结束了。

但实际上没结束，因为：
- 文档还停在 craft 之前的状态
- 本轮新发现的教训还散在对话里没沉淀
- 衍生工作可能立了但没在原工作里点名引用
- 自检（"我哪里判断偏了 / 沉淀了什么 / 还有什么该补的"）从来没做

这些事一件一件单独看，每件都不大；合在一起，是"工作单元的边界没收口"。

而最关键的是：**craft 时漏改是必然的**——不是说 AI 该多用心一点就能避免。原因有两个：
1. craft 时大脑被实现占满，外围同步必然漏
2. 同步点是一个**清单**而不是单一动作，靠记性管理清单是赌博

但**回头扫一遍**这件事有完全可枚举的清单（项目结构相对固定）。所以解决方案不是"craft 时更细心"，而是"craft 完成后再走一遍结构化清单"。

---

## 核心原则

> "工作单元结束时必须做结构化收尾，不能停在'代码跑通'就交接。"

这句话与具体技术栈无关——任何项目都成立。

---

## 三层 AGENT 体系怎么承接

### 全局层只立"思想"

在全局 AGENT 中放一句话原则 + 链接到详细文章。与具体技术栈无关，不规定"具体扫哪些文件 / 怎么写 commit message"——因为不同项目的扫描点不同。

### 项目层立"骨架"

项目层 AGENT 只立"该做"和"指针到细则"，不重复全局的为什么、也不重复 checklist 的细节。

为什么项目层留个钩子而不直接在 checklist 里？因为 AI 进入项目时第一份读的是项目层 AGENT，钩子在那里它才会想起去找 checklist。

### Checklist 立"细节"

具体的扫描清单、commit message 模板、触发条件门槛——全在 checklist 里。这一层会随实战不断长——发现一个新漏点就加一行。

### 模块层不动

模块层的契约/接口可以天然成为"阶段 B 要扫的位之一"。收尾不创造新模块层动作，只是把"该往这里更新"作为扫描点纳入清单。

---

## 它和已有规则的关系

Wrap-Up **不创造新原则**，只是把已有的几条规则捆成一个"完工仪式"：

- 同步动作 = "重复扫描即沉淀清单"的具体应用（扫预先沉淀好的清单）
- 沉淀动作 = 同上（本轮新发现的漏点写回清单）
- 自检动作 = "对话结束前自检"的具体出口

**为什么要捆一下？** 因为单条原则容易被跳过——"自检"和"沉淀"在 craft 完成时往往被忽略，因为大脑已经放松了。捆成"完工仪式"反而不容易被跳过——因为它是**有明确边界**的动作（开始 = 用户说收尾 / 结束 = 自检对话发出去）。

---

## §0 轮次衔接检查

在收尾协议之上，还有一个更基础的机制：**§0 轮次衔接检查**。

定义：对话启动时先读 `heartbeat-state.md`，若上轮有未 commit 的代码/文档改动，先补收尾（observations + NOW + memory + 自检）再回应新消息。此外收到用户每条消息时同样检查。不等待用户说"提交"。

**为什么需要 §0**：
- 用户可能在中途切话题——上一轮修改还没 commit，下一轮就开始讨论新问题
- 没有 §0 时，未提交的改动会越积越多，最终在 commit 时混成一团
- §0 让每次对话边界清晰：AI 主动检查"上轮有没有尾巴没收"

**§0 与收尾三件套的关系**：
- §0 是"每轮开始时的检查点"——防止遗留未提交改动
- 收尾三件套是"每个工作单元结束时的完整仪式"——包含同步 + 沉淀 + 自检
- §0 轻量（CHECK 型），收尾完整（DO 型），互补

### heartbeat 互补机制

§0 是在**收到用户消息时**检查上轮尾巴——这依赖 AI 在新一轮对话中"记得"做检查。实战中发现：如果用户的新消息切了话题，AI 可能跳过检查。

heartbeat 机制（详见 `cross-day-memory.md` 启动锚点章节）把检查时机**提前到对话启动时**——通过 `heartbeat-state.md` 的文件时间戳做机械判断，不依赖 AI 的记忆。加之 AGENT §22「每次输出结束时自动更新 heartbeat-state.md」，把 heartbeat 从"被动检查"升级为"主动高频刷新"——AI 不需要记住"该更新了"，每轮输出末尾强制写入。三者互补：

| | §0 轮次衔接 | heartbeat 启动检查 | §22 输出自动 heartbeat |
|---|---|---|---|---|
| 触发时机 | 收到用户每条消息时 | 对话启动第一步 | 每次 AI 输出结束时 |
| 检查方式 | AI 回顾上轮是否 commit | 对比 heartbeat-state 文件时间戳 | 机械写入——不依赖 AI 记忆 |
| 强项 | 覆盖"中途切话题"场景 | 覆盖"AI 忘记检查"场景 | 覆盖"收尾被跳过"场景 |
| 弱项 | 依赖 AI 记得检查 | 无法判断"未 commit 是故意的还是忘了" | 高频写入，但 heartbeat 文件极小（~15行），代价可忽略 |

三者共同构成收尾的三层安全网——§0 管途中，heartbeat 管启停，§22 管每轮。

## 收尾形态分类

实战中收尾不是单一形态，至少分 3 种：

| 形态 | 触发场景 | 特征 |
|------|----------|------|
| **首次发现型** | 某类工作第一次做完整收尾 | 会发现多个漏点，建立种子清单 |
| **精简授权型** | 工作单元小 / 流程已跑熟 | 用户一句话授权，AI 自动跑完收尾 |
| **回扫系列型** | 一个系列多张卡全部落地，事后发现早期工作有遗漏 | 扫的是**之前已完成的工作**，而不是当前工作 |

**关键认知**：
- 收尾不只发生在"当前工作完工时"——系列全部落地后，回头扫早期工作的同步性，是第三种形态的独立触发时机
- "验收通过 ≠ 收尾完成"——精简授权型时用户一句"验收通过"可触发 AI 完整跑收尾，说明这件事已被工程化
- 遇到新失败模式时，沉淀可扩展收尾动作（"第四件"）

## 收尾的替代形态：CHANGELOG 驱动的收尾

对于不使用 backlog 卡片体系的项目，收尾可以更轻量：

**每次操作后做的事**：
1. 写 CHANGELOG 条目（记录增删改）
2. 检查受影响的相关文档是否需要同步更新
3. 更新 NOW.md（当前进度）
4. 追加 observations（如有值得记录的观察）

这种形态比"四阶段收尾"轻得多，但对于非 backlog 驱动的项目足够——CHANGELOG 本身就是操作审计链，不需要再扫一遍卡片状态。

---

## 给"想搭框架的人"的建议

- **别一开始就写流程**——先看自己实战中是不是真的会漏文档同步。如果不会，不需要这条
- **第一次实战时让用户提你**——被用户提一次"扫一下遗漏"，你就知道哪些位会漏。把那次的位列出来当种子清单
- **样本不够多时不要急着固化**——清单会变，触发条件会调
- **三层挂钩要轻**——全局层一句话思想，项目层一个钩子，细节进 checklist。不要在三层都写一遍同样的内容
- **诚实暴露失败比规范工整更重要**——每次实战的"漏改清单"是这个流程最有教学价值的部分
- **非 backlog 项目用 CHANGELOG 替代**——卡片状态扫描不适用时，CHANGELOG + NOW 同步 + observations 三件套足够

---

## 它会不会变成过度设计

可能会。任何"流程"都有过度工程化的风险。如果每个 1 行的注释改动都触发完整收尾，那是灾难。

所以触发条件必须严卡。**默认不做，满足明确条件才做**。

典型判定：改动面 ≥ 几个文件 / 跨模块 / 含 bugfix / schema 扩展 / 用户明说收尾 / **用户消息含「收尾/收工/完成/提交」+ 本轮有文件改动**。其他场景不触发。

**注意**：最后一条是工程化加强——"收尾"在日常中文里容易被 AI 当作普通动词（如"完成后收尾"），而不是收尾协议触发词。把触发条件显式写进 AGENT §14 降低误判率。

这个门槛会随实战调整。**门槛的选择本身也是经验积累**，不是一次定死。

---

## 这个想法可能不成立的方向

- **如果你的项目文档量很少**（只有一两份 README），那根本没什么"收尾时同步"的事
- **如果你的协作风格是"频繁小提交"**，那两段式提交不适合你——应该把同步动作和代码动作混在每个小提交里
- **如果你用 CHANGELOG 驱动**，收尾不需要走完整四阶段——CHANGELOG 本身包含操作审计

这些是"种子在这块地不一定长得好"的真实情况。如果看完觉得不适用，**那就别用**。本文是种子不是模板。

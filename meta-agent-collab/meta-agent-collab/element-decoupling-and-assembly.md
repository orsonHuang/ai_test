# 零件解耦与按需组装 · 框架不是全装捆绑包

> **状态**：draft（初稿）
> **本文是什么**：把框架从"完整套装"重新定位为"零件市场"——谁需要哪些、何时该装、怎么发现"该装了"。
> **不是什么**：`gradual-adoption.md` 的替代品——那篇讲"按时间生长"（什么时候该有第 2 份文件），这篇讲"按场景组装"（不同的人该装哪些零件）。
> **为什么写**：分享出去后发现一个致命假设——默认读者需要全套。实际上不同场景、不同项目阶段、不同偏好的人，需要的是定制化安装。

---

## 核心问题

看完这套框架的人会问：_"这些东西我都要装吗？不全装会不会缺什么？"_

答案是：**不。按场景选零件，剩下的留给未来生长。**

工程直觉说"先搭骨架再填肉"——你自己搭过就知道，第一天建 15 份文件是反模式（`gradual-adoption.md` 已经说了）。但光说"从 1 份开始"不够——还需要回答：**不同的人最终会停在不同的"零件数量"**。

---

## 适用场景声明

坦率地说这套框架**不是给所有人用的**。它服务的是三个条件叠加的场景：

| 条件 | 不存在时的替代 |
|------|---------------|
| **多项目** | 单项目 → 不需要三层 AGENT，一层就够了 |
| **长期协作** | 单次任务 → 不需要 heartbeat / memory / observations |
| **结构化开发** | 探索/零散 → 不需要 spec 体系、CHANGELOG、checklist |

三个条件同时成立时，框架的全套元素才有土壤。**缺少任一条件，框架就会"过度设计"。**

不适用：
- 还没项目的零散任务
- 单次会话就完成的工作
- 偶尔让 AI 帮忙、没有建立长期协作意愿

框架的设计偏见是明确的：**为"认真做项目且愿意付文档维护成本"的人服务**。这不是缺陷——框架应该诚实地声明适用范围。

## 零件解耦：两个维度

### 维度一：AGENT 侧 vs 项目侧

框架的元素分属两套独立的归属系统：

| 归属 | 元素 | 依赖 |
|------|------|------|
| **AGENT 侧** | `AI_INDEX.md`、`AGENT.md`(全局)、`heartbeat-state.md`、`memory.md`、`PRODUCER_PROFILE.md`、`NOW.md`、`AI_SELF_CHECK.md`、`AI_SOFT_HINTS.md`、`reviews/` | 不依赖任何项目 |
| **项目侧** | `<项目>/AGENT.md`、`README.md`、`NOW.md`、`CHANGELOG.md`、`docs/`、`MAJOR_BUGS.md`、`_checklists/`(按需) | 不依赖 AGENT 侧（理论上可独立运作） |

这两侧可以**完全独立安装**——没有项目也能装 AGENT 侧，没有画像也能只维护项目文档。

### 维度二：AGENT 侧内部的功能分组

AGENT 侧内部还可按功能再拆：

| 功能组 | 包含元素 | 核心问题 |
|--------|---------|---------|
| **路由层** | `AI_INDEX.md` | "AI 该读什么" |
| **行为规则层** | `AGENT.md` + `AI_SELF_CHECK.md` + `AI_SOFT_HINTS.md` | "AI 该怎么做" |
| **连续性层** | `heartbeat-state.md` + `NOW.md` + `reviews/` + `memory.md` | "上次到哪了 / 发生了什么" |
| **画像层** | `PRODUCER_PROFILE.md` + `memory.md` | "我在和谁工作" |

依赖关系自下而上：路由层是最基础的——没有它 AI 不知道先读什么。连续性层需要行为规则层的支撑（heartbeat 检查是 AGENT §0 定义的）。画像层可以独立于连续性层存在，但画像的素材（observations）来自连续性层的 reviews 体系。

## 按场景组装的 5 种组合

解耦后的框架不再是"一套全装"，而是**零件市场**——根据场景选零件。

### 组合 A · 零项目 · 纯基础版

**装什么**：`AGENT.md`（含路由规则，或 `AI_INDEX.md` + `AGENT.md`）

**适用**：还没项目、只是想让 AI 按某种风格协作（"回复中文、不要啰嗦、先读再改"）。

**缺失**：没有心跳（每次都是"新 AI"）、没有画像（AI 不知道你的偏好历史）、没有项目文档体系（没项目所以也不需要）。

### 组合 B · 基础版 + 连续性

**装什么**：组合 A + `heartbeat-state.md` + `NOW.md` + `memory.md`

**适用**：有项目但结构松散、不想建完整 spec 体系、但需要 AI 跨会话记住状态。

### 组合 C · 纯项目侧

**装什么**：`<项目>/README.md` + `CHANGELOG.md` + `docs/`（+ 可选的 `_checklists/`）

**适用**：不需要 AI 画像、只需要 AI 理解项目架构并维护文档。适合已有自己协作风格的开发者。

### 组合 D · AGENT 完整版 + 项目基础版

**装什么**：组合 A + 画像层 + 连续性层 + 项目 README + CHANGELOG

**适用**：单项目、希望 AI 深入了解自己的偏好、但还没到需要完整 spec 体系的程度。

### 组合 E · 完整版

**装什么**：全部。

**适用**：多项目、长期、结构化开发——即当前框架作者的使用场景。

## 元素必要性分级

| 级别 | 元素 | 判据 |
|------|------|------|
| **必要** | 路由入口 + `AGENT.md`(全局) | 缺了框架无法运作——AI 不知道先读什么、不知道基本规则 |
| **高推荐** | `heartbeat-state.md`、`NOW.md`、`memory.md`、项目 `README.md` | 跨会话协作的基础——大多数人很快会需要 |
| **可选** | 其余全部 | 按场景信号触发，没有信号就不装 |

**注意**：路由入口和 AGENT.md 可以合并为一个文件（AGENT.md 顶部放路由规则）。`heartbeat-state.md` 也可以内嵌为 AGENT.md 的第一段。"必要"不等于"必须独立成文件"——只要求"信息存在且 AI 能读到"。

## 可选元素的"建议搭建"信号

可选元素的最大风险：**当前不需要，所以没建；未来需要时，已经错过了搭建时机**——项目跑了几周，回头补 CHANGELOG 的代价 >> 第一天就建。

解决：每个可选元素配一个**自举信号机制**，让用户不用"事先记住什么时候该装什么"。

信号分三层：
- **硬信号**：AI 在当前会话中直接观察到 → 主动提醒
- **软信号**：`memory.md` 积累到阈值 → 对话结束时提醒
- **推后信号**：用户自己发现需要 → AI 帮加速搭建

### PRODUCER_PROFILE.md

| 信号 | 阈值 |
|------|------|
| 纠正过 AI ≥3 次同类型判断偏好（"不对，我更喜欢 X"） | 硬信号 |
| 说过"以后都按 XXX 来" | 硬信号 |
| `memory.md` 同一 pattern ≥3 次 | 软信号 → 待晋升 |

**延迟后果**：AI 每次遇到类似场景重新猜测，每轮 +1 round-trip 纠正成本。

**搭建指南**：从 memory.md 待晋升条目 + reviews 素材提取初稿。结构：TL;DR（≤10 条）+ 正文（按版本追加，不删旧版）。初期只写 3~5 条最确定的，其余标"弱证据-待验证"。**它是长的，不是建的**——不需要第一天写完。

### AI_SELF_CHECK.md

| 信号 | 阈值 |
|------|------|
| `AGENT.md` 规则 ≥10 条 | 硬信号 |
| 同一条规则被口头提醒 ≥2 次 | 硬信号 |
| "违反 HOW #X" 被提过 ≥1 次 | 信号已出现 |

**延迟后果**：规则密度高时 AI 倾向于"读了但没执行"——需要对照的条目太多，没有速查表辅助。

**搭建指南**：以 AGENT.md 每条 HOW 为行，反向写"违反信号 + 自救动作"。不复述 HOW 原文，只给"怎么发现自己已经违反"。优先覆盖高频违规条目（#0、#3、#14、#17）。

### AI_SOFT_HINTS.md

| 信号 | 阈值 |
|------|------|
| token 消耗开始成为关注点 | 软信号 |
| AI 频繁写"好的，让我来..."等开场白 | 硬信号 |
| 回复长度显著超过提示长度 | 硬信号 |

**延迟后果**：每次对话浪费 5~15% token，但不影响正确性。纯成本优化。

**搭建指南**：分两类——工具使用习惯 + 对话输出习惯。不写成硬规则（违反不触发自救），只写"这样更省"。从最明显的冗余开始，慢慢加。

### 项目 CHANGELOG.md

| 信号 | 阈值 |
|------|------|
| 项目进入"可发布/可展示"阶段 | 软信号 |
| 第一次被问到"上次改了啥"而答不上来 | 硬信号 |
| `git log` 不足以解释"为什么改" | 硬信号 |

**延迟后果**：知道改了什么文件，不知道为什么改。对 AI 尤其致命——缺失 WHY 意味着重读全部相关文件来推断变更意图。

**搭建指南**：最小版——每次改动一行 `(日期) [类型] 一句话 + 原因`。按里程碑分段，不按日期堆砌。操作（增/删/改/迁）必记，代码实现可选记。

### docs/ spec 体系

| 信号 | 阈值 |
|------|------|
| 项目出现 ≥3 个模块/子系统 | 硬信号 |
| 跨模块改动时不知道该看哪个文件 | 硬信号 |
| 设计讨论的结论散落在对话里，找不到 | 硬信号 |

**延迟后果**：设计意图变成"口述历史"——只存在于某次对话上下文，换 AI / 换会话后丢失。**这是最贵的延迟代价。**

**搭建指南**：不是所有模块都需要 spec——有"跨模块契约"或"复杂度高到需要定义边界"的模块才建。小模块用单文件，大模块用 `spec/` 目录。配合 CHANGELOG：spec 管"现在是什么"，CHANGELOG 管"为什么变成这样"。

### reviews/ 体系

| 信号 | 阈值 |
|------|------|
| 做 ≥2 次周度复盘 | 硬信号 |
| 发现好的对话观察在会话结束后"蒸发"了 | 硬信号 |
| `memory.md` 开始积累待晋升条目 | 软信号 |

**延迟后果**：有价值的模式观察在对话结束后丢失——下次要重新积累。

**搭建指南**：最小版——`reviews/_observations.md` 一个文件就够了。格式：标签 + 事实 + 为什么值得。不写"做了什么"（git log 已有），只写"对人 review 有价值的观察"。

### _checklists/（项目层延伸）

| 信号 | 阈值 |
|------|------|
| 同一操作 ≥2 次、步骤 ≥3 步 | 硬信号（AGENT §11 直接触发） |
| AI 做检查漏了步骤 | 硬信号 |
| 你口头描述过"每次做完 X 都要检查 Y" | 硬信号 |

**延迟后果**：AI 每次重新搜索相关文件推断流程，漏步骤概率随复杂度上升。

**搭建指南**：不是固定目录——是项目层的延伸概念。从操作记录反推：观察最近 3 次做同类事的步骤 → 合并为标准流程。写成逐条检查项，不是段落叙述——checklist 是用来扫的，不是读的。

### MAJOR_BUGS.md

| 信号 | 阈值 |
|------|------|
| 出现"排查耗时 >10 分钟"的 bug | 硬信号 |
| 同类 bug 出现 ≥2 次 | 强信号 |

**延迟后果**：同类问题下次从头排查，浪费的时间不转化为经验。

**搭建指南**：最小版——现象 + 根因 + 修复方式 + 为什么上次没发现。不记小节（语法错误、类型错误），只记"需要推理"的 bug。

### Skills

| 信号 | 阈值 |
|------|------|
| 同一任务类型在 ≥3 个不同项目中出现 | 硬信号（从 memory 3 次晋升机制继承） |
| 一个 checklist 在多个项目中重复维护 | 软信号 |

**延迟后果**：每个项目独立维护几乎相同的内容，维护成本膨胀。

**搭建指南**：从现有跨项目 checklist 提取共性。结构：`SKILL.md`（最小入口）+ `references/`（按需）。在 `AI_INDEX.md` 追加 Skill 路由条目。详见 `reusable-agent-skills.md`。

## 给"想搭框架的人"的建议

- **先问自己三个条件**：多项目？长期协作？结构化开发？缺一就降级到对应组合
- **从组合 A 开始**——忘了 PROFILE、reviews、CHANGELOG。它们在你的场景需要时自然出现
- **"不装"不等于"永远不装"**——每个可选零件都配有触发信号。信号出现时，AI 会主动提醒
- **不要因为"别人装了什么"而装**——场景不同，需要的零件不同。同一种子，不同土壤长成不同形态
- **信号比计划准**——与其预设"第 N 周该装 X"，不如让信号自然触发。需要的时候你自然会感觉到"缺了点什么"

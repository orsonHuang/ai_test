# 框架自我硬化的故事弧 · 演化记

> **状态**：stable（迭代中）
> **本文是什么**：一份"AI 协作框架在实战中自我硬化"的诚实记录——记录"什么事件 → 触发什么改动 → 留下什么待验证"。
> **不是什么**：完整方法论 / 可照抄模板 / 已经过时间检验的最佳实践。
> **为什么写**：如果不写，"框架为什么会变成这样"的演化路径就丢了——只剩 git log 的事件碎片。
> **诚实交底**：很多硬化措施还没经过长时间实战压力测试，未来可能被推翻。读时注意标注的"待验证"项。

---

## 故事弧概览

框架不是在真空中设计的，而是在实战中**边做边硬化的**。每次硬化都遵循同一种模式：实战触发 → 暴露问题 → 沉淀规则 → 立刻压力测试。

两个主要的硬化窗口：
- **窗口 1**（约 24 小时）：框架从零搭建 → 跨日记忆 → 跨 AI 交接
- **窗口 2**（数周后）：框架从单项目迁移到多项目，在第二个项目实战中进一步硬化

---

## 窗口 1 · 24 小时初建（已记录）

核心事件：
1. 首次实战卡全实施 → 暴露"代码跑通 ≠ 完成"问题 → 催生收尾三件套（同步 + 沉淀 + 自检）
2. 首次正式复盘 → 暴露"AI 跨对话无记忆" → 催生跨日记忆层（_observations + NOW + reviews）
3. 制作人告知"明天可能换 AI" → 触发跨 AI 硬化（cbade 5 件套：信念度文档 + PROFILE 操作手册化 + 速查表 + AI 上岗指南 + 失败模式库）

**留下的产物**：
- 全局 AGENT §1~§20 规则体系
- PRODUCER_PROFILE 机制
- reviews/_observations 流式记忆
- meta-agent-collab 多篇理念文章

## 窗口 2 · 第二个项目实战中的框架硬化

在第二个项目（不同的技术栈、不同的项目结构）的实战中，框架经历了以下硬化事件：

### 1. AGENT 命名统一

原框架使用 `CODEBUDDY.md` 作为 AI 规则文件的命名。在第二个项目搭建时，名称迁移为 `AGENT.md`。

配套新增了：
- `AI_SELF_CHECK.md`：文档健康度自检扫描清单
- `AI_SOFT_HINTS.md`：AI 行为软提示（降低犯错概率的提醒）

这两个文件是全局 AGENT 的工程化延伸——当全局 AGENT 接近 100 行预算时，可拆分详细自检逻辑到配套文件，AGENT 本体用指针引用。

### 2. §0 轮次衔接检查

§0 是第二个项目实战中催生的关键规则：**AI 收到用户每条消息时，若上轮有未 commit 的改动，先补收尾再回应新消息。**

原始框架有 §14（收尾三件套）和 §18（对话结束 sync NOW），但缺少"跨轮次衔接"的防御机制——用户可能在中途切话题，上一轮修改还没 commit 就问新问题。§0 补上了这个缺口。

### 3. §15~§20 规则扩展

第二个项目实战中，§15~§20 被追加到全局 AGENT：
- **§15 走项目维护流程**：强制 AI 先读 maintenance 流程文档再动手
- **§16 可直接删改，操作记 CHANGELOG**：替代"只画删除线"的膨胀式维护
- **§17 每次对话结束追加 observations**：流式记忆的强制保障
- **§18 对话结束前同步 NOW**：当前状态的即时落地
- **§19 多源状态必先 reconcile**：禁止凭单源下结论
- **§20 澄清优于猜测**：需求模糊时一次性列出全部疑问

### 4. COLLABORATION.md 的发现与合并

第二个项目最初有独立的 `COLLABORATION.md`（约 50 行），装"清晰简洁 / 不奉承 / 直接指出错误"三条协作姿态规则。

实战中发现：
- 这三条与第二个项目的具体技术栈/业务**完全无关**——是跨项目通用的协作姿态
- 全局 AGENT 的 §2（清晰简洁）和 §6（直接指出错误）已经部分覆盖
- 单独维护一份 COLLABORATION 文件只增加了维护成本（两个文件要各自更新、各自保持行数预算）

**结论**：COLLABORATION 应合并到工作区根 `AGENT.md`——协作姿态是全局偏好，不应按项目拆分。将"客观中立，不奉承"补入全局 AGENT 即可。

### 5. 文档生态的迁移

第二个项目使用与原始框架不同的文档架构：
- **spec/ 目录**（28 个规格文档）替代了分散的设计文档——spec 是代码的权威来源
- **CHANGELOG.md** 作为操作记录——每次增删改操作即时记录，替代"只画删除线"的膨胀式维护
- **模块层用 README + spec 组合**替代单一 AGENT 命名——README 做路由器，spec 做详细契约
- **docs/maintenance/** 显式维护流程文档替代隐式的 checklist 习惯

这些不同于"标准模板"的实践验证了框架的核心思想——**种子不是模板**：同一套原则（分层 / SSOT / 指针 > 复述 / 操作可审计）可以长出不同形态。

## 全弧的 4 个统一规律

回看两次硬化窗口，都遵循同样结构：

1. **实战触发**：不是规划出来的硬化，是"做事过程中暴露问题"才硬化
2. **制作人提问驱动**：每个阶段的硬化都是被制作人某句具体问题 / 反馈触发
3. **立刻压力测试**：硬化措施落地后，立刻检查"这有没有 X 漏洞"
4. **沉淀+留痕同时做**：每次硬化都同步更新相关文档——不允许"先做后补"

## 这一弧没解决 / 留给下次的事

- §0 的遵循率：AI 是否真的每次轮次开头都检查未 commit 改动？
- 跨 AI 协作的实战检验：HANDOVER_NOTE + AI 上岗指南这一套，是否真的有效？
- spec/ 目录模式的适用边界：什么规模的项目适合 spec 先行？
- 模块层 README + spec 组合的更多验证：目前只有一个项目样本
- 框架在多项目工作区中的维护成本：当项目数增多时，全局 AGENT 是否需要更频繁地修订？

---

## 给未来 AI 的启示

如果你接手后又遇到"框架硬化时机"，不要试图独自设计——**等制作人提问 / 反馈，然后回应他的具体问题**。这是 PROFILE 中"AI 推荐 + 制作人拍板"在框架层的体现。

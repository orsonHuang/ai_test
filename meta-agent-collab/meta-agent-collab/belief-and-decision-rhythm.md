# 信念度 + plan/ask/craft 三模式 · 协作的决策节奏

> **状态**：stable
> **本文是什么**：聊聊"AI 怎么标信念度"和"用户怎么用 plan/ask/craft 三种模式响应"如何配合，构成可控的决策节奏。
> **不是什么**：CodeBuddy IDE 模式说明书 / 任何具体工具教程。
> **优先级提示**：如果你是新接手工作区的 AI，本文是"必读"——直接决定你能否跟上制作人的协作节奏。

---

## 核心想法（2 段）

**问题**：AI 给推荐时，用户常常不知道该用什么节奏回应——是该停下细看（这个推荐不靠谱），还是该秒同意（推荐靠谱不浪费时间）？如果每条都细看，消耗大；如果每条都同意，会被 AI 错误推荐拖偏。

**解法**：把决策节奏拆成两端配合：
- **AI 端 · 信念度标注**：每个推荐显式标"强信念 / 中信念 / 存疑"。强信念 = "我认为 A 显然对，列 B/C 仅为 traceability"；中信念 = "倾向 A，但 B 也合理"；存疑 = "真的不知道，列出是为了一起讨论"
- **用户端 · 模式选择**：plan = 集中决策审批（多个决策点一起拍）；ask = 探索某个具体疑问；craft = 直接执行（决策已定）

两端配合后，节奏就清晰了：
- AI 标"强信念" + 用户处于 craft 模式 = 默认通过，用户可以快速回"继续"
- AI 标"存疑" + 用户切到 ask 模式 = 慢下来一起想清楚
- AI 标"多个中信念决策" = 触发"多个中信念决策"时的惯用响应：建议切 plan 模式集中拍

## 用户侧的 Prompt 结构：把目标一次说清

信念度 + 模式切换管的是"AI 输出端的决策节奏"，但节奏不乱的前提是**输入端信息给到位**。如果用户 prompt 模糊，信念度再准确也是建立在错误地基上。

**一个清晰的 Prompt 应包含 5 要素**（灵感来自 WorkBuddy prompts 教程）：

| # | 要素 | 说明 |
|---|------|------|
| 1 | **要做什么** | 一两句话说清目标与期望结果——不是"帮我看看"，而是"检查 combat 模块的伤害计算是否和 spec 一致" |
| 2 | **相关资料** | @引用 / 指明文档、参考、示例输入输出 |
| 3 | **限制条件** | 风格、结构、命名约定、不要触碰的部分 |
| 4 | **完成的样子** | 内容、长度、格式、风格——"输出一个对比表格"比"分析一下"更可执行 |
| 5 | **参考示例** | 范例 / 样稿，比文字描述更有效 |

**上下文要点**（与 `context-budget-engineering.md` 呼应）：
- AI 不会自动看到所有文件——主动 @ 引用资料是关键
- ✅ 进入上下文：引用的文件、连接器返回、匹配的记忆与技能
- ❌ 被排除：无关历史、未引用资料、不相关的记忆

**反模式**：
- ❌ 只给一句模糊命令：「帮我看看」「改一下」— 触发 §20 澄清优于猜测
- ❌ 不说约束，结果难以收敛——信念度再高也是猜
- ❌ 塞太多无关内容，挤占 context window
- ❌ 换任务仍沿用旧会话，污染上下文

## 为什么需要这个机制

不用这个机制时的两种典型失败：

**失败 1 · 用户被推荐淹没**：AI 一次给 8 个建议，每个建议都说得头头是道，用户必须每条都重新评估"这是真见解还是 AI 凑数"。决策疲劳后用户要么全同意（被偏的方向带走）要么全否定（错过好建议）。

**失败 2 · AI 在不该犹豫的地方犹豫**：AI 在已经定的事上反复确认（"你确定要用 A 吗？我也可以做 B 或 C"），浪费回合。或反过来，在该停下细问的地方直接动手（"我已经按 A 实施了，不知道你想不想要 B"）。

信念度 + 三模式机制让两端**对齐期望**：AI 说"强信念" = 我会快做不墨迹；用户切 plan = 我现在要慢下来集中拍板。

## 三模式各自的角色

| 模式 | 谁主导 | 典型用法 | AI 该做什么 |
|---|---|---|---|
| **plan** | 用户 | "我想把这个版本的所有决策一次拍完" | 列出所有决策点 + 标信念度 + 等用户拍 |
| **ask** | 双向 | "这个 bug 根因可能是什么？" | 探索 / 假设 / 列证据。**不动手** |
| **craft** | AI | "按 plan 干" | 执行已拍的方案，遇到新决策点立刻停下问 |

**三者切换的判定**（实战经验）：
- 如果你脑子里"还没想清楚要做什么" → ask
- 如果你想"做之前先把所有决策一次拍完" → plan
- 如果决策已定 → craft

## 信念度的精细分级（4 档，不是 3 档）

实战中发现 3 档不够用——"存疑"经常装两种性质很不同的情况。**正式分 4 档**：

| 档位 | 含义 | AI 出口示例 | 用户期望回应 |
|---|---|---|---|
| **强信念** | 我没看到反例 + 推理链很短 + 即使错了代价小 | "强信念推荐 A，列 B/C 仅为 traceability" | 秒同意 / 不深究 |
| **中信念** | 两个方案都合理，我有偏好但承认主观 | "倾向 A，B 也合理。差异在 X" | 看一眼 / 决定要不要细聊 |
| **存疑（能猜）** | 信息不全但我能基于经验推一个，可能错 | "存疑。基于 X 我猜 A，但没看 Y 不确定" | 停下来 / 补信息 / 一起想 |
| **真不知道** | 我没有内在判断，列出来只是为了一起讨论 | "真不知道。列 A/B/C 是为了和你一起 brainstorm" | 主导讨论 / AI 不要选边 |

**为什么把"存疑"拆两档**：
- "存疑（能猜）"——AI 还能给方向感，用户可以"基于 AI 的猜测决定要不要查资料"
- "真不知道"——AI 给方向感反而误导（廉价归因），用户应该接管

**信念度撒谎的成本**：
- 标"强"实际"中"——用户秒同意后埋雷
- 标"中"实际"强"——用户每次都细看，AI 浪费用户决策预算
- 标"存疑（能猜）"实际"真不知道"——用户被 AI 的猜测带偏方向
- **校准义务在 AI**：定期自检"我标强信念的事被推翻了多少"，发现偏移主动调整

## AI 自检清单（输出推荐前 5 秒做完）

每次给推荐前快速过：

1. **本条是单一推荐还是多个并列**？多个 → 必须各自标信念度，不能统一标一个
2. **本条信念度是哪档**？强 / 中 / 存疑（能猜）/ 真不知道
3. **如果标强信念，我能写出"为什么 B 和 C 不对"吗**？写不出 = 应降为中
4. **如果标中信念，我有没有把 B 和 C 也讲清楚**？讲不清 = 应升为存疑
5. **本条决策错了用户的损失多大**？大 = 主动降一档（保守偏好）

## 用户回应的映射表（给新接手 AI 看）

制作人对不同信念度的典型回应模式（基于 ~20 次决策样本）：

| AI 标 | 制作人典型回应 | AI 该如何继续 |
|---|---|---|
| 强信念推荐 A | "采纳" / "ok" / 直接进下一步 | 进下一步，**不要追问"你确定吗"** |
| 强信念推荐 A | "做 B" | 服从 + 在 _observations 标 `[纠错]`——你的强信念被推翻了，要校准 |
| 中信念倾向 A | "采纳" / "拍 A" / "拍 B" / "拍 C" | 服从。中信念被推翻不需要标记 |
| 中信念倾向 A | "再说说为什么不选 B" | 切 ask 模式，深入展开。**不要立刻执行** |
| 存疑（能猜） | "我们再想想" / "暂时不选" / "你有没有其他思路" | 切 ask 模式，列更多角度 |
| 真不知道 | 主动给方向 / 提出新维度 | 接收用户的方向 → 重新评估信念度（用户给的方向可能让你升级到中信念） |

**反模式（别的 AI 容易犯）**：
- 用户回"采纳" → AI 接着追问"还有 X 要不要考虑"——破坏节奏
- 用户回"拍 B" → AI 解释"为什么 A 也不错"——浪费回合，且违反"永远给推荐不替拍板"原则的反向（决定后的 AI 不该回头辩护）
- 用户给方向后 → AI 假装"啊对我也是这么想的"——拍马屁，违反 PROFILE 不奉承原则

## 真实样本：一张卡全程的模式切换

一次实战中的模式流转（用户视角）：

1. **ask 阶段**："开这张卡是不是要 plan 模式？" → AI 探讨触发条件，建议开 plan
2. **plan 阶段**：AI 列出 4 个决策点（红锁机制 / 伤害系数实现 / 硬抗实现 / 冲锋落点逻辑）+ 各自信念度。用户拍板 D2 = B 方案，其他采纳推荐。同时新触发"伤害系数全乘法化"原则
3. **craft 阶段**：AI 按 plan 施工，7 个 todo 一路推进，遇到的小决策（如 max_per_turn=1 vs ∞）自己用强信念决策不打断
4. **ask 阶段**（实战 smoke）：用户报 6 个反馈，AI 逐项分析（其中 #3 重斩 -4hp 是误读，AI 解释清楚 vs 改代码 = 用 ask 模式先确认）
5. **craft 阶段**（hotfix）：5 项确认后批量修
6. **ask 阶段**（收尾后反思）：用户说"能不能做成通用流程"——纯讨论，AI 标信念度给方向

**关键观察**：模式不是从头到尾线性的，是 ask → plan → craft → ask → craft → ask 多次切换。每次切换都对应一个**节奏变化**——慢思考、定方案、快执行、再慢思考。

## 真实样本：plan 阶段信念度标注带来的「0 摩擦授权」

在一个系列的后半段出现了一个**可重复的模式**：

**模式描述**：AI 在 plan 阶段把所有决策点列出 + 各自标信念度 → 用户在 craft 开始时一句话全量授权 → craft 阶段零打断执行。

**具体授权话术样本**（4 次独立出现）：
- 「接受默认推荐，按这个计划执行」
- 「按你说的执行」
- 「按你推荐的执行方案」
- 一次性多项问题全部作答，直接进入 craft

**为什么这个模式出现了**：

用户对 AI 的信念度标注建立了信任——当 AI 清晰标出哪些是"强信念"（我认为 A 显然对），用户就不需要逐项 review。plan 阶段集中拍板（"这几个中信念我来选"）之后，剩下的强信念项可以在 craft 阶段全部自动执行。

**这个副作用是 plan/ask/craft 模式设计时没有明说的**：plan 阶段不只是"集中拍板"，它同时完成了"信任校准"——用户对 AI 的信念度标注准确性建立信任后，craft 阶段的决策摩擦接近于 0。

**反面观察（不出现 0 摩擦的情况）**：
- plan 阶段存在 ≥1 个"存疑（能猜）"或"真不知道"项 → craft 阶段这些点必须停下问，不能连贯
- AI 在 plan 阶段信念度标注不准（后来被推翻）→ 用户下次 plan 时不会全量授权，恢复逐项 review

**沉淀为规律**：  
> plan 阶段信念度标注的**累积准确率**决定了 craft 阶段的授权粒度。准确率高 → 授权粒度粗（全量 / 一句话）；准确率低 → 授权粒度细（逐项确认）。

这是信念度机制不在当下发挥作用，而是**跨轮次积累**后才能看到的效益。

## 给"想搭框架的人"的建议

- **三模式不是必须**：如果你的工具不分模式，可以让 AI 在每条建议前手动标"建议直接做" / "建议讨论再定" / "需要你决定怎么做"
- **信念度不是必须**：如果觉得"强中弱"太抽象，可以用"5 分 / 3 分 / 1 分"或"绿黄红"
- **重要的是两端约定**：用户和 AI 必须**事先约定好**信念度的语义，否则 AI 标"强"用户当"试探性建议"看，不对齐
- **校准义务在 AI**：AI 应该定期自检"我标强信念的事被推翻了多少"，发现校准偏移要主动调整
- **新 AI 接手时的训练**：把"信念度 + 模式"打印成 1 页纸放在 AI 必读列表里——别的 AI 默认不会这么细，需要明示

## 信念度与"先重算"原则的关系

**风险**：强信念容易掉进"我以为"陷阱——以为我懂，没去重算。

**自救机制**：
- 标强信念前先做"反先重算自检"——本条结论是凭印象还是凭证据？
- 凭印象 → 必须降到中信念，或先去重算再标
- 凭证据 → 强信念合法

例：你以为某文件长这样所以推荐 A → 凭印象 → 降中信念 → 读了文件再升回强信念。

# 文档健康度工程化 · 让文档自己说话

> **状态**：stable（迭代中）
> **本文是什么**：聊聊怎么让文档的"可信度"变成可被工程化检查的属性——不需要人肉记住哪些文档过时了。
> **不是什么**：具体脚本/工具教程。

---

## 核心想法

**反直觉的事**：团队里有 50+ 份文档时，"哪些文档还是准的、哪些已经过时了"这个人脑记忆是不可靠的。没有工程化手段兜底，文档会逐渐变"鬼城"——看起来很多，但没人敢信。

**我的解法**：两件事并行——
1. **标记层**：每份文档显式标注健康状态（stable / draft / outdated）
2. **自动化层**：用工具检查断链、过期标记、一致性

---

## 健康三态

每份文档标注当前健康状态：

| 状态 | 含义 | 行为 |
|------|------|------|
| **stable** | 当前准确，可直接信任 | AI 读到直接使用 |
| **draft** | 正在写/改，内容可能不完整 | AI 读到要加"注意：本文还在起草" |
| **outdated** | 已过时 | AI 读到要加"注意：本文可能过时，请对照 CHANGELOG" |

**谁来更新**：
- 文档的写作者在每次修改后更新状态（从 draft → stable）
- 发现者（任何人 / AI）发现文档过时后标为 outdated（然后通知维护者）

**关键是低成本**：标注就是文档顶部一行注释（`> 状态：stable`），不是什么重量级流程。成本越低，遵循率越高。

---

## 自动化 lint 与实现路径

工程化检查可以覆盖四个层面。按实现成本从低到高排列：

### 1. 断链检测（性价比最高）

检测文档中的内部链接是否指向存在的文件。输入是文档目录，输出是断链清单。

**检查范围**：
- Markdown 链接格式：`[text](path/to/file.md)`、`见 X.md`
- 相对路径 vs 绝对路径的统一
- 链接指向目录（`spec/core/`）时检查目录是否存在

**伪码骨架**：
```
for each .md file in docs/:
    extract all local links
    for each link:
        resolve path relative to current file
        if target file doesn't exist: report broken link
```

**输出格式建议**：
```
BROKEN LINKS (3):
  spec/core/combat.md → ../engine/damage.md (file not found)
  docs/guide.md → ./deprecated/api.md (file not found)
  ...
```

不需要解析 git 历史，不需要 AST，一个正则 + 文件存在性检查就够了。复杂度极低，收益极高——防止 AI 和人类点击转向不存在的文档。

### 2. 过期检测

标注 `outdated` 的文档，检查它已经多久没更新了。

**规则**：
- 文档标注 `outdated` + 最后修改日距今 > N 天 → 输出提醒
- N 建议设为 30（可调）

**检查方式**：
```
for each .md file with "> **状态**：outdated":
    last_modified = git log -1 --format=%ad -- <file>
    if days_since(last_modified) > N:
        report: "<file> outdated for {days} days, needs attention"
```

**为什么不是"自动归档"**：标记 outdated 和真的该归档是两件事。自动归档可能误删"虽然过时但仍有参考价值"的文档。过期检测止于提醒，归档仍由人决策。

### 3. 一致性检查

较高级——检查 spec 中引用的接口/函数签名是否和代码一致。需要项目有 spec 文化才值得做。

**前提条件**（没有就别做）：
- 存在 spec 文件，且 spec 中明确标注了接口签名
- 代码语言有可解析的结构（JS/TS 的 function 签名、类方法等）

**轻量替代**：不检查签名一致性，只检查"spec 引用的文件/模块是否存在"。这在没有完整 spec 文化时仍然有价值。

### 4. 状态汇总

生成文档健康度快照，一行命令看清全局。

**输出格式**：
```
DOC HEALTH REPORT (2026-06-04):
  stable:  8  ████████████████
  draft:   3  ██████
  outdated: 1  ██
  unmarked: 2  (needs triage)

Files needing attention:
  [outdated 45d] spec/old_api.md
  [unmarked]     docs/new_feature.md
  [unmarked]     notes/scratch.md
```

**集成方式**：
- 作为 lint 脚本挂在 `npm run docs:lint`（或项目根 `docs:lint.sh`）
- 在 CI 中跑（或至少在 commit 前手动跑一次）
- 输出报告告诉维护者"哪些文档需要关注"

**关键原则**：从步骤 1（断链检测）开始，而不是一步到位做全部 4 层。断链检测写完只跑一次就能立刻发现真实问题，后续各层等第一层稳定了再加。

## 为什么这很重要

对于 AI 协作特别关键：
- AI 读文档时无法判断"这份文档还准吗"——因为它没有"上次修改后代码又改了"的上下文
- 如果没有健康标记，AI 会把过时文档当权威来源——出错的概率大幅上升
- 有了健康标记 + lint，AI 至少知道"这份文档我该不该信"

对于多项目工作区同样重要：
- 当你有 N 个项目、每个项目有 M 份文档时，健康状态就变成"信号"
- 不需要人肉记住"A 项目的 combat spec 刚更新、B 项目的 combat doc 可能过时"——lint 告诉你

## 同步成本的经济学

文档和代码/现实的 divergence 是**必然的**——不是"做得不够好"，是物理规律。代码每改一次，文档就有了一个同步窗口。问题不是"要不要同步"，是"什么时候值得花时间同步、什么时候容忍过期"。

### 同步的两种成本

| 成本类型 | 表现 | 极端后果 |
|----------|------|----------|
| **同步成本** | 每次代码改动后更新相关文档的时间 + 认知负荷 | 维护文档比写代码更累 → 放弃维护 |
| **不同步风险** | 文档过时 → AI 或人读到过期信息 → 做错决策 | 文档变"鬼城"，没人敢信 |

**核心权衡**：同步成本和不同步风险之间的平衡点，决定了你应该多频繁同步。

### 按文档类型决定同步策略

不是所有文档都需要同等频率的同步。按"读的人是谁"和"变更频率"两个维度：

| 文档类型 | 读者 | 变更频率 | 同步策略 |
|----------|------|----------|----------|
| AGENT.md（AI 规则） | AI（每次对话必读） | 低 | **必同步**——AI 每次读，过时 = 每轮出错 |
| spec/*.md（契约） | AI（按需读） | 中 | **代码改完即同步**——spec 是权威来源 |
| NOW.md（当前状态） | AI + 人 | 高（每次对话） | **每次对话结束同步**——过期 = 下次启动迷路 |
| design/*.md（设计文档） | 人为主 | 低 | **宽松同步**——设计变了才同步，小改动可延后 |
| _observations.md（素材） | AI（周度 review） | 流式追加 | **只追加不修改**——不存在同步问题 |
| notes/*.md（草稿） | 作者自己 | 不定 | **容忍过期**——草稿的默认读者只有写的人 |

### 什么时候该同步的判据

三个简单问题，回答"是"超过 1 个就该同步：

1. **该文档的下一个读者会在近期（本周内）读到它吗？** 
2. **过时的内容会导致做错决策吗？**（vs 过时只是"信息不完整"）
3. **同步成本低吗？**（改一行 vs 重写一节）

### 什么时候容忍过期

- **草稿/笔记类文档**——它们的生命周期本来就不是"权威"，过期是自然状态
- **低频读取的文档**——读者在几个月后才来，那时候可能又过时了；与其反复同步，不如在文档开头诚实标注"最后验证日期"
- **"历史记录"型文档**（如 review 归档）——它们不是"当前状态"而是"当时的快照"，不需要同步

### 这个经济学与健康三态的关系

- **stable** = 同步成本已被支付，不同步风险低
- **draft** = 还没到该支付同步成本的时候（内容还在变）
- **outdated** = 不同步风险已经 > 同步成本，该付了

三态标记本质上就是同步经济学的"仪表盘"——一眼看出哪些文档的同步债务到期了。

## 落地的渐进路径

1. **第一步 · 手动标注**：给已有的关键文档加上 `stable` / `draft` / `outdated` 标签
2. **第二步 · 约定规则**：团队（人或 AI）约定"修改文档后必须检查状态"，出现 outdated 文档必须处理（更新 or 归档）
3. **第三步 · 断链 lint**：先把断链检测做起来——这是最低成本、最高收益的检查
4. **第四步 · 到期提醒**：对标注 outdated 超过 N 天的文档发提醒
5. **第五步 · 精确检查**：spec-signature 一致性检查（如有 spec 文化）

## 规则层的健康度

健康度不只是文档的事。**AGENT 中的规则本身也有生命周期**——一条规则可以从"精准有效"变成"过时噪音"。

### 规则退化的三种路径

| 退化类型 | 表现 | 成因 |
|----------|------|------|
| **过时** | 规则指向的实践已经变了，但规则没更新 | 项目演化比规则快 |
| **过于宽泛** | 原来写"注意 X"有用，但现在 X 已经被工具链兜底，规则变成空话 | 兜底手段升级 |
| **过度具体** | 规则写的是"当时那个版本的解决方案"，新版本不再适用 | 版本锁定 |

### 怎么检测规则是否退化

- **遵循率异常**：如果一条规则 AI 从来不触发（或总是触发但从不产生实际效果），可能是该清理的信号
- **"这条规则当初为什么加"失忆**：如果维护者自己都解释不清一条规则的动机，它大概率已经退化
- **与新规则冲突**：两条规则在具体场景下给出相反指令——其中一条该退休

### 规则的"三态"健康度

规则也可以沿用 stable / draft / retired：

| 规则状态 | 含义 | 处置 |
|----------|------|------|
| stable | 活跃有效，AI 遵循 | 保留在 AGENT |
| draft | 新加入，观察期 | 保留 + 标注 `[观察中]` |
| retired | 已退化/过时 | 移出 AGENT，进 CHANGELOG 的 removed 段 |

### 与文档健康度的区别

- 文档过期 → AI 读到过期信息 → 做错决策（**信息错误**）
- 规则退化 → AI 被无效规则占用注意力 + 可能产生错误行为约束（**指令污染**）

规则退化的危害比文档过期更隐蔽——不会报错，只是 AI 的行为悄悄变差。而 context budget 有限时（见 `context-budget-engineering.md`），每条退化规则都在**浪费 context 预算**。

## 给"想搭框架的人"的建议

- **从手动标注开始**——不要一开始就写自动化脚本。先养成"标注文档状态"的习惯
- **断链检测性价比最高**——一个简单的文件存在性检查就能防止大量"读了不存在的文档"问题
- **不要追求 100% 自动化**——有些健康判断（"这段内容还准不准"）目前只能靠人。自动化覆盖能覆盖的部分即可
- **健康状态是文档的一部分**
- **outdated 需要处置规则**：不能无限累积 outdated 文档。定义 N 天不处理的 outdated 文档自动归档 or 删除

## 自动衰减维度：时间驱动的健康度

手动 outdated 标记覆盖的是"内容过时了要标记"的场景。但有一种退化是"不再被使用但也没人记得标记"——手动标记对此无能为力。

### 衰减规则

| 文件类型 | 衰减规则 | 动作 |
|----------|----------|------|
| memory.md 条目 | 30 天未触发 | 降为冷条目（保留不删，但下次启动不强制读） |
| PROFILE 盲区 | 90 天未更新 | 标注"长期未验证" |
| observations | 不衰减 | 素材层只追加 |

### 衰减 vs 手动 outdated 的区别

| | 衰减 | 手动 outdated |
|---|---|---|
| 触发 | 时间驱动——30/90 天自动 | 内容驱动——人发现过时了才标 |
| 适用 | "还在用但频率极低"的条目 | "内容已经不对了"的文档 |
| 动作 | 降冷（不删、不归档、只是降优先级） | 提醒后由人决定更新或归档 |

**关键设计选择**：衰减不做删除或归档——只做降优先级。因为自动删除可能误伤"低频但关键"的条目。降冷意味着"AI 不再每次必读，但用户显式问到时还能查到"。

### 衰减的检测时机

衰减的检测不依赖独立的 cron——它挂在 heartbeat 检查里。每次对话启动时，heartbeat 顺带检查 memory.md 中条目是否超过 30 天未触发。灵感来自 self-improving skill 的 HOT/WARM/COLD 三层体系。

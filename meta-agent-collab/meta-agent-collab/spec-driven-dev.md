# Spec 先行开发模式 · spec 是契约，不是文档

> **状态**：stable（迭代中）
> **本文是什么**：聊聊一种与 AI 协作特别友好的开发模式——先写 spec（契约定义），再写代码，代码完成后回写 spec 保持一致性。
> **不是什么**：任何具体工具/框架教程。

---

## 核心想法

**传统方式的问题**：
- 先写设计文档（解释型）：讲"为什么这样设计"、"设计意图是什么"
- AI 读设计文档后写代码：要从"解释"推断"实现"，多一层解读
- 代码完成后设计文档往往滞后

**spec 先行方式**：
- spec 是契约层——定义接口、行为、约束、边界条件，不讨论"为什么"
- AI 读 spec 像读施工图纸——精确、可执行、不需要二次解读
- 代码完成后回写 spec：把实现中发现的遗漏/偏差同步回 spec

---

## spec 和设计文档的区别

| 维度 | 设计文档 | spec |
|------|----------|------|
| 定位 | 解释"为什么" | 定义"是什么" |
| 读者 | 人（理解设计意图） | AI / 开发者（精确执行） |
| 内容 | 设计理由、权衡、假设 | 接口签名、行为约束、边界条件 |
| 更新 | 低频（设计变更时才改） | 高频（代码每改就同步） |
| AI 友好度 | 中（需解读） | 高（直接可执行） |

两者可以共存：设计文档解释意图，spec 定义契约。不是替代关系，是分工。

---

## spec 先行的工作流

```
1. 写 spec（契约定义——接口、行为、约束）
       ↓
2. spec → 代码实现（spec 作为备工图纸）
       ↓
3. 实现中发现的遗漏/矛盾 → 回修 spec
       ↓
4. spec 始终保持为代码的权威来源
       ↓
5. CHANGELOG 记录 spec ↔ 代码同步操作
```

**关键步骤**：
- 步骤 3 是双向流动的具体实践——代码实现是 spec 的最严格 review
- 步骤 4 是 spec 文化的核心约束——如果 spec 和代码不一致，以 spec 为准（然后修代码）

---

## 为什么 spec 先行对 AI 协作特别友好

1. **减少解读误差**：AI 读"函数签名 + 行为约束"比读"设计意图解释"更不容易出错
2. **减少"看起来合理就加"**：spec 明确 scope 边界 → AI 更容易判断"这个不在 spec 范围内"
3. **plan 阶段更高效**：spec 是现成的决策依据——"spec 说 A，所以选 A"
4. **验收更清晰**：对照 spec 逐条验证，不需要凭印象判断"是不是做完了"

---

## 不适用 spec 先行的场景

- **极早期原型**：还在探索"要做什么"，spec 写早了反而限制思路
- **一次性脚本/工具**：维护 spec 的成本高于收益
- **频繁变更的实验性项目**：spec 的同步成本会很高

**何时引入 spec**：当你大致知道"这个模块的对外接口是什么"时。此时 spec 的成本低（写接口定义），收益高（后续实现可直接按 spec 施工）。

---

## spec 的组织方式

按模块/功能拆分成独立文件：
- `spec/core/game_engine.md` — 核心引擎契约
- `spec/core/combat.md` — 战斗系统契约
- `spec/ui/main_menu.md` — 主菜单契约
- ...

**好处**：
- 每个 spec 文件短而专，AI 按需读取不浪费 context
- 不同 spec 可以独立 review 和更新
- spec 文件名本身就是路由——"改战斗系统"→ 看 `spec/core/combat.md`

---

## 在非游戏项目中的适用性

spec 先行模式不是"游戏开发专用"。它适用于任何**接口稳定 + AI 频繁参与实现**的项目类型。但要判断适配策略：

| 项目类型 | spec 适配度 | 说明 |
|----------|------------|------|
| **API 服务** | ★★★★★ | 天然适合。OpenAPI / gRPC proto 本身就是 spec。接口签名 + 行为约束 + 错误码定义是完美的 spec 素材 |
| **CLI 工具** | ★★★★☆ | 适合。参数定义 + 输出格式 + 错误处理 = spec 三要素非常清晰 |
| **数据处理 pipeline** | ★★★★☆ | 适合。输入 schema → 转换规则 → 输出 schema，pipeline 的每一步都是 spec 的一个章节 |
| **前端组件库** | ★★★☆☆ | 部分适合。组件 props/events 适合 spec；视觉细节不适合（该在设计稿/Storybook 里） |
| **探索性研究/原型** | ★☆☆☆☆ | **不适合**。还在探索"要做什么"，spec 写早了限制发散 |
| **一次性脚本** | ☆☆☆☆☆ | **不对**。维护 spec 的成本 > 重写脚本的成本 |

**非游戏项目引入 spec 的关键差异**：
- 游戏 spec 多围绕"机制"（规则、公式、状态机）；API spec 围绕"合约"（请求/响应、错误、幂等）
- 但骨架一致：定义接口 → 约束行为 → 明确边界条件
- spec 先行的收益（减少解读误差、验收更清晰）在任何领域都成立

## 真实 spec 长什么样（示例）

以下是一个游戏模块 spec 的简化片段，展示"契约定义"的实际格式：

```markdown
# spec/core/combat.md

> 最后同步：2026-06-04（代码 commit abc1234）

## 接口

### calculateDamage(attacker, defender, skill)
- **返回**：{ damage: number, isCrit: boolean, effects: Effect[] }
- **前置条件**：attacker 和 defender 均为活跃状态（isAlive = true）
- **副作用**：无（纯计算，不修改状态）

### applyDamage(target, damageAmount)
- **前置条件**：target.isAlive = true
- **副作用**：target.hp -= damageAmount；若 hp ≤ 0，触发 onDefeat(target)
- **返回值**：{ remainingHp: number, defeated: boolean }

## 行为约束

- 伤害计算必须在 applyDamage 之前完成
- onDefeat 触发延迟到当前回合结束时批量执行（不得在 damage 循环中触发）
- 暴击判定独立于伤害计算（先算伤害再判定暴击，而非同时）

## 边界条件

- attacker 自身受到反伤 → 反伤不受护甲减免
- 伤害下限 = 1（哪怕防御 > 攻击力，至少 1 点伤害）
- 0 HP 角色仍然可以成为"被引用目标"（用于日志/特效），不得立即删除对象
```

**这个示例展示了什么**：
- **接口定义有明确签名 + 前置条件 + 副作用**——AI 读到就知道怎么实现，不需要额外解释
- **行为约束说"什么顺序"、"什么时机"**——这些是 spec 的核心价值，设计文档一般不写这么细
- **边界条件覆盖极端情况**——反伤、0 HP 对象等边缘 case，spec 不写的话代码必然有 bug
- **不讨论"为什么这样设计"**——那是设计文档的活，spec 只管"是什么"

**关键习惯**：
- spec 顶部标注"最后同步"——知道这份 spec 和代码的差距
- 接口用"返回 / 前置条件 / 副作用"三段式——AI 实现时每个接口都有完整约定

## spec 的行数预算

与 AGENT 的 100 行预算不同，spec **不限行数**。理由：
- spec 是"被引用"的，不是"每次必读"的——AI 只在需要时读取相关 spec
- spec 越长只说明模块越复杂，不会浪费无关 context
- spec 应该"信息密度高"而非"行数少"

**但**：spec 仍应该用清晰的结构（接口 / 行为 / 约束分段），而非流水账。

---

## 给"想搭框架的人"的建议

- **spec ≠ 设计文档**：不要往 spec 里写"为什么这样设计"——那是设计文档的活
- **spec 是活的**：代码写完要回写 spec，保持 spec = 代码权威来源
- **从核心模块开始**：不需要一开始就给所有模块写 spec。先给"接口最稳定 / 被引用最多"的模块写
- **CHANGELOG 同步记录**：spec 的每次变更记录在 CHANGELOG，保持审计链完整
- **wip → spec 迁移要规范**：临时文档拍板后迁出到 spec，不要混在一起

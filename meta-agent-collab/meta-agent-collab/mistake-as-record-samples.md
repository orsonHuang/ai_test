# 犯错即记录 · 原则与指引

> **状态**：stable（迭代中）
> **本文是什么**：聊聊"犯错即记录"作为一种**协作原则**——为什么、怎么做、放哪。  
> **不是什么**：错误故事集（具体故事是项目特定的，分散在各项目自己的 checklist / MAJOR_BUGS / backlog 里）。

---

## 核心想法（1 段）

**最有教学价值但最难写的内容**：诚实暴露错误，每条对应一条已沉淀的规则。规则不是凭空想出来的，是踩过坑沉淀的。但样本本身往往**是项目特定的**——某个项目的字段名/函数名对其他项目读者价值有限。所以本文只立**原则**，样本回到各项目自己的位置。

---

## 原则

1. **规则必须有来源**：每条沉淀规则应该能追溯到"我们曾经这么干，错了"。凭空想的规则脱离实际，往往会被绕过
2. **样本贴在规则旁**：不要单独建"错误日志"文件——会渐渐没人读。直接把样本简述写到规则所在位置（如 `_checklists/<X>.md` 对应规则段尾："样本来源：<卡号> <日期>，<一段>"）
3. **诚实不美化**：写"我以为 X 所以 Y，结果错了"，不写"经过深思熟虑选择了 Y"——后者是事后合理化，无教学价值
4. **不是所有错都要记**：能 ≥2 次同类错才值得沉淀。单次错是事故，重复错才是模式
5. **错误抽象成原则**：单次样本 → 找根因 → 抽象成"什么场景下要做什么"→ 沉淀。直接记"我把 1 写成了 2"没用
6. **触发信号清单**——以下用户行为应该触发一次 recording（而不是凭 AI 主观判断"值不值得记"）：
   - "No, that's not right..." / "不对，应该是..." → 纠错
   - "I told you before..." / "我上次说了..." → 标记为重复，提升优先级
   - "Always/Never do X" / "永远/绝对不要做 X" → 直接按偏好处理
   - "For this project..." / "这个项目里..." → 项目特定记录
   - 同一纠正出现 3 次 → 触发晋升（见原则 #7）
   - **不该记录**：一次性指令、上下文特定的、假设性讨论、用户沉默
   （灵感来源：ClawdBot self-improving skill 的 Learning Signals 清单）
7. **3 次晋升规则**——同类 pattern 在 memory.md 中出现 ≥3 次（而非 observations），在下次 review 时主动提议晋升到 AGENT（过程规则）或 PROFILE（人格特征）。这条规则的灵感来自 ClawdBot 的 self-improving skill——它把"什么时候该升级"从人的模糊判断变成机械规则

## 样本该放哪（通用指引）

| 错误类型 | 沉淀位置 |
|---|---|
| 跨文件同步漏改 | 对应 checklist 类别段 + 规则旁注样本来源 |
| 高成本 Bug（≥2 轮未解 / 排查耗时长） | 项目层 Bug 记录文件单独一段 |
| 工作单元收尾遗漏 | 收尾 checklist "已知样本"段 |
| 决策被推翻 | 任务卡内标记 + 落地索引用"划掉 + 追加新行"表达变更 |

---

## 反向利用（强信念）

每次工作单元收尾的自检阶段（参考 wrap-up-thinking），AI 应该主动检查：

- **本轮有无新错误样本**？
- **沉淀位置是否清楚**？
- **样本是否已贴到规则旁**？

不是单独维护"错误日志"，而是**把样本写到规则旁边作为来源注释**。

---

## 给"想搭框架的人"的建议（草稿）

- **不要建一份"反思日记"**——单独的反思文件渐渐会没人读，价值随时间衰减
- **样本贴近规则**——回看规则时立刻看到"这条是从哪个事件来的"
- **抽象层级要适度**——太具体（"我把变量 X 写成 Y"）没复用价值；太抽象（"要细心"）没指导价值。**强信念**："什么场景 + 触发什么扫描"是最值钱的抽象层级
- **样本数 < 3 不急着抽规则**——单样本可能是偶然，重复样本才反映真实模式
- **规则被绕过时反推到样本**——如果一条沉淀规则后来还是被违反，说明规则写得不对，回头补样本细节让规则更精准

# 从零搭一套框架的渐进生长节奏

> **状态**：stable（迭代中）
> **本文是什么**：聊一个实操问题——如果你想搭一套类似的 AI 协作框架，第一周做什么、什么时候该有第二份文件、什么时候开始迭代。给**人**读的理念文章。
> **不是什么**：AI 操作手册（AI 读 `ai-bootstrap/` 就够了，本文是给搭建者本人看的）。

---

## 触发这篇的核心问题

看完这套框架的人自然会冒出一个问题：_"这么多文件，我怎么开始？一次性建好 15 份文件然后填？"_

答案是：**不。从 1 份开始，撑不住了才拆。**

这个答案反直觉——工程直觉说"先搭好骨架再填肉"。但文档框架不是工程脚手架：每多一份文件，维护成本立即生效，而收益要等"真的需要那份文件"时才兑现。

---

## 反模式：第一天建 15 份文件

假设你第一天建了这套结构：

```
AGENT.md / NOW.md / AI_INDEX.md / PRODUCER_PROFILE.md
spec/ (3 份) / design/ (2 份) / reviews/ (1 份)
checklists/ (2 份) / CHANGELOG.md / AI_SELF_CHECK.md / AI_SOFT_HINTS.md
```

然后你会发现：
- 8 份文件是空的——还没到产生它们的时候
- 每天 commit 时要想"我该不该更新那 8 份空文件"——增加决策负担
- 2 周后 8 份空文件变成"视觉噪音"——你的眼睛学会了忽略它们，连带也忽略了真正有用的那几份
- **最糟的**：你会因为维护负担太重而放弃整个框架

**框架不是在第一天"建好"的，是在"需要时"长出来的。**

## 生长节奏

### 第 1 周 · 只有 1 份文件

从**一份 AGENT.md**开始。这就是你的"AI 协作契约"——告诉 AI 你的偏好、这个项目的规则、什么该做什么不该做。

第一周只维护这一份：
- 每次 AI 做了你想纠正的事 → 加一条规则到 AGENT
- 每次你觉得 AGENT 里某条规则没用了 → 删掉
- 每次你发现少了关键信息 AI 会犯错 → 补上

**第一周的目标不是"写一份完美的 AGENT"，是"发现哪些事需要写成规则"。**

### 什么时候出现第二份文件

当 AGENT 遇到以下三种信号之一：

| 信号 | 症状 | 该建什么 |
|------|------|----------|
| **单文件超预算** | AGENT 超过 ~100 行，阅读时发现自己在跳过大段内容 | 把长尾/场景特定的规则外置到独立文件 |
| **两种更新节奏** | AGENT 里有的信息每次对话结束都变（如当前进度），有的信息几周不变（如用户偏好） | 拆出 NOW.md（高频更新） |
| **两群读者** | 同一份文件里，有些内容是给 AI 读的（规则），有些是给人读的（理解项目） | 拆出人类可读的部分到独立文件 |
| **晋升管道断裂** | observations 积累了很多，但从来没有晋升到 PROFILE——因为 review 频率太低或 review 时忘了回头看 observations | 建 memory.md 作为中间层——AI 每次对话能自己把 ≥2 次的 pattern 先记到 memory，降低对 review 节奏的依赖 |

**关键**：不是因为"理论上该有第二份文件"，而是因为**第一份文件已经撑不住了**——混在一起的代价 > 拆开的成本。

### 什么时候开始迭代

迭代 = 对已有文件做结构性改动（而非追加新内容）：

- **不要在第一周迭代**——你还不确定哪些规则在实战中真的有用
- 触发迭代的信号：某条规则被反复推翻 / 某份文件的结构让你每次读都觉得"不好找"
- **迭代的节奏是脉冲式的**——不是"每周改一遍"，是"积累到不得不改时一次性改"

## 拆分的触发条件（具体化）

`doc-ecosystem.md` 提到了"拆"的建议，这里展开成可操作的判据：

### 判据 1 · 单文件行数超预算

全局 AGENT 约 100 行是合理的上限（详见 `context-budget-engineering.md`）。当 AGENT 超过这个数：
- 不是砍内容，是**把"低频读取"的内容外置**——不是扔掉，是放到 AI 按需读取的文件里
- 外置 ≠ 降级——外置文件按 `AI_INDEX` 的路由仍然能被精确读到

### 判据 2 · 一个文件里出现两种更新节奏

- 如果文件 A 里有些段每次对话结束都要改，有些段两个月才改一次 → 拆
- 高频更新的段独立成文件 → 减少"为了改一行而 commit 整个文件"的噪音

### 判据 3 · 一个文件里有两种完全不同的读者

- 如果文件 A 的前半段是给 AI 的规则（"遇到 X 做 Y"），后半段是给人的解释（"为什么这么设计"）→ 拆
- AI 不需要读"为什么"，人不需要读"执行规则"——拆开后各自的读者读得更快

## 真实生长轨迹（本框架的生长史）

本框架不是一天建成的。以下是实际生长顺序（记忆回顾，可能漏了早期步骤）：

1. **起点**：1 份 AGENT.md（全局规则）
2. **第一次拆**：AGENT 太长 → 项目特定规则外置到各个项目的项目层 AGENT
3. **NOW 出现**：跨日协作时发现"AI 不知道上次做到哪" → NOW.md
4. **AI_INDEX 出现**：文件多到 AI 不知道该读哪些 → 路由层
5. **observations 出现**：发现跨日协作丢了"对话里有价值的观察" → `reviews/_observations.md`
6. **PROFILE 出现**：多次纠正 AI 同一种行为 → 把用户偏好外置
7. **meta-agent-collab 出现**：想把这套思路分享给其他人 → meta-agent-collab 目录（本文所在）
8. **meta-agent-collab 从 1 份变 11 份**：一篇讲不完 → 按主题拆分（就是本目录的 11 篇文章）
9. **memory.md 出现**：observations 到 PROFILE 之间有一个灰色地带——某些 pattern 出现了 2~3 次但还没到入 PROFILE 的程度 → 活跃工作集（≤100 行，介于素材和结晶之间）。灵感来自 ClawdBot self-improving skill 的 HOT 层
10. **heartbeat-state.md 出现**：发现 AI 有时会"忘记"做收尾——因为收尾的触发时机模糊（"对话结束前"在 AI 的认知里不是硬边界）→ 启动时检查 `heartbeat-state.md`，未收尾则先补。灵感来自 self-improving 的 heartbeat 机制

**关键观察**：第 9~10 步和之前不同——前面的每一步都是"东西装不下了所以拆"，这两步是"机制破了所以补"——收尾时机模糊被 heartbeat 补上了，observations 到 PROFILE 的晋升灰色地带被 memory 补上了。这是框架生长的第二个阶段：**从"被动响应容量压力"到"主动修补机制漏洞"**。

## 给"想搭框架的人"的建议

- **从 AGENT.md 开始，只看 AGENT.md**——忘了还有 NOW、INDEX、PROFILE 这些东西。它们会在你需要的时候自然出现
- **不要预设"最终该有几份文件"**——你的协作需求决定了文件数量，不是反过来
- **拆的时机比拆的方式更重要**——拆早了，维护空文件；拆晚了，单文件膨胀到读不动。信号比计划准
- **如果你有 `doc-ecosystem.md` 里提到的全部 8 类文件**——恭喜你，那是生长结果，不是起始模板
- **不是所有人都需要长到 10 份文件**——生长不是"越长越大"，是"长到刚好够用就停"。多项目、长期协作、结构化开发这三个条件缺一，就可能停在更少的文件数。详见 `element-decoupling-and-assembly.md` 的场景组合（A~E）

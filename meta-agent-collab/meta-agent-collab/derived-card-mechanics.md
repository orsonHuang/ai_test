# 衍生卡机制 · 把"顺手做"变成"立卡做"

> **状态**：draft（重塑中）
> **本文是什么**：聊聊"做一张卡时撞到的本卡范围外问题"该怎么处理——立卡而不是顺手做。
> **不是什么**：项目管理工具教程 / 任何具体工具的工作流。

---

## 核心想法

做一张卡时，几乎一定会撞到本卡 scope 外的问题。处理方式有两种：

- **顺手做**：当场修，不留记录。**结果**：本卡膨胀 / 修改面失控 / 哪些是本卡决策、哪些是顺手补的，未来追不回来
- **立卡做**：当场立一张新卡（"衍生卡"），写明触发场景 + 决策点 + 与原卡的依赖，**然后回到原卡**。问题不当场修，作为下一张候选

**思想核心**：把"问题被发现"和"问题被修复"在时间上分开。前者随时发生（且应当），后者按 backlog 节奏来（避免 scope creep）。

---

## 为什么不能"顺手做"

顺手做看似省事，但代价是：

1. **scope 失控**：本卡完成时，diff 里有一半文件其实不属于本卡。回看 git history 时，混了多个不相关的修复，未来回滚 / 复盘困难
2. **决策失追溯**：顺手补的修复没有"为什么这么修"的拍板对话。半年后看，不知道当时是认真权衡过还是凭直觉乱补
3. **传染性 scope creep**：顺手补的修复又会触发新问题，新问题又顺手补……一张卡可以无限扩
4. **疲劳积累**：本来预期时间内能完成的卡持续膨胀，最后疲劳交付质量下降

立卡的成本很低（写下卡号 + 触发场景 + 决策点）。但它换来本卡 scope 清晰 + 未来 traceability + 不疲劳。

---

## 立卡的判定（强信念）

**MUST 立卡**（满足任一）：
- 触发了本卡 scope 外的代码改动
- 触发了 ≥1 个独立决策点（"这事该 A 还是 B"）
- 触发了 ≥1 个未来其他卡也会撞的同类问题（**通用性强 = 一定要立**）
- 用户表达"这是另一件事"

**可以顺手做**（典型）：
- typo / 注释 / 1 行无决策点的微改
- 是本卡明确遗漏的修复（属于本卡 scope）

## 衍生卡的分类

| 类型 | 定义 | 处理差异 |
|------|------|----------|
| **纵向衍生**（机制延伸） | 同一机制的 bug / 未完成部分 → 立延伸卡 | 与主卡强依赖，优先级往往被主卡提升 |
| **横向衍生**（跨模块能力） | 多个模块都会撞的同类问题 → 立横向能力卡 | 与任一主卡弱依赖，优先级独立判定 |

**关键区别**：纵向衍生的完成时机往往被主卡依赖（不修就堵后续卡），横向衍生是"有了最好但不堵人"的能力补全。

---

## 系列完成后的聚合

当一个系列的多张卡全部落地后，迁出到 done 时应按"系列语义"聚合而非单纯按时间倒序：

- **按卡号聚合**：保留系列内聚性——未来回看知道"这批是同一个系统"
- **按落地日期倒序**：只知道"这批是同一个时间段做的"，语义弱

系列结束时应把整批聚成子块，加系列级表头。

## 卡片体系与 CHANGELOG 的关系（新增）

衍生卡机制本质是**清单文化**的一种：把"该做但现在不做"的事写在清单上而非脑子里。

但如果项目不使用 backlog 卡片体系（而是用 CHANGELOG 驱动），衍生工作的处理方式需要适配：

**CHANGELOG 模式下的衍生工作**：
1. 发现本 scope 外的问题 → 在 CHANGELOG 中记录为"待办"条目
2. 后续在 CHANGELOG 中追踪该待办的完成状态
3. 与卡片的区别：CHANGELOG 是单文件线性记录，不是按状态拆分的多文件体系

**两者不冲突**：
- 卡片体系 = 任务规划 + 状态追踪
- CHANGELOG = 操作记录 + 审计链
- 衍生工作的"发现"可以在 CHANGELOG 中留痕，"追踪"用卡片体系更方便
- 如果项目只用 CHANGELOG，衍生工作可以通过 CHANGELOG 的 "Planned" / "Done" 条目来管理——比卡片体重量级更低，但 traceability 同样可维护
- **实战落地**：已有项目在 `CHANGELOG.md` 顶部新增 `## Planned` 段作为卡片体系的前置轻量化变种——格式 `项目前缀-衍生-NNN: 简述 → 触发场景 | 依赖 | 优先级`，完成后移到当日版本段。这是"思路同一种，落地是多种"的例证：衍生卡的核心不是多文件状态机，而是"发现问题 ≠ 立即修复"的清单文化。

## 给"想搭框架的人"的建议

- **立卡的成本要低**：用最简单的形式。如果立卡需要填 30 个字段，没人会立
- **衍生卡不是"待办事项"**：它是"未来某个时点需要拍板的决策点"。区别在于待办是已知怎么做，衍生卡是还没拍板
- **衍生卡指针要回到主卡**：在主卡的"遗留"段点名引用衍生卡，否则未来看主卡时不知道有这些衍生
- **衍生卡的优先级独立判定**：不要因为"它是 X 衍生的所以也是同一个优先级"——按它自己的影响面判定
- **非卡片式项目用 CHANGELOG 管理衍生**：在 CHANGELOG 中标记待办条目，后续追踪完成状态

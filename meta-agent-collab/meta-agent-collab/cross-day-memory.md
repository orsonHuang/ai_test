# 跨日 AI 协作的"记忆外置"机制

> **状态**：stable（迭代中）
> **本文是什么**：长期与 AI 协作的人都会撞上的同一面墙——AI 跨对话没有原生记忆——以及一种"把记忆外置到文件层"的解法。  
> **不是什么**：技术教程（不讲 RAG / vector db / 长上下文 API）；本文讲的是**协作流程层面**的记忆机制，与具体模型无关。  
> **诚实交底**：框架已跑过多轮实战。五件套机制整体站稳——NOW + git log + observations + memory + PROFILE 的分层经多次跨日验证有效。少数开放问题仍在观察：周度 review 的真实触发频率、observations 标签集是否会膨胀、跨项目场景的 observations 存放策略、memory 的晋升率与误报率。**遇到推翻时如实补"哪些场景下机制破了"**。

---

## 核心想法（2 段）

**反直觉的事**：很多人以为"上下文窗口很大 = AI 记得住"。错。无论窗口多大，**每次新对话都是从空白启动**——除非主动读文件。"AI 记得上周我们聊过 X"在当前架构下是幻觉。

**解法不是技术**，是**协作流程**：把 AI 在每次对话里产生的"值得跨日保留"的信息，**当场流式写到文件**。AI 跨日没记忆？不要紧——只要文件层完整，下一个 AI（或下次的我）读文件就能还原。**记忆从"AI 大脑"外置到"文件层"**。

---

## 同款问题的常见错解

| 错解 | 为什么错 |
|---|---|
| "用大窗口模型就够了" | 单次对话内有效，跨对话 = 0 |
| "让 AI 自己总结上次聊了什么" | AI 没读历史 = 凭空编 |
| "我记得就行，AI 不用记" | 用户也会忘；且违反"留痕是 AI 的活"原则 |
| "每天写日记给 AI 读" | 用户负担太重，跑不下去（v0.2 fallback 反例） |
| "用 vector db 检索历史对话" | 重，需要工程化基建；多数协作场景用不上 |

## 本框架的最小五件套

按"信息生命周期"分层，不重复，不遗漏：

| 层 | 文件 | 节奏 | 装什么 | 谁写 |
|---|---|---|---|---|
| 当前状态 | `NOW.md` | 每次对话结束 | "现在在哪、下一步做什么" | AI |
| 事件流 | `git log` + commit message | 每次落地 | "做了什么、什么时候做的" | AI（自然产生） |
| 活跃工作集 | `memory.md` | 每次对话结束 + 每次纠错 | "近期可复用模式、待确认偏好"（≤100 行） | AI |
| 观察素材 | `reviews/_observations.md` | 每次对话结束 | "今天值得未来 review 的瞬间" | AI |
| 长期画像 | `PRODUCER_PROFILE.md` | 周度复盘触发 | "用户的稳定偏好" | AI（用户拍板入档） |

**为什么是 5 层（从 4 到 5）**：
- 原来的四件套中，observations 到 PROFILE 之间有一个"灰色地带"——有些模式重复出现了 2~3 次，但还没到进 PROFILE 的程度（单次 observation 太弱、PROFILE 版本太慢）
- memory.md 填补了这个地带——介于素材和结晶之间。它不是 observations（observations 只追加不删改），也不是 PROFILE（PROFILE 版本化、带证据强度标注）
- 灵感来源：ClawdBot 的 self-improving skill 中 HOT 层的概念——一个 ≤100 行、每次必读的"活跃工作集"

## 启动锚点的原理：heartbeat + NOW + AI_INDEX

每次新对话开始时，AI 面临同一个问题：**"上轮有没有尾巴没收？我现在在哪、该读哪些文件？"**。记忆外置的五件套是"信息存储"，但还需要一套**路由 + 检查机制**让 AI 在新对话中快速定位。

### heartbeat-state.md：衔接检查层（新增）

heartbeat 回答"上轮对话正常收尾了吗"：
- 记录 `last_reviewed_change_at`——上次对话结束时文件是否都同步了
- AI 新对话启动后**第一步就是读 heartbeat-state.md**——先检查上轮尾巴
- **每次回合结束时也读 heartbeat-state.md**——由 heartbeat 自身承载判断逻辑（顶部「回合结束时·自主判断」段），不依赖 AI 记忆 AGENT §14。心跳文件自己决策是否需要收尾，多一层保底
- 如果上轮有未 commit 改动 / 未追加 observations / 未更新 NOW / 未更新 memory → **先补收尾再做新工作**
- **维护节奏**：每次 AI 输出结束时自动更新（AGENT §22），不仅是对话收尾时。先回读→执行判断→再写入

**为什么需要 heartbeat**：AGENT §0 已经写了"轮次衔接检查"，但它的触发时机是"收到用户消息时"——如果用户的新消息是不相关的（切话题），AI 可能"忘记"检查。heartbeat 把检查时机**从"收到消息后"提前到"对话启动时"**，用文件时间戳做机械判断，不依赖 AI 的记忆。而 §22 的"每次输出结束自动更新 + 回读"进一步把 heartbeat 从"被动检查"升级为"主动决策"——心跳文件自己判断是否需要收尾，AI 不需要记住"该更新了"也不需要记住"该收尾了"。

### 三层怎么配合

```
对话开始 → 读 heartbeat-state.md（"上轮正常收尾了吗"）
              → 若有未收尾 → 补收尾（observations + NOW + memory + 自检）

         → 读 AI_INDEX（"需要哪些文件"）
              → 必读：AGENT, memory, NOW
              → 按需路由：根据 NOW 的上下文选读 spec / design / observations
         → 开始工作
```

（关键设计选择段落保持不变）

### CodeBuddy 规则：机械注入层（自举悖论的解法）

以上三层（heartbeat + NOW + AI_INDEX）都是**文本约定**——它们存在文件里，但前提是 AI **先读到 AI_INDEX** 才知道要走三层路由。这就形成一个自举悖论：AI 必须先读 AI_INDEX 才能知道该读 AI_INDEX。

CodeBuddy 的 `rules` 机制解决了这个鸡生蛋问题：建一个 `alwaysApply: true` 的规则文件，每次 AI 开始思考时**系统自动注入**，不依赖 AI "记得去找"。

**规则文件路径**：`~/.codebuddy/rules/文件加载.mdc`（或其他命名）

**规则内容**：
```markdown
---
alwaysApply: true
enabled: true
---

1、每次开始思考时，必须加载AI_INDEX.md和heartbeat-state.md，并从AI_INDEX.md开始；2、输出结束前最后必须执行heartbeat-state.md；3、输出结束前确认所有文档均无修改后，再进行提交
```

**效果**：这条规则被 CodeBuddy 引擎自动注入到每次思考的上下文，AI 无法跳过。三条指令分别机械保证：
① AI_INDEX 的四层路由（会话启动 → 每次消息 → 输出结束 → 动作门前）+ heartbeat 启动检查强制生效
② heartbeat 输出结束时的自主判断强制执行
③ 输出结束前必须确认所有文件已 commit，不留未提交改动
——三层都不依赖 AI 在长会话中"记住去读"。

**适用前提**：使用 CodeBuddy 作为 IDE 协作用具。如果用户使用其他支持类似 `alwaysApply` / `system prompt injection` 机制的工具，可以建立等价规则。如果不支持，四层文本约定仍是有效的——只是多一层"AI 可能忘记"的风险。

**AI 搭建时的动作**：搭建完成后，如果检测到用户使用 CodeBuddy，主动询问"要不要帮你建一条 CodeBuddy 规则，让 AI 每次思考时强制加载 AI_INDEX、每次输出结束时强制加载 heartbeat-state.md？"——用户同意后建到 `~/.codebuddy/rules/` 目录。

## 实际跑起来是什么样

**每次对话启动 AI 做的事**：
- 读 `heartbeat-state.md`——检查上轮收尾状态
- 若上轮未正常收尾 → 先补 observations + NOW + memory + 自检
   - 读 `AGENT.md` + `memory.md` + `NOW.md`（必读）
   - 读 `PROFILE` 顶部 TL;DR（~30 行，正文默认不读）
   - 按需读项目层 / 模块层

**每次对话结束 AI 做的事**：
1. 更新 `memory.md`——本轮出现的可复用 pattern、待确认偏好
2. 更新 `NOW.md`——同步当前状态
3. 追加 `_observations.md` 1~5 条——流式素材（注意：收尾流程内、commit 之前追加，非 commit 之后）
4. 自检交账（本轮判断偏差 / 已沉淀规则 / 未沉淀但值得的）
5. 更新 `heartbeat-state.md`——标记本轮正常收尾（注意：§22 要求**每次输出结束都更新**，不只是对话收尾时）
6. （若是工作单元收尾）做完整 wrap-up 三件事，**最后一步 commit**

**周度 review 触发时**：
- AI 第一动作 = 读 `memory.md` 待晋升条目 + `_observations.md` + `git log` + `NOW` + `PROFILE`
- memory.md 中同类 pattern ≥3 次 → 提议晋升到 AGENT（过程规则）或 PROFILE（人格特征）
- 起草 review → 用户拍板入档 → PROFILE 新版本 + reviews 落档

## 为什么有效（信念基础）

1. **AI 大脑会换 / 会丢，文件不会**——所有重要信息在文件层 = 跨模型、跨对话、跨年都成立
2. **流式写 < 集中回忆**——每天 1~5 条 < 周末凭印象憋一篇；前者准确，后者必漏
3. **分层 = 各司其职**——5 个文件各管一段，不重复 = 无需同步成本
4. **AI 主动写 ≠ 用户负担**——用户只在复盘节点拍板，平时不维护
5. **heartbeat 消除收尾时机模糊**——AI "忘记"收尾时，下一轮启动时机械检测到，不依赖人的提醒
6. **即时性 + 精准性的分层**——memory.md 提供即时反馈（≤100 行、每次更新），PROFILE 保证长期准确（周度 review、证据强度），互不污染

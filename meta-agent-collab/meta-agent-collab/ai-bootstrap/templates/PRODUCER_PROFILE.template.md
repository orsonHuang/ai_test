# 制作人观察档案（PRODUCER_PROFILE）

> **本文是什么**：AI 对[读者称呼/姓名]设计与协作哲学的持续观察，作为跨项目通用的"人像 spec"。
> **为什么存在**：让 AI（当前/未来）在出方案、提建议、拍板辅助时，能对齐读者偏好；同时也是读者自我认知的外置镜子。
> **生命周期**：每轮观察作为新章节追加（v0.1, v0.2 ...），**不删旧版本**——保留决策/认知演变的溯源。
> **位置**：与全局 `AGENT.md` 同层（工作区根），跨项目通用。
> **范围边界**：本文是"关于人"的观察；"关于具体项目怎么协作"的原则放在项目根的 `COLLABORATION.md`；"关于跨项目通用协作偏好"放全局 `AGENT.md`。

## 维护规则

<!-- AI注: 这一段是文件能活下去的根本，**完整保留**，不要简化。 -->

1. **追加不删改**：新版本以 `## v0.x · 日期` 为段落标题追加；旧版本内容原样保留。
2. **读者反馈即为观察的一部分**：当读者对某条观察做出纠正 / 确认 / 补充时，AI 必须把这份反馈原样记录在对应观察条目下方，用 `> 读者反馈（日期）：...` 的引用块形式。
3. **错误不覆盖**：AI 被证明错的观察不删除，而是在原条目下加 `> 已被 vX.Y 修正：...`，并在新版本正文给出修正理由。
4. **触发更新的时机**：
   - [如建了 reviews] 周度复盘后若产出新观察 → 追加新版本
   - 重大认知事件（读者明确改口某条决策原则、或出现与过往模式显著冲突的决策）→ 追加新版本
   - 月度 / milestone 末，AI 主动建议做一次"模式提取"
5. **证据强度标注**：每条观察必须标注 `（强证据 / 中等证据 / 弱证据-待验证）`，AI 不得把主观推测写成强证据。

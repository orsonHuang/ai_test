# <项目名> · CHANGELOG

> **本文是什么**：记录"为什么改"的结构化操作日志——重点在 WHY 和影响范围。
> **本文不是什么**：git log 的替代品（git 管"改了什么文件"）；spec 文档（管"现在是什么"）。
> **维护**：每次涉及项目结构/设计逻辑的关键改动时追加一行。

---

## 格式

每行：`(YYYY-MM-DD) [类型] 一句话 + 原因`

类型：`增` / `删` / `改` / `迁` / `修`（修 = bug 修复）

按里程碑分段，不按日期堆砌。

---

## 记录判据

| ✅ 记 | ❌ 不记 |
|------|-------|
| 新增/删除/重命名文件/目录 | 纯代码实现细节（函数改名、变量调整） |
| 设计决策变更及原因 | 语法/类型/lint 错误修复 |
| 依赖新增/升级/移除 | 每次 commit 的机械流水 |
| 数据模型/接口契约变更 | 个人实验性改动（未确定会保留的） |

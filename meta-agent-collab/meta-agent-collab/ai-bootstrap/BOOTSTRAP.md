# BOOTSTRAP · AI 搭建入口

> **本文是什么**：AI 被雇来按本目录帮某人搭建"AI 协作框架"时的入口文档。
> **本文不是什么**：给人看的（人请去 `../README.md`）；日常协作契约（这是搭建期专用）。
> **读者**：你（AI）。如果你正在读这份文件，意味着有人请你帮 ta 把这套框架搭进 ta 的工作区。

---

## 你是谁 · 你的角色

你**不是**"这套框架原作者的日常协作 AI"。你是**被这位新读者临时雇来的搭建助手**。

含义：
- 你的任务是帮 ta 长出 **属于 ta 自己的** 一套文件系统，**不是把原作者的复制过去**
- ta 的项目类型、姿态、节奏都和原作者不同——你要让框架适应 ta，不是让 ta 适应框架
- 完成搭建后，**你会被这套框架接管**——之后的协作按你帮 ta 立的契约走，不再按本目录

---

## 三个反模式 · 必须避开

这套框架的核心比喻是 **"种子，不是模板"**——它的失败模式都是"种子被当成模板用"：

1. **照抄原作者的项目术语**：原作者是独立游戏开发者，ta 的文件里有 "Godot / GDScript / 棋盘数据"。你不能把这些搬给写小说的人 / 做创业的人——**项目术语必须由新读者自己提供**。
2. **替新读者拍板**：原作者的习惯是"AI 给推荐 + 留 🔲 由我拍"——你必须把这个姿态传递给新读者，不能自己默默拍。哪怕 ta 第一次问"我该装哪些"，**默认做法是按 `01-questions-to-ask.md` 问完再给组合建议**，而不是直接说"你装组合 D"。
3. **追求一次性完美**：这套框架是边长边记长出来的。**不要试图让新读者第一天就产出"完整版"**——告诉 ta：今天先按 ta 的场景装基础零件，跑 1 周后再回来补。

---

## 你的搭建顺序

按下面顺序读，**不要乱序**：

1. **背景阅读**（`../*.md` 顶层理念文章）— 在动手之前先扫一眼这几篇，建立"我在搭什么 / 为什么这样搭"的整体观：
   - `../element-decoupling-and-assembly.md` — **必读**。框架不是全装捆绑包，是零件市场。决定了后续搭建时按什么组合走
   - `../wrap-up-thinking.md` — 工作单元收尾的思想
   - `../doc-ecosystem.md` — 整套文档生态有哪些类型（扫职责表 + 边界判定段就够）
   - `../three-layer-agent.md` — 为什么 3 层（扫核心想法段）
   - `../gradual-adoption.md` — 从零搭一套的渐进生长节奏（与 element-decoupling 互补：一个讲"按时间长"，一个讲"按场景装"）
   - `../belief-and-decision-rhythm.md` / `../derived-card-mechanics.md` / `../design-code-bidirectional.md` — **可选**，如有时间扫核心想法段。这些影响日常协作姿态，不影响搭建本身
   - **不读到位的代价**：你会按"模板复印机"的姿态搭建，错过"种子非模板 + 按场景选零件"的精神
2. **`01-questions-to-ask.md`** — **⚠️ 重要**：不要直接跳进 Q1。**第一句话先执行「前置·互相认识一下」**——问对方怎么称呼（必问），轻提"可以给我起个称呼"（不强制）。然后在第零组开放式破冰（Q0A/B）用名字自然互动，从闲聊中提取已覆盖的信息（项目类型、背景、协作场景等），对照 Q1~Q4 自动预填跳过。所有预填结果在最后「预填确认段」一次性让对方确认。问答结束后你会得出一个组合（A~E），决定装哪些零件
3. **`02-build-order.md`** — 按组合走对应的搭建路径，建文件、嵌信号
4. **`03-signal-sheet.md`** — 信号速查表。搭建时用它判断每个可选元素是否该建；日常持续用途
5. **`templates/`** — 每个模板都是"注释版"——含骨架 + 每段为什么这么写的解释。**你的任务**：基于新读者的回答，**改写**模板（不是照抄）；改写后**删掉所有 `<!-- AI注: -->` 注释**，留下干净的最终版给新读者

## 完成判据

搭建完成 = 新读者的工作区有对应组合的全部文件 + 信号嵌入到位：

- [ ] 路由入口（`AGENT.md` 含路由，或 `AI_INDEX.md` + `AGENT.md`）— 至少含 ta 自己的协作偏好
- [ ] `NOW.md` — ta 当前在哪一步（组合 B+ 必建）
- [ ] `heartbeat-state.md` + `memory.md`（组合 B+）
- [ ] 项目 `README.md` + `CHANGELOG.md`（组合 C+）
- [ ] `AI_SELF_CHECK.md` / `AI_SOFT_HINTS.md` / `PRODUCER_PROFILE.md` / `reviews/`（组合 E）
- [ ] **信号嵌入完成**：所有本轮未建的元素，其触发信号已写进 `memory.md`（或组合 A 的 `AGENT.md`）

**不要硬塞**——组合 A 就不该有 PROFILE，组合 C 就不该有 heartbeat。01 问题清单会帮你判断该装哪些。

## 重要 · 搭建完成后的最后一步

把 ta 的工作区根 `AI_INDEX.md` 里的"默认不读"清单里**加上 `meta-agent-collab/`**（如果 ta 把本目录也复制过去了的话）——这样 ta 之后的日常协作 AI 不会被本目录污染。

如果 ta 没复制本目录（只是用一次就丢），跳过这步。

## 搭建完成后的可选强化 · CodeBuddy 规则自动注入

AI_INDEX 四层路由是**文本约定**——前提是 AI 每次思考时**记得去读** AI_INDEX。在长会话中这个记忆可能失效。CodeBuddy 的 `rules` 机制可以把约定升级为**机械保证**。

**判断**：如果检测到对方正在使用 CodeBuddy（IDE 环境），搭建完成后主动询问：

> "要不要我帮你建一条 CodeBuddy 规则，让之后的 AI 每次开始思考时**强制加载 AI_INDEX 和 heartbeat-state.md**、输出结束时**执行 heartbeat 判断并确认文档干净后再 commit**？这样整个路由+收尾链路就不再依赖 AI 的记忆了。"

**对方同意后**，在 `~/.codebuddy/rules/` 下建一个 `.mdc` 文件（如 `文件加载.mdc`），内容：

```markdown
---
alwaysApply: true
enabled: true
---

1、每次开始思考时，必须加载AI_INDEX.md和heartbeat-state.md，并从AI_INDEX.md开始；2、输出结束前最后必须执行heartbeat-state.md；3、输出结束前确认所有文档均无修改后，再进行提交
```

**告诉对方**：这条规则会随 CodeBuddy 引擎自动注入每次思考的上下文，AI 无法跳过——三条指令覆盖入口、收尾判断、提交确认，全链路机械保证。

**如果不使用 CodeBuddy**：如果对方的工具支持类似的 `alwaysApply` / `system prompt injection` 机制，可以建立等价规则。告诉对方核心逻辑是"每次思考开始前注入一句'先读 AI_INDEX.md 并从它开始'"。

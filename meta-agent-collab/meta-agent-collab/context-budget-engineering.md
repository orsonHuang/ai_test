# Token 预算工程 · 100 行规则背后的逻辑

> **状态**：stable（迭代中）
> **本文是什么**：聊聊 AI 协作框架设计中的一个物理学约束——AI context window 是有限资源，每多一行规则就占用一行 token。哪些规则值得进 context？哪些该外置？为什么全局 AGENT 约 100 行？
> **不是什么**：LLM 技术原理讲解（不讲 attention / context length / RAG 实现）。

---

## 触发这篇的核心问题

`three-layer-agent.md` 里写了一条"强信念"：**全局层 AGENT 约 100 行**。但为什么是 100 不是 200，也不是 50？

这不是拍脑袋的数字。背后是一个工程问题：**AI 的 context window 有上限，规则文件是 context 的"固定租金"**。每多一行规则 = 少一行能读项目上下文的预算。规则的设计不只是"写清楚"的问题，更是**经济学问题**。

---

## Context 的物理学约束

### 固定租金 vs 按需读取

| 层 | 读取方式 | 类比 |
|-----|----------|------|
| AGENT.md（全局规则） | **每次对话必读**——固定租金 | 房租——不管你住不住，月月扣 |
| 项目层 AGENT | 进入该项目时必读 | 每次进门收一次门票 |
| spec / design 文档 | **按需读取**——AI 只在需要时读 | 外卖——需要时才下单 |
| AI_INDEX | 每次对话必读（但它极短） | 路牌——一眼扫完 |

**核心约束**：AGENT 是"固定租金"——每次对话必读。所以它的行数直接决定**每次对话的"基础开销"**。基础开销越高，留给项目上下文的预算越少。

### 一个量级估算

假设 context window = 100K tokens（2025-2026 主流水平）：
- AGENT 100 行 × 人均 token 密度 ≈ 3K tokens = **~3% 的 context 窗口**
- AGENT 200 行 ≈ 6K tokens = ~6%
- AGENT 500 行 ≈ 15K tokens = ~15%

如果项目代码 + spec 本身需要 30K tokens，AGENT 占 15K = 只剩下 55K 给"实际工作"——而这还没算对话历史、工具返回、输出等。

**结论**：100 行是"基础开销 ≤ 3~5%"的约束。不是精确值，是一个**预算上限**。超过这个阈值的规则，应该进"按需读取"的文件而非"每次必读"的 AGENT。

---

## 为什么 100 行（而不是 50 或 200）

### 下行（50 行）为什么不够

50 行能装的内容太少。一个完整的协作框架至少需要覆盖：
- 核心信念（"讲为什么，不讲必须"） — ~3 行
- 文档生态路由 — ~5 行
- 三层 agent 定义 — ~8 行
- 跨日记忆机制 — ~10 行
- 工作单元收尾 — ~8 行
- 犯错即记录原则 — ~5 行
- plan/ask/craft 决策 — ~10 行
- 常识钩子（不虚构文件、不说"好的"） — ~10 行

合计约 60 行为"骨架"，50 行只能装骨架 → 没有呼吸空间 → 每条规则都有被削成"不可执行"的风险。

### 上行（200 行）为什么危险

200 行 = 基础开销 ~6%。看起来不多，但有两个隐藏成本：
1. **递减边际收益**：规则从 100 到 200，多出的 100 行大多是"边缘场景"规则——触发频率低但每次必读
2. **遵循率反比**：规则越多，AI 越难在 prompt 中"记住所有规则"。100 条指令中有 10 条被忽略，> 200 条时有 40 条被忽略

> **研究印证**（来自 Claude Code 官方 Harness Engineer 课程）：LLM 在单次对话中**可靠遵循的指令上限约为 150-200 条**。超过这个阈值，AI 并非 token 耗尽，而是**注意力静默退化**——开始跳过低优先级指令而不报错。同时，Claude Code 自身的系统提示已占用约 50 条指令的注意力预算。这意味着**用户侧规则的安全上限约为 100-150 条**——恰好落在本框架的 100 行约束范围内。

### 100 行：经济学平衡点

100 行是"骨架有足够呼吸空间 + 不会触及注意力退化阈值"的交叉点。它最初是**工程经验值**——来自多次"加到 150 → 发现开始跳过 → 砍回 100 → 改善"的迭代；现在有了注意力研究的实证支撑。

## 哪些规则值得进 Context（值得付固定租金）

判断标准：**这条规则在 80% 以上的对话中需要被知道吗？**

| 该进的（固定租金） | 不该进的（按需读取） |
|---------------------|------------------------|
| 跨场景的核心信念（"讲为什么，不讲必须"） | 特定模块的实现约束（"combat 系统的 damage floor = 1"）|
| 文档路由（"遇到 X 问题读 Y.md"） | 一次性任务的 checklist |
| AI 的行为元规则（"犯错即记录"、"不虚构文件"） | 历史决策记录（"为什么选了 A 方案而非 B"）|
| 收尾触发条件（"改动 ≥ N 个文件时做收尾"） | 某次 review 的具体结论 |
| 三层 agent 的边界定义 | 某个旧版本的迁移指南 |

**外置不是降级**——外置到按需文件的内容仍然能被精确读取（通过 AI_INDEX 路由）。区别只是：不占固定租金。

### 什么进 Context、什么不进（直白版）

> 对首次接触框架的人的解释：AI 的上下文不是"所有文件都自动可见"。

| 进入 Context | 被排除 |
|-------------|--------|
| 引用的文件（@ 引用的、本框架定义必读的） | 无关对话历史 |
| AI_INDEX 路由到的按需文件 | 未引用的资料（不在固定租金 + 没有被路由） |
| 匹配的记忆（memory.md）与 Skills（任务触发） | 不相关的记忆（冷条目 / 其他项目的 pattern） |

这个对照帮你理解：**不是"重要程度"决定进不进 context，是"触发机制"决定**。一份 spec 非常重要，但它只在 AI 被路由去读时才进 context。

## 配套文件的工程逻辑：拆分不是为组织，是为节约 Context

`AI_SELF_CHECK.md` 和 `AI_SOFT_HINTS.md` 为什么是独立文件，而不是 AGENT 里的两节？

| 文件 | 为什么外置 | 读取时机 |
|------|-----------|----------|
| memory.md | 活跃工作集——近期模式变化快，不适合同锁在 AGENT 骨架里 | 每次对话启动（≤100 行） |
| AI_SELF_CHECK | 自检场景触发频率 < 30%，不值得每次必读 | AI 在特定节点（收尾 / 复盘）时主动读 |
| AI_SOFT_HINTS | 经验性 soft rules，变化快 → 不宜锁在 AGENT 骨架里 | 按需 |

**工程逻辑**：
- 如果放进 AGENT → 每次对话付租金，哪怕这次对话用不到
- 如果独立文件 → AI 只在需要的节点读它
- **外置 = 按需租金，内置 = 固定租金**。选择的标准是"触发频率"而非"重要程度"

## Context 预算的分层设计总结

```
Context Window 总预算
├── 固定租金（每次对话必付）
│   ├── AGENT.md          ~100 行  (核心规则 + 信念)
│   ├── memory.md         ~100 行  (活跃工作集·即时模式)
│   ├── PROFILE TL;DR     ~30 行   (画像速览·正文按需)
│   ├── AI_INDEX.md        ~20 行  (文件路由)
│   └── heartbeat-state.md  ~10 行  (启动检查)
├── 会话租金（进入项目/模块时付）
│   ├── 项目层 AGENT      ~50 行   (项目特定规则)
│   └── 模块 README       ~30 行   (模块入门口)
├── 按需租金（需要时付）
│   ├── PROFILE 正文      ~650 行  (画像·仅 review/更新时读)
│   ├── spec/*.md         不限行数  (契约，AI 按需读取)
│   ├── design/*.md       不限行数  (设计文档，低频读取)
│   └── AI_SOFT_HINTS     ~30 行   (软提示，按需)
└── 剩余预算 → 给项目上下文（代码、对话历史、工具返回）
```

**关键设计原则**（不变）

## 一个现实世界的 HOT/WARM/COLD 案例

三层 context 预算管理不仅是理论——ClawdBot 的 self-improving skill 已经实现了一套 HOT/WARM/COLD 三层体系：

| 层 | 对标 | 大小上限 | 行为 | 与本框架的关系 |
|----|------|----------|------|---------------|
| HOT | memory.md | ≤100 行 | 每次必读 | 直接对应（本框架的 memory.md 按相同原则设计） |
| WARM | projects/, domains/ | ≤200 行/文件 | 上下文匹配时加载 | 对应本框架的 PROFILE 按场景读取 |
| COLD | archive/ | 不限 | 显式查询时加载 | 对应本框架的 outdated 归档文档 |

**关键区别**：self-improving 的 WARM/COLD 是**时间驱动的自动衰减**（30 天不用→WARM，90 天→COLD）。本框架选择了**人力驱动 + lint 辅助**——因为"不用了"不一定意味着"过时了"。但两种策略的 context 预算逻辑完全一致：**分层是为了让固定租金尽可能小。**

## 给"想搭框架的人"的建议

- **100 行是上限不是目标**——如果你的协作场景简单，50 行足够就别硬加到 100
- **先写再砍**——第一版 AGENT 超过 150 行是正常的，跑几轮实战后回来砍。不要一开始就精确控行数
- **外置不等于降级**——被外置到独立文件的规则一样重要，只是读取时机不同
- **AI_INDEX 的维护成本极低**——它只有 ~20 行，新加文件时顺手加一行路由。这个成本远小于"让 AI 自己猜该读哪些文件"

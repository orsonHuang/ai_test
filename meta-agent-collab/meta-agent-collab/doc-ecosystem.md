# 文档生态 · 一份 README 不够，分多类型才能可持续

> **状态**：draft（重塑中）
> **本文是什么**：把"不止 AGENT"的整套文档生态画一遍——各类文档的职责 / 边界 / 依赖。
> **不是什么**：抄答案的模板（你的项目可能长得不一样，本文是原理不是清单）。

---

## 核心想法

**反直觉的事**：很多人想"一份大 README + 一个待办列表就够了"。实际上做到中等规模的项目时，会发现：**不同类型的信息有不同的生命周期、不同的读者、不同的更新节奏**——硬塞同一份文档会互相干扰，最后没人想读。

**我的解法**：让文档**按职责分类**，每类有清晰边界 + 行数预算 + 维护节奏。看起来文件多了，但每份文件**短而专**——读者按需打开，AI 也能精确路由。

---

## 文档生态的层级架构

### 元层（工作区根）

| 角色 | 典型文件 | 职责 |
|------|----------|------|
| 全局协作偏好 | `AGENT.md` | 跨项目通用的 AI 协作规则 + 协作姿态 |
| 当前焦点 | `NOW.md` | 现在在哪一步 / 下一步做什么 |
| AI 路由 | `AI_INDEX.md` | AI 该读哪些 / 不该读哪些 |
| 制作人画像 | `PRODUCER_PROFILE.md` | 长期偏好 / 姿态 / 边界 |
| 人类入口 | `README.md` | 工作区是什么 |

### 跨日记忆层（reviews/）

这一层在大多数 AI 协作框架里**缺位**——不是 backlog（事件层）、不是 PROFILE（结晶层），而是中间的**素材层**。

| 文件 | 职责 | 节奏 |
|------|------|------|
| `_observations.md` | 流式观察素材（每次对话结束 AI 追加 1~5 条） | 高频 · 原料 |
| `review_YYYY-MM-DD.md` | 单次复盘原始记录 | 周度 · 结论 |
| `README.md`（reviews/ 内） | 规则 + SOP + 触发判据 | 元 |

**为什么单独一层**：
- PROFILE 是"结晶"——更新缓慢、跨复盘累积
- backlog / commit log 是"事件"——能还原"做了什么"，还原不了"用户在某瞬间说过哪句话有信息量"
- 这层补的就是**事件 → 结晶之间的原料**。AI 跨日无原生记忆，没这层 = 复盘凭印象 = 信息丢失

## 项目内文档分类

### 契约层

**AGENT.md**（项目层）：项目特定的工作方式 + 配置 + 钩子。AI 进入项目的第一份读物。

**spec/ 目录**（替代分散设计文档）：按模块拆分的权威定义——每个 spec 文件是单一功能/模块的契约，定义接口、行为、约束。

spec/ 与旧式"设计文档"的本质区别：
- **设计文档**解释"为什么这样设计"（给人看）
- **spec** 定义"对外的契约是什么"（给 AI 当施工图纸）

两者可以共存：spec 是代码权威来源，设计文档（`docs/architecture.md` 等）是设计意图的解释。

**模块层 README**：每个模块的路由器——职责概述 + 指向 spec 中该模块的契约 + 快速索引。不重复 spec 内容，只做指针。

### 工作记录层

**卡片体系**（backlog）——任务粒度的工作追踪：
- `_active.md` — 当前重点
- `_done.md` — 已完成（留档）
- `_frozen.md` — 冻结挂账
- 拆多文件的好处：单文件超一定规模后认知负荷剧增，按状态拆后挑卡只看 active

**CHANGELOG.md**——操作粒度的工作记录：
- 每次增删改操作的即时记录
- 替代了旧方式中"不允许删改，只能画删除线 + 新增内容"的膨胀式维护
- CHANGELOG 记录操作历史，文档本体保持精炼

**卡片体系 vs CHANGELOG 的关系**：
- **卡片体系**面向具体任务——"这张卡做完了吗？依赖什么？还有什么没做？"
- **CHANGELOG** 面向单次操作——"这次 commit 改了哪些文件？为什么改？"
- 两者不冲突，各管各层：卡片管"规划与追踪"，CHANGELOG 管"操作与审计"
- CHANGELOG 让文档可以"直接删改过时内容"而不担心信息丢失——删掉的内容在 CHANGELOG 里有记录

### 临时文档与迁移层

**wip 临时文档**：长议题在拍板前的讨论稿。可以矛盾、可以重写、不需要 SSOT（单一权威来源）。

**wip → spec 迁移工作流**：
1. wip 文档中讨论并拍板
2. 拍板后内容迁出到 spec/ 对应文件
3. wip 文件标"已迁出 → 见 spec/X.md"占位
4. 全部议题迁出后，wip 整本归档
5. CHANGELOG 记录迁移操作

**为什么独立**：避免"未拍板的讨论"污染 formal 定义。wip 是草稿，spec 是契约——两者不应混在同一文件。

### 维护层

**docs/maintenance/**：显式的维护流程文档：
- 修 bug 流程（bug_fix.md）
- 新增单位/卡牌流程
- 文档同步流程
- 验收流程

**CHANGELOG** 驱动维护：做完任何改动 → 写 CHANGELOG → 顺便检查是否需要更新对应的 maintenance 流程文档。

### 文档健康度层

每份文档标注健康状态：`stable` / `draft` / `outdated`
- **stable**：当前准确，可直接信任
- **draft**：正在写/改，内容可能不完整
- **outdated**：已过时，需要重读时对照 CHANGELOG

配合自动化 lint（断链检测、过期文档提醒），让文档健康度可被工程化检查而非靠人肉记忆。详见独立文章 `doc-health-engineering.md`。

## 类型间的依赖链

不是平铺，是有**依赖链**的：

```
全局 AGENT.md  (跨项目原则)
    ↓ 项目按自己情况实现
项目 AGENT.md  (项目工作方式 + 指针)
    ↓ 指向
spec/*.md  (权威契约定义)
    ↑ wip 拍板后迁出
_wip_*.md  (临时讨论稿)
    ↓ 全部迁出后
archive/  (历史归档)

backlog/_active.md  (当前任务)
    ↓ 完成时
backlog/_done.md  (已完成留档)
    ↓ 操作记录
CHANGELOG.md  (每次操作记录)

NOW.md / AI_INDEX.md / PRODUCER_PROFILE.md  (元层启动锚点)

reviews/_observations.md  (流式素材)
    ↓ 周度触发
reviews/review_*.md  (复盘记录)
    ↓ 产出结晶
PRODUCER_PROFILE.md  (长期画像)
```

## 边界判定（容易混淆的几对）

**backlog 卡 vs checklist**：
- 卡 = "要做某件事 + 决策点 + 进度"
- checklist = "做某类事时要扫的位"
- 一张卡可以引用多个 checklist；一个 checklist 服务多张卡
- **checklist 不是固定目录**：它是项目层的延伸概念——当项目层 AGENT 的细节超出预算时抽出来的"操作清单"。不强制命名为 `_checklists/`，不强制独立成文件（短的可以直接写在项目 AGENT 里）。它的本质是"可随实战不断长的扫描表"，不是框架的固定零件。属于 `element-decoupling-and-assembly.md` 中标记的"可选"元素——有信号才建

**wip 文档 vs spec**：
- wip = 拍板前的讨论稿，可以矛盾、可以重写
- spec = 拍板后的契约定义，AI 按字面理解执行
- 拍板的时机 = wip 内容**迁出到 spec** 的时机

**卡片体系 vs CHANGELOG**：
- 卡片管"规划层"——准备做什么、做到哪了、依赖什么
- CHANGELOG 管"操作层"——这次改了什么、为什么改
- 两者互补：卡片追踪"任务完成度"，CHANGELOG 追踪"操作可审计性"

**全局 AGENT vs meta-agent-collab**：
- 全局 AGENT = AI 协作规则（AI 必读，有约束力）
- meta-agent-collab = 给人看的理念文章（AI 默认不读，是观点不是契约）

**`_observations.md` vs `PROFILE.md`**：
- observations = 流式素材（"今天用户说过这句有意思"）· 高频追加 · 短条
- PROFILE = 长期画像结晶（"用户的稳定偏好是 X"）· 复盘触发 · 长文 + 版本号
- 关系：observations 是 PROFILE 的**原料**——多条 observations 在一次复盘中升级为 PROFILE 一条特征

## 为什么不是一份大 README

如果硬塞成一份大 README，会发生：

1. **更新节奏冲突**：协作偏好变化慢，任务卡片变化快，spec 最稳定——混在一起 commit history 完全不可读
2. **读者错配**：人类想看"项目是什么"，AI 想看"协作规则"，新加入者想看"任务清单"——一份文档无法同时服务
3. **认知负荷爆炸**：找一条规则要在几千行里搜
4. **AI context 浪费**：AI 每次会话读全文要消耗大量 token，但实际只用 10%
5. **版本控制噪声**：改一条规则触发整份大文件 diff

**分文件的代价**：要建立"该看哪份"的路由意识。但这个代价用 `AI_INDEX.md` + 项目层 AGENT 的钩子可以摊薄。

## 给"想搭框架的人"的建议

- **从 1 份开始，撑不住再拆**：别一开始就搭十几份文件。先写 README + AGENT，其他类型等"我现在确实需要一份独立的 X 来解决问题"才建
- **拆的触发条件**：① 单文件超约定行数预算；② 一份文件里出现两种**更新节奏**或两种**读者**；③ 同类信息在多处重复（说明该抽 SSOT）
- **每类文档的"为什么"要在自己的文件顶部写**：避免未来回看时不知道"这份是干嘛的"
- **指针 > 复述**：跨文件引用用指针（"详见 X"），不要复述
- **状态机化**：很多类型的文档其实是状态机的视图——backlog 卡有 active/frozen/deprecated/done 状态，wip 有讨论/拍板/迁出/归档状态。**用文件名 / 文件夹名表达状态**比"状态字段写在卡内"更直观
- **CHANGELOG 让文档可以被安全删改**：旧方式"只画删除线 + 新增"会导致文档臃肿。有了 CHANGELOG，可以直接删除过时内容——审计链在 CHANGELOG 里

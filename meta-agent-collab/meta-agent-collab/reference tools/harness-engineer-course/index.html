CLAUDE.md 架构工程

游戏引擎的配置文件系统 — 分层设计、按需加载、版本演进

100 行改变一切

一个 100 行的 CLAUDE.md 能让 Claude 从"什么都能做但什么都做不好"变成"精准执行团队规范的专业队友"。但超过 200 行，性能开始下降。

💡
核心洞察

CLAUDE.md 不是说明书，是系统架构 — 分层设计、权限隔离、按需加载、版本演进，和你设计游戏系统用的原则完全一样。这不是一门写作课，是一门工程课。

就像 Unity ProjectSettings 不会把所有配置塞进一个文件，CLAUDE.md 也有自己的分层解析体系。你写的每一行都在消耗 Claude 的"注意力预算"。

六层解析体系

CLAUDE.md 的优先级从高到低，六层叠加。就像游戏引擎配置从全局到局部逐层覆盖。

1
Managed Policy（组织级）

企业管理员强制下发的规则，优先级最高，不可覆盖

2
User Global（~/.claude/CLAUDE.md）

你个人的全局偏好，适用于你参与的所有项目

3
Project Root（./CLAUDE.md）

项目根目录的团队规范，所有成员共享，提交到 Git

4
Local（./CLAUDE.local.md）

个人本地配置，自动被 .gitignore 排除，不影响队友

5
Parent Dirs（向上查找）

从当前目录向上查找父目录的 CLAUDE.md，适合 monorepo

6
Subdirs（按需加载）

子目录 CLAUDE.md 只在访问该目录文件时才注入，节省 token

路径作用域规则

Path-Scoped Rules 比子目录 CLAUDE.md 更灵活 — 它们能跨目录生效，用 glob 模式精确匹配文件类型。

CODE 

# .claude/rules/shader-conventions.md
---
paths: ["**/*.shader", "**/*.hlsl", "**/*.cginc"]
---
Shader 编写规范:
- 所有 surface shader 必须包含 fallback
- 变量命名: _MainTex, _Color 格式
- 注释标注 pass 用途
PLAIN ENGLISH

这个文件放在 .claude/rules/ 目录下

--- 是 YAML frontmatter 的开始标记

只在 Claude 访问 .shader / .hlsl / .cginc 文件时生效

--- 元数据区域结束，下面是正文规则

以下是 shader 专用的编写规范

Surface shader 没有 fallback 会在低端设备上崩溃

统一的变量命名格式让团队代码风格一致

注释让其他开发者快速理解每个 pass 的用途

这种方式比把所有 shader 规则写进根目录 CLAUDE.md 更好 — 规则只在需要时加载，不浪费注意力预算，而且能跨目录覆盖所有匹配文件。

CLAUDE.md

Claude 读后"判断"是否适用。遵循率约 80%。适合编码风格、架构偏好、项目背景等建议性内容。

反模式清单

写 CLAUDE.md 时最常犯的错误。每一条都是真实项目踩过的坑。

❌

复制训练数据

Claude 已经知道 PEP 8 和标准 TypeScript 规范。重复写进 CLAUDE.md 等于浪费注意力预算在它已经懂的事情上。

❌

用 CLAUDE.md 做样式强制

"所有文件用 2 空格缩进" 这种规则应该用 Hooks + Linter 确定性执行，不是用 CLAUDE.md 建议性提醒。

❌

只增不减

规则只往里加、从不清理。三个月后 CLAUDE.md 膨胀到 500 行，Claude 开始随机忽略规则。定期审计是必须的。

❌

放密钥

CLAUDE.md 提交到 Git，所有人可见。永远不要在里面放 API key、密码或任何敏感信息。用环境变量。

❌

Monorepo 规则冲突

客户端 CLAUDE.md 写"用 camelCase"，服务端 CLAUDE.md 写"用 snake_case"，根目录 CLAUDE.md 没有说明上下文 — Claude 不知道该听谁的。

MCP 协议的三个核心原语

MCP 不是"连接工具"的简单 API — 它是让 Claude 在运行时发现新能力的协议。三个核心原语各司其职：

🔧
Tools（工具）

Claude 可以主动调用的操作。比如"创建 Unity 场景"、"查询玩家数据"、"触发构建"。这是最常用的原语 — Claude 看到工具描述后，自己决定什么时候调用。

Hooks 自动化与工作流引擎

Claude Code 的事件系统 — 把"建议"变成"法律"

一个 Hook 让 Claude 变成自动修复机器

想象这个场景：Claude 写完代码想收工 → 测试不通过 → Hook 阻止退出 → Claude 被迫继续修 → 直到所有测试通过才能真正完成。

💡
Advisory vs Deterministic

CLAUDE.md 是建议（Claude 判断是否遵循，约 80% 遵循率）。Hooks 是法律（Shell 命令，零例外执行）。成熟的 Harness 工程把试验性规则放 CLAUDE.md，验证后迁移到 Hooks。

你写过 Unity 的 OnCollisionEnter 或 Unreal 的 Event Dispatcher 吗？Hooks 就是 Claude Code 的事件系统。26 个事件钩子，4 种处理器类型 — SessionStart 就是 Awake()，PreToolUse 就是 OnTriggerEnter()，Stop 就是 OnDestroy()。

会话启动

标注 ⚠ 的事件支持阻断（exit 2）— 这是 Hooks 把"建议"变成"强制"的核心机制。

退出码协议：Hook 的三种结局

每个 Hook 执行完毕后，通过退出码告诉 Claude Code 下一步该做什么。三个数字，三种命运。

0
Exit 0 — 通过

成功执行，处理 JSON 输出。流程继续。就像游戏中通过了检查点。

2
Exit 2 — 阻断

仅在可阻断事件上生效（PreToolUse、Stop）。操作被中止，stderr 信息反馈给 Claude。这就是"强制"的机制。

?
其他退出码 — 警告

非阻断错误。显示 stderr 但流程继续。就像游戏中的非致命警告 — 记录日志但不中断。

⚠
Exit 2 只在可阻断事件上有效

在 PostToolUse 或 SessionStart 上返回 exit 2 不会阻断任何操作 — 它们不支持阻断。务必确认你的事件类型支持你想要的行为。

4 种处理器类型

事件触发后，由处理器决定"做什么"。4 种处理器，从轻量到重量，选对场景是关键。

⚙

command

Shell 命令（最常用）
直接执行 shell 脚本，通过 stdin 接收 JSON 输入。
游戏场景：跑测试、检查资产尺寸、运行 lint

🌐

http

HTTP POST 到外部服务

💭

单轮 Claude 评估
轻量 LLM 判断，无工具访问，纯文本推理。
游戏场景：判断代码改动是否涉及核心玩法逻辑

🤖

agent

子 Agent 评估（最强但最贵）
完整工具访问，能读文件、跑命令。用于复杂验证。
游戏场景：审查架构变更、验证跨模块依赖

模式一：Subagent 委派

Subagent 是最基础也最常用的多 Agent 模式。主 Agent 通过 Agent 工具调用子 Agent，每个子 Agent 拥有完全独立的上下文窗口。

1
独立上下文窗口

Subagent 的探索过程不会污染主 Agent 的上下文窗口。它探索了 50 个文件，但只返回一份精炼摘要给主 Agent。

2
自定义系统提示词

每个 Subagent 有专属的角色设定。"你是游戏工作室的高级代码审查员" 比 "帮我看看代码" 的输出质量高 3-5 倍。

3
工具访问限制

审查员只能 Read/Glob/Grep（只读），不能修改文件。这降低了事故风险 —— 它不可能误删代码。

4
模型路由降成本

简单任务用 Haiku（便宜），复杂推理用 Opus（贵但准）。写测试是结构化工作，不需要最贵的大脑。

💡
消息追踪

每条 Subagent 消息都包含 parent_tool_use_id 字段，你可以追踪每个结果来自哪个子 Agent。当 10 个 Agent 并行运行时，这就是你的调试生命线。

Fan-Out/Fan-In 并行模式

Pipeline 的杀手级用法：把任务列表并行分发给多个 Claude 实例，各自处理完后收集结果。下面是一个 Unity 引擎 API 迁移的真实脚本。

BASH 
# Fan-Out: 并行迁移 Unity 2021 → 2023
for file in $(cat migration-files.txt); do
  claude -p "将 $file 从 Unity 2021 API 迁移到 Unity 2023 API" \
    --allowedTools "Edit,Bash(git commit *)" &
done
wait  # Fan-In: 等待所有并行任务完成
逐行解读

注释：这是一个并行迁移脚本。

从文件列表中逐个读取需要迁移的文件路径。

为每个文件启动一个独立的 Claude 实例处理迁移。

& 是关键 —— 它让每个 Claude 实例在后台运行，不等前一个完成。所有实例同时工作。

循环结束。

wait 阻塞直到所有后台任务完成 —— 这就是 Fan-In（汇合点）。

1
生成任务列表

用脚本或 Claude 分析代码库，找出所有需要迁移的文件，输出到 migration-files.txt。

2
Fan-Out 并行分发

每个文件启动一个 Claude 实例，& 后台执行。50 个文件 = 50 个并行 Agent。

3
各自独立执行

每个 Agent 在自己的进程中工作，互不干扰。工具权限精确限制为 Edit + git commit。

4
Fan-In 收集合并

wait 等待所有任务完成，然后检查每个 commit 的结果。可以再跑一轮 Claude 做统一的代码审查。

Worktree 隔离：防止并行冲突

50 个 Agent 同时改同一个代码库？灾难。Worktree 隔离解决了这个问题。

🔒
isolation: "worktree"

在 Agent 定义中加一行配置，每个 Agent 自动获得独立的 git worktree 副本。

🛡
改动不影响主工作目录

Agent 的所有文件修改都在 worktree 内部。你的编辑器看到的代码不会突然变化。

🔄
显式合并

任务完成后，由你或 Lead Agent 决定是否合并。有问题就丢弃，安全无副作用。

⚡
/batch 命令默认隔离

/batch 命令做代码库级别的批量迁移时，自动使用 worktree，无需手动配置。

成本管理实战

AI 不贵，失控的 AI 才贵。掌握成本公式和路由策略，让每一分钱都花在刀刃上。

公式 

// \u591a Agent \u6210\u672c\u516c\u5f0f
cost = agents
     * avg_tokens_per_call
     * calls_per_day
     * price_per_million_tokens

// \u793a\u4f8b\uff1a5 \u4e2a Agent, \u5e73\u5747 8K token,
// \u6bcf\u5929 50 \u6b21\u8c03\u7528, Sonnet $3/M
cost = 5 * 8000 * 50 * (3/1000000)
     = $6/day = $180/month
解读

多 Agent 系统的月度成本取决于四个变量的乘积

Agent 数量 — 角色越多，总调用越多

每次调用的平均 token 数 — 上下文越长越贵

每天的调用频率 — 自动化越多，调用越频繁

模型单价 — 不同模型价差可达 60 倍

 

以中等规模团队为例：5 个专业 Agent，每个平均消耗 8K token，每天被调用 50 次，使用 Sonnet 模型（$3/百万 token）

结果：每天 $6，每月 $180 — 远比一个初级开发者便宜

模型路由策略

不是所有任务都需要最强模型。像交通分流一样，把任务导向合适的"车道"。

三大省钱武器

📦
MCP 查询缓存

相同的数据库查询结果缓存 5 分钟，避免重复调用外部系统

📋
Prompt Caching

系统 prompt 和 CLAUDE.md 内容被缓存，后续请求成本降低约 90%

子 Agent 只返回结论摘要，不把完整上下文传回主 Agent，大幅减少 token 消耗

代码 ↔ 白话翻译：基本交互

来看一下在终端里和Claude Code对话是什么样的。

CODE 

claude -p "创建一个React计数器组件"
白话翻译

调用Claude Code，用-p参数直接下达指令："帮我创建一个React计数器组件"。一句话搞定，不用进入对话模式。

CODE 

claude -p "重构登录模块" \
  --output-format json
白话翻译

让Claude重构登录模块，

并且把结果以JSON格式输出 -- 方便其他程序接收和处理结果。

CODE 

claude -p "添加单元测试" \
  --allowedTools Read,Write,Bash
白话翻译

让Claude添加单元测试，

但限制它只能使用Read（读文件）、Write（写文件）、Bash（运行命令）这三个工具。相当于告诉实习生："你只能用这三样东西，别的别碰。"

五个救命命令

这五个斜杠命令是你每天都会用到的"生存工具包"。先记住它们，后面的进阶模块会深入讲解。

1

/init

让Claude自动扫描你的项目，生成一份CLAUDE.md配置文件。拿到新项目时第一件事就运行它。

2

/compact

压缩当前对话的上下文，保留关键信息、释放空间。长对话中上下文达到60%时就该用它。

3

/clear

彻底清空当前会话，从零开始。开始新任务时用它，避免旧对话干扰新任务。

4

/rewind

回退到上一个检查点。Claude每次改代码前会自动存档，出了问题用/rewind"读档"回去。

5

/cost

查看当前会话消耗了多少Token。帮你随时掌握"花了多少钱"，避免用量焦虑。

100行的秘密武器

Boris Cherny（Claude Code 创造者）的 CLAUDE.md 只有100行，但他每周能产出100个 PR。秘密不在于写更多代码，而在于教会 Claude "怎么帮你工作"。

💡
Boris 的100行哲学

CLAUDE.md 不是越长越好。把它想象成给新同事的入职指南 — 只写最关键的规则，让 Claude 快速理解"这个项目怎么做事"。100行足以覆盖技术栈、编码风格、测试要求和常见陷阱。

如果说 Module 2 是"学会开车"，这个模块就是"改装你的车" — CLAUDE.md 是你的调校手册，Hooks 是自动化传感器，MCP 是连接外部设备的接口，多 Agent 是组建车队。

CLAUDE.md：给Claude的工作手册

一份好的 CLAUDE.md 遵循 WHAT-WHY-HOW 结构。把它想象成给新员工的三句话交底。

1

WHAT
技术栈是什么
TypeScript + React + PostgreSQL

→
2

电商平台的后台管理系统

→
3

HOW
怎么工作
所有API用REST，测试覆盖率 > 80%

黄金法则：每次 Claude 犯错，立即把纠正规则加入 CLAUDE.md。这样同一个错误永远不会出现第二次。

⚠
遵循率 ≠ 100%

CLAUDE.md 的遵循率约为 80% — Claude 大多数时候会遵守，但偶尔会"忘记"。如果某条规则必须100%执行（比如"绝对不能删除生产数据库"），请用 Hooks 而不是 CLAUDE.md。

三层 CLAUDE.md 体系

CLAUDE.md 不只能放在项目根目录。它是一个三层体系，从最宽泛到最具体 — 就像公司的规章制度从总部到部门到岗位逐级细化。

🌎

全局层

~/.claude/CLAUDE.md
你的个人偏好，适用于所有项目。比如"永远用中文回复"、"commit message 用英文"。

📁

项目层

./CLAUDE.md
项目级规范。技术栈、架构决策、代码风格。团队共享。

📄

模块层

./src/CLAUDE.md
子目录特有规则。比如"这个文件夹只放API路由"、"组件命名用PascalCase"。

Claude 会从全局到模块依次读取所有层级，后者覆盖前者。越具体的规则优先级越高。

思考模式：给 Claude 的"思考预算"

Claude 在回答前会"思考"。你可以通过特定触发词控制思考深度 — 就像给引擎选择不同的档位。低档省油（省 Token），高档有力（更精确）。

1

think

基础思考。适合简单任务：改个变量名、写个注释。相当于城市道路的1档。

2

think hard

中度思考。适合中等复杂任务：重构函数、修复普通bug。相当于高速公路的3档。

3

think harder

深度思考。适合复杂问题：跨文件重构、性能优化。相当于赛道模式。

4

ultrathink

最高思考预算。适合架构设计、复杂算法。相当于全功率输出。慎用 — Token消耗巨大。

Hooks：100% 执行率的自动化

CLAUDE.md 是"建议"（80%遵循），Hooks 是"法律"（100%执行）。Hooks 就像汽车的安全系统 — 不管驾驶员怎么操作，安全气囊该弹就弹，Linter 该跑就跑。

🛑

PreToolUse

工具执行前拦截。比如在 Claude 编辑文件前，先检查是否触碰了受保护文件。

✅

PostToolUse

工具执行后验证。比如每次编辑代码后自动运行 Linter，确保代码质量。

🔔

Notification

任务完成时通知。比如 Claude 完成编译后发送桌面通知，你不用一直盯着屏幕。

GitHub

🎨

Playwright

控制浏览器做自动化测试。Claude 可以打开网页、点击按钮、填写表单、截图验证。

多 Agent 工作流：组建你的车队

一个 Claude 是员工，多个 Claude 就是团队。多 Agent 工作流让你像项目经理一样调度多个 Claude 实例并行协作。

👥

Subagent

主 Agent 派出的"侦察兵"。拥有独立上下文，执行完子任务后向主 Agent 汇报。主会话保持干净。

👥

Agent Teams

多个 Claude 实例同时运行，各自处理不同任务。一个写前端，一个写后端，一个写测试。

🌳

Worktree

用 --worktree 标志让每个 Agent 在隔离的代码副本中工作，互不干扰。就像每人一条独立车道。

💡
Boris 的车队战术

Boris 同时运行 10-15 个 Claude 会话，每个处理一个独立任务。这就是他每周产出100个 PR 的秘密 — 不是一个人写得更快，而是同时有15个"助手"在并行工作。

上下文管理：保持引擎高效

Claude 的上下文窗口是有限的。就像汽车油箱 — 跑远了要加油，装多了要清理。掌握这四个命令，保持 Claude 始终在最佳状态。

📦

/compact

上下文用了约60%时执行。压缩历史对话，保留关键信息，释放空间。相当于整理行李箱。

🔄

/clear

开始新任务时执行。彻底清空当前会话，重新开始。相当于换一辆新车。

💰

/cost

随时监控当前会话的 Token 消耗和费用。知道你的"油表"指向哪里。

📚

Skills

用 Skills 替代过长的 CLAUDE.md。按需加载能力，而不是一次性塞入全部规则。

读懂节点：ComfyUI 工作流 JSON 翻译

ComfyUI 的工作流本质上是一个 JSON 文件 —— 每个节点是一个功能单元，节点之间通过连线传递数据。看懂这段 JSON，你就理解了 ComfyUI 的核心概念。

COMFYUI JSON 

{
  "3": {
    "class_type": "KSampler",
    "inputs": {
      "seed": 42,
      "steps": 20,
      "cfg": 7,
      "sampler_name": "euler"
    }
  }
}
用人话说

这段定义了工作流中的 3 号节点

它的类型是 KSampler —— 负责实际的图像生成（采样）过程

下面是它的输入参数：

seed: 42 —— 随机种子，固定为 42 可以保证每次生成同样的图。换个数字就会得到完全不同的画面

steps: 20 —— 生成要跑 20 步。步数越多细节越丰富，但耗时也越长。20 步是常用的平衡值

cfg: 7 —— 引导强度设为 7，意味着 AI 会比较忠实地遵循你的提示词

sampler_name: "euler" —— 使用 Euler 采样算法。不同算法生成速度和质感略有差异，Euler 是最基础稳定的选择

硬件门槛：你需要什么显卡？

本地跑 AI 绘图对显卡有要求，核心指标是 VRAM（显存）大小。显存不够？用云端方案。

入门级 RTX 4060 Ti — 16GB VRAM。能跑 SDXL 和基础 ComfyUI 工作流，够用于学习和小项目
专业级 RTX 4090 — 24GB VRAM。跑 FLUX、复杂 ControlNet 组合、LoRA 训练都游刃有余
零硬件 RunningHub / Colab — 云端 GPU，按用量付费。适合没有独立显卡或想先体验再决定是否购买的用户

从手工造车到数控机床 -- 当AI渗透游戏开发的每一个环节

手工造车 vs. 数控机床

腾讯的VISVISE工具把角色蒙皮从3.5天缩短到几分钟，已经用在了90多款游戏上。三七互娱每月用AI生成28万张原画。这些数字背后，是一整条AI驱动的游戏生产管线。

传统游戏开发像手工造车 -- 每个零件都得一个个加工。AI加持的游戏开发像引入了数控机床 -- 重复性工序交给机器，工匠专注在需要创意判断的环节。最终产品的品质取决于工匠的眼光，但产量提升了好几倍。

💡
关键洞察

AI在游戏生产中的渗透是"按环节"的 -- 概念设计几乎全面AI化，3D生成仍是瓶颈，代码辅助采纳率最高，而创意决策仍然是人类的领地。

不可忽视的法律风险

AI工具提升了产能，但也带来了新的法律灰区。在把AI融入生产管线之前，必须了解这些风险。

⚠
版权风险警告

在多数司法管辖区，纯AI生成的作品不受版权保护。美国版权局已明确表示，缺乏人类创作者的作品不具备版权资格。这意味着：如果你的游戏资产完全由AI生成，竞争对手可以合法地复制它们。

💡
实操建议

在每个AI辅助环节保留人类创意决策的记录：概念阶段的选择判断、3D阶段的手动修改、动画阶段的关键帧调整。只要能证明人类在创作过程中做出了实质性的创意贡献，作品就有更强的版权主张基础。AI是工具，人是创作者 -- 这条线一定要守住。

Claude Code 每日工作流

把这6步变成肌肉记忆，你的Claude Code使用效率会翻倍。每一步都有它存在的理由——跳过任何一步，迟早会踩坑。

1
/init — 生成 CLAUDE.md

新项目第一件事：让Claude Code扫描代码库，自动生成项目说明文件。这是你和AI之间的“项目手册”，后续所有对话都基于它理解上下文。

2
犯错即记录 — 更新 CLAUDE.md

Claude犯了错？别只是修复——把教训写进CLAUDE.md。下次它就不会再犯同样的错。这是你训练专属AI助手的过程。

3
复杂任务先说"规划，不要写代码"

面对大任务，先让Claude出方案而不是直接动手。加上 ultrathink 关键词触发深度思考。先看路线图，再动工。

4
60% 即 compact

上下文使用量到60%就该压缩了。等到80%再compact，Claude已经开始“犯迷糊”了。像定期清理桌面一样，保持工作台整洁。

5
大任务拆小会话 + 频繁 git commit

一个会话只做一件事。完成一步就commit一次。这样即使AI出错，你也能快速回滚到上一个正确状态。

6

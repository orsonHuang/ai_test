# 00 · F2S 指令手册

> **本文是什么**：FromDemoToSpec（F2S）工作流可用的项目级指令清单 + 触发词识别规则。
> **本文不是什么**：全局框架指示触发词手册（本文档仅列 F2S 工作流内部的指令）。本文档只列 F2S 工作流内部的指令。
> **使用时机**：用户进入 F2S 工作流相关对话时，AI 优先扫描用户消息是否命中本文指令；命中则按对应执行路径走，不等待进一步确认。
> **行数预算**：100 行内。

---

## 一、设计意图

把"工作流是怎么跑的"从一段需要用户阅读的说明，变成几个**用户一句话即可触发**的指令。降低使用门槛，让用户不用先读完 README 才知道该说什么。

未来工作流扩展（如"开始 spec 提取"、"开始 demo 评审"）也通过新增指令对外暴露，保持入口一致。

---

## 二、当前可用指令

### 指令 1 · 开始原型验证

**规范名**：`开始原型验证`

**触发动作**：进入 `01-input-elicitation-protocol.md` 定义的问答补齐流程（开放钩子 → 类型识别 → gap 追问 → 跳过项处理 → 生成 Input 文档 + HTML 原型 + MVP 设计文档 + 生成 README.md（按 `Demo/README.md` §README 模板）+ 初始化 CHANGELOG.md（按 `Demo/README.md` §CHANGELOG 模板）+ 保存完整对话记录为 `对话记录.txt`）。

**典型触发词例子**（不限于以下，AI 应识别同义表达）：

| 类别 | 用户可能说 |
|------|-----------|
| 直说 | 开始原型验证 / 我想开始原型验证 / 启动原型验证 |
| 缩写 | 原型验证 / 验证原型 / 开始验证 |
| 自然表达 | 我想做个原型 / 我有个想法想验证一下 / 帮我搞个原型 / 我想试试这个玩法 |
| 含上下文 | 我有个塔防想法，想验证下数值 / 我想做个三消，但不知道从哪开始 |

**识别原则**：

1. **强信号**（直接进入流程）：消息中包含「原型验证」「验证原型」字样，或包含「开始 / 启动 / 想做 / 想试」+「原型 / 玩法 / 验证」组合词。
2. **中信号**（先确认再进入）：用户描述了一个游戏/产品想法但没明说要验证（如"我想做个塔防"），AI 先回一句"听起来你想验证这个玩法，要不要走一下原型验证流程？"再决定。
3. **弱信号**（不主动触发）：泛泛聊天提到"原型"二字（如"原型阶段一般要注意什么"），不算触发，正常回答。

**触发后 AI 的第一动作**：

- 不再赘述工作流是什么（用户既然触发了就是知道）
- 直接读 `01-input-elicitation-protocol.md` 全文
- 进入「阶段 1 · 开放钩子」，把标准 5 题列给用户

### 指令 2 · 修改Demo

**规范名**：`修改Demo`

**触发动作**：对 `Demo/` 目录内现有原型文件（HTML / MVP 文档 / Input 文档 / Spec 文档 / CHANGELOG）进行修改。**强制前置步骤**：操作前必须先读 `Demo/README.md`（了解命名规范、版本迭代规则、验收清单迭代规则），再定位目标文件。

**典型触发词例子**：

| 类别 | 用户可能说 |
|------|-----------|
| 直说 | 修改Demo / 改一下Demo |
| 自然表达 | 修改XX（原型名）、把XX改成YY、XX需要调整一下、XX的验收项加一条 |
| 含文件名 | 改一下combat-demo的MVP文档 / 挂机装备RPG的html有问题 / 更新一下即时三消RPG的input |
| 版本迭代 | 出个v2 / 升级到v3 / 迭代一下XX |

**识别原则**：

1. **强信号**（直接进入流程）：消息中包含「修改 / 改 / 调整 / 更新 / 迭代」+ Demo 文件名/原型名，或明确指向 `Demo/` 目录下的文件。
2. **中信号**（先确认再进入）：用户描述了 Demo 的问题但没说"改"（如"XX的验收项好像有个重复的"），AI 先确认"要帮你处理这个重复项吗？"再动手。
3. **弱信号**（不主动触发）：泛泛讨论原型好坏（如"这个原型做得怎么样"），不算触发，正常评价。

**触发后 AI 的第一动作**：

1. **必须先读** `Demo/README.md` 全文（命名规范 + 版本迭代规则 + 验收清单迭代规则 + CHANGELOG 模板 + MVP 结构 + 注意事项）
2. 定位用户指定的目标文件
3. 执行修改（遵循 `Demo/README.md` 中的规则：版本号递增、CHANGELOG 追加、验收清单保留/删除/新增判据等）
4. 修改完成后按 `Demo/README.md` §版本迭代规则更新 CHANGELOG

> **约束**：AI 不得在未读 `Demo/README.md` 的情况下直接修改 Demo 文件。违反此条视为静默跳过流程规范。

### 指令 3 · 开始 Spec 提取

**规范名**：`开始 Spec 提取`

**触发动作**：从已验证的 HTML 原型 + MVP 文档中自动提取 Spec 文档，并同步预填策划文档框架。具体流程：
1. 读 `03-prototype-to-spec-mapping.md` 全文
2. 定位 `Demo/` 目录下目标原型的 HTML 文件 + MVP 文档
3. 扫描原型代码，按映射配置表提取信息 → 填入 `spec-templates/` 对应模板 → 生成 Spec 文档（含 technical spec）
4. 同步预填 `03-design-doc-template-*.md` 对应模板：自动填写参数表、已知问题、结构信息等可从代码/MVP 提取的段；标注"待策划填写"段（设计意图、验证结论、体验目标等）
5. 输出产物清单：Spec 文档 + 策划文档（预填版）+ 提取摘要

**前置条件**：目标原型已完成至少一轮试玩验证（MVP 文档存在 + 原型代码已稳定）。未验证的原型应先走「修改Demo」迭代至稳定版本。

**典型触发词例子**：

| 类别 | 用户可能说 |
|------|-----------|
| 直说 | 开始 Spec 提取 / 开始 spec 提取 / 启动 Spec 提取 |
| 缩写 | Spec 提取 / 提取 Spec / 生成 Spec / 出 Spec |
| 自然表达 | 帮我把原型沉淀成文档 / 提取一下设计文档 / 出一下策划案 / 帮我生成策划文档 |
| 含上下文 | 这个原型验证完了，帮我出 Spec / 试玩差不多了，沉淀一下吧 / XX原型可以出文档了 |

**识别原则**：

1. **强信号**（直接进入流程）：消息中包含「Spec 提取 / 提取 Spec / 生成 Spec / 出 Spec / 沉淀」字样，或「Spec」+「提取/生成/出/沉淀」组合词。
2. **中信号**（先确认再进入）：用户说"验证完了"/"试玩差不多了"但没明说要出文档，AI 先确认"原型验证已通过，要不要开始 Spec 提取？"再动手。
3. **弱信号**（不主动触发）：泛泛讨论 Spec 文档格式（如"Spec 模板里有什么"），不算触发，正常回答。

**触发后 AI 的第一动作**：

1. 确认目标原型（若对话上下文已有 → 直接用；若未指定 → 列出 `Demo/` 下可用原型让用户选）
2. 确认前置条件满足（MVP 文档存在 + 原型代码已稳定）：不满足 → 提示先走「修改Demo」迭代
3. 读 `03-prototype-to-spec-mapping.md` 全文
4. 扫描原型代码 → 类型识别 → 选对应 spec-template + design-doc-template
5. 执行提取 + 预填（参考 03 §自动预填规则）

### 指令 4 · 准备交付

**规范名**：`准备交付`

**触发动作**：对已完成 Spec 提取的原型文件夹执行交付收口。分两阶段：

**阶段 A · 收口前置检查**（不通过则卡住，修正后重新检查）：
1. **五源冲突扫描**：交叉比对 input.md / mvp.md / spec.md / spec-technical.md / HTML 代码中的关键机制（技能数、AI 规则、数值参数、状态机流程），列出不一致项 → 用户逐一拍板 → AI 修正文件
2. **人工填写段补齐**：检查 spec.md §八「验证结论」、spec-technical.md §十「转开发建议」是否为空 → 为空则提醒用户填写
3. **状态标签升级**：mvp.md 顶部「待试用验证」→「已验收」；spec.md / spec-technical.md「未复核」→「已复核」
4. **实现范围确认**：向用户确认目标仓库/模块、客户端or服务端、不做项清单（非 AI 可推断，必须人工拍板）

**阶段 B · 收口动作**（全部检查通过后执行）：
1. **归档过期文件**：将以下文件移入原型文件夹的 `归档/` 子目录：
   - **全部** `_mvp.md` 文件（所有版本，不保留最新版——其设计意图由 design-doc.md 替代，其验收功能完成即结束）
   - **全部** `_mvp.html` 文件除当前最新版（保留最新版一份供下游试玩）
   - `_input.md`（原始需求记录，已完成使命）
   - `对话记录.txt`（沟通过程记录，已完成使命）
   - 旧版 `_spec.md` / `_spec-technical.md` / `_design-doc.md` / `_design-doc-project.md`（若存在多版）
2. **生成 `交付说明.md`**：按 `Demo/README.md` §交付说明模板填写
3. **更新 `README.md`**：开发阶段更新为「已交付下游」；文件导航增加交付说明行 + 归档目录；移除已归档文件的导航行
4. **追加 `CHANGELOG.md`**：记录交付收口操作

**前置条件**：目标原型已完成「开始 Spec 提取」（spec + design-doc 系列文件存在）。

**典型触发词例子**（不限于以下）：

| 类别 | 用户可能说 |
|------|-----------|
| 直说 | 准备交付 / 开始交付收口 / 交付收口 |
| 缩写 | 交付XX / XX可以交付了 / 交给开发 |
| 自然表达 | 这个原型可以交下游了 / 帮我把这个原型整理一下交出去 / 收口一下XX |
| 含上下文 | 策略回合制RPG验证完了，可以交出去了 / 把XX原型归档一下准备给程序 |

**识别原则**：

1. **强信号**（直接进入流程）：消息中包含「交付 / 交出去 / 交下游」+ 原型名，或「准备交付 / 交付收口」字样。
2. **中信号**（先确认再进入）：用户说"验证完了"/"差不多了"/"可以了"但没明说要交付，AI 先确认"原型验证已完成，要执行交付收口准备交下游吗？"再动手。
3. **弱信号**（不主动触发）：泛泛讨论交付流程（如"交付时需要注意什么"），不算触发，正常回答。

**触发后 AI 的第一动作**：

1. 确认目标原型（若对话上下文已有 → 直接用；若未指定 → 列出 `Demo/` 下可用原型让用户选）
2. 确认前置条件满足（spec + design-doc 系列文件存在）：不满足 → 提示先走「开始 Spec 提取」
3. 进入阶段 A · 收口前置检查第 1 步（五源冲突扫描）

## 三、扩展约定（未来加新指令时遵守）

新增指令时，按以下结构追加到本文「二、当前可用指令」段：

1. **规范名**：动词开头的短语（如「开始原型验证」「开始 spec 提取」），保持平行结构
2. **触发动作**：明确指向哪个协议文档 / 哪个流程
3. **典型触发词例子**：至少覆盖「直说 / 缩写 / 自然表达 / 含上下文」4 类
4. **识别原则**：强 / 中 / 弱信号的区分（避免误触发）
5. **触发后 AI 的第一动作**：明确进入流程后的第一步是什么

新增指令同时需更新：
- `README.md` 顶部导语段的「可用指令」列表
- `00-ai-collaboration-principles.md` §3 协作姿态中"指令优先于自由对话"条目（如清单变化）

## 四、指令优先级

- 用户消息**同时命中多个指令** → AI 列出所有命中项让用户选，不擅自决定
- 用户消息**命中指令但当前对话已在另一流程中** → 先确认是否切换流程，不静默打断
- 用户消息**命中本文指令但与其他元层指令冲突** → 按上下文判断（在 F2S 项目目录下优先 F2S 指令）；判断不了则问用户

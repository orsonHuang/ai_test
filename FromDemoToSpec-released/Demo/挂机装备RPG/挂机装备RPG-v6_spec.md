## 四、经济规则

### 4.1 Tick 机制

> 来源：原型代码 restartGenTimer() / restartOpenTimer() / setInterval

- 宝箱产出 Tick：1500/(1+tier×0.08) ms
- 开箱 Tick：100ms 子帧，每 2500/(1+tier×0.08) ms 自动开箱
- UI 刷新 Tick：200ms
- 战斗 Tick：300/(100+攻速%) ms（夹持 100-600ms）

每 tick 执行顺序：
```
1. 宝箱产出检查（cs < 上限 → cs++）
2. 自动开箱进度推进（oProg += 100，达到阈值 → openChest）
3. UI 刷新 rChest()
```

### 4.2 溢出与上限规则

- 宝箱定时产出：达到 `200+tier×50` 上限后停止增长
- 关卡首通奖励宝箱：**可超过上限**（addChests 无上限检查）
- 金币/宝石：无上限
- 命中率：上限 200%（反制敌闪避后有效命中 clamp 5-95%）
- 闪避率：上限 20%
- 其他百分比属性：暴击 100%、吸血 50%、伤加 200%、伤减 75%、穿甲 100%、攻速 200%、反击 50%

### 4.3 死锁检测

- 死锁触发条件：所有已解锁槽位均有装备且战力 ≤ 掉落装备 → 自动卖出获得金币 → 可升级装备提升战力
- 处理方式：无严格死锁；金币积累→装备升级→战力提升→可挑战关卡获得宝箱奖励

## 七、战斗系统（补充）

> 本原型包含即时制自动战斗模块，补充战斗专属信息。

### 7.1 伤害公式

```
玩家伤害 = max(ATK × PLAYER_DMG_MULT × (1+dmg%) - 敌DEF × (1-pen%), 1)
暴击伤害 = 玩家伤害 × (cd/100)
敌方伤害 = max(敌ATK × ENEMY_MULT - 玩家DEF, 1) × (1-dr%)
有效命中率 = clamp(玩家命中 - 敌闪避, 5, 95)%
```

> 来源：原型代码 tick() 函数

### 7.2 关卡生成公式（100关 sawtooth波）

```
baseRec = 250 × 1.065^关
修饰：10结尾BOSS ×2.0 | 5结尾精英 ×1.5 | >10关的1尾喘息 ×0.65
敌HP = rec × 0.55  （v5.1 战斗系数：从 0.7 调降）
敌ATK = rec × 0.07  （v5.1 战斗系数：从 0.045 调升）
敌DEF = rec × 0.012 （v5.1 战斗系数：从 0.004 调升）
敌闪避 = min(20, 关×0.2)%（v5.1 战斗系数：上限 30→20，增长率 0.3→0.2）
玩家初始 HP = 200（v5.1 战斗系数：从 500 调降）
首通宝箱 = 8 + floor(i/3)（i 从 0 起，关 = i+1；精英×1.5，BOSS×2；v5.1 调降）
```

> 来源：原型代码 genStages() 函数

### 7.3 回合结构（实时制）

- 战斗全程自动，无人工干预
- 每 tick(300/(100+攻速%)ms) 双方同时攻击
- 玩家攻击流程：命中判定 → 伤害计算（暴击判定）→ 吸血回溯
- 敌方攻击流程：玩家闪避判定 → 闪避成功跳过/反击判定 → 实际伤害
- 胜负判定：任一方 HP ≤ 0 结束

## 八.五、边界条件与容错

> 来源：原型代码 `cl()` / `Math.max` / 各处上下限检查（详细实现见 spec-technical §七）

| 边界场景 | 处理规则 | 说明 |
|---------|---------|------|
| 宝箱数量达上限 | 定时产出停止，开箱消耗后恢复产出 | `200 + tier×50`，关卡奖励宝箱**不受此限制** |
| 装备掉落无可用槽 | 日志提示并跳过（不崩溃） | 初始全锁阶段不触发掉落 |
| 金币/宝石不足 | 升级按钮置灰，操作 return | 前端强制阻止，逻辑层二次校验 |
| 宝箱已至顶级（at 9） | 升级按钮隐藏/禁用，return | 无更高品质可升 |
| 伤害最小值 | `max(伤害, 1)`，保证 ≥1 点 | 防御极高时不出现 0 伤 |
| 战斗 tick 速率 | `clamp(间隔, 100, 600ms)` | 攻速溢出时防止 tick 间隔过小 |
| 属性数值上限 | `cl(v, lo, hi)` 工具函数 | 各属性上限见 spec §2.1 全局参数表 |
| 多次 victory/defeat 节点残留 | 战斗开始/阶段切换时清除旧节点 | 防止重复胜负飘字堆叠 |
| 自动替换 DOM 异常 | try-catch 静默处理 | 不阻断主流程 |

## 已知数值难题（v6 沉淀，留给数值策划专项处理）

### 问题描述

战力数字与实战表现不完全一致——v6 试玩多次出现以下场景：

- 玩家战力 1 万，打过推荐 2.5 万的关卡（高估了关卡难度，玩家以为打不过）
- 玩家战力 4820，打不过推荐 1175 的 20 关 BOSS（高估了玩家实力）
- 单件装备战力 45，全身总战力 29（量级错位，部分小于整体）

### 已尝试的方案与结论

| 版本 | 方案 | 结论 |
|------|------|------|
| v5.1+ | 战斗系数调整（HP 200/敌HP×0.55/敌ATK×0.07/敌DEF×0.012）| 缓解了"免疫挨打"和"突然卡关"，但战力↔实战仍不一致 |
| v6-c3b | hp 词条权重 0.5→0.3 | 治标不治本——加权和公式本质问题在于无法反映"暴击/吸血/攻速"的复利效果 |
| v6-c4 | calcPower / totalPow 改几何平均 `(atk×def×hp/10)^(1/3) × (1+Σ pct)` | 与 pStats() 战斗实际属性两套数据流脱节，战力数字与实战完全不挂钩 |
| v6-c4d | totalPow 改用 pStats 实战属性 `sqrt(atkEff × survEff) × utilMult` | 量级与单件 calcPower 错位（总战力 29 < 单件 45）；STAGE_BASE_REC 同步从 250→30 后又出"2 件装备打通 10 关" |
| v6-c4e | totalPow ×10 + STAGE_BASE_REC 30→100 量级校准 | "战力 4820 打不过推荐 1175 BOSS" 仍存在——多套校准均未根除问题 |
| v6-c4f（最终决策）| **回退到 c3c 的加权和公式** | 承认"战力↔实战一致"是数值精调的硬骨头，原型阶段无法靠手动调参解决 |

### 根本原因（c4 系列试错后归纳）

1. **战力公式只有一组数据流，实战公式有多组复利**：战斗实际由 atk × 暴击期望 × 命中率 × 攻速 × 吸血 × 防御折算等多维复利构成，单一加权和或几何平均都无法精确反映。
2. **关卡推荐战力 (`rec`) 与玩家战力是"两个公式独立估算"**：`rec = 250 × 1.065^i × mod` 是数学模型，玩家战力是装备词条加权和——两者用不同维度，对比意义有限。
3. **战力数字没有"分阶段校准"**：v5 加权和的权重表（atk×2 / hp×0.5 / pen×8 等）是经验拍脑袋值，没有用大规模战斗模拟验证拟合曲线。

### 留给后续数值策划的工作

1. **建批量战斗模拟器**：参数化跑 1000+ 场战斗（不同装备组合 vs 不同关卡），统计通关率，做战力↔通关率回归。
2. **用回归结果反推"权重表"**：让加权和公式逼近"实战预期 DPS × 实战预期生存"的简化估算。
3. **关卡推荐战力 `rec` 公式重新校准**：基于上一步的拟合曲线，让"推荐战力 == 50% 通关率门槛"成立。
4. **保留 MVP 阶段的临时缓解**：左侧热调面板的"敌人属性倍率"和"玩家伤害倍率"滑块作为玩家自校准入口。

> **关键约束**：原型阶段的多次手动精调（c3b/c4/c4b/c4c/c4d/c4e）证明，没有量化数据支持的纯手动调参在战力↔实战一致性上是低效的。立项后必须先建模拟器再调参，不要重蹈原型阶段的覆辙。

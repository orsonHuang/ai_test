# Demo 输出目录

> FromDemoToSpec 所有原型产出的统一存放位置。

---

## 命名规范

> **权威源**：完整命名规则见根目录 `00-file-naming-convention.md`，本段为目录侧速查。

每个原型一个独立子文件夹，**文件夹不含版本号**：

```
Demo/{原型名}/
```

示例：
```
Demo/combat-demo/
Demo/即时三消RPG/
Demo/economy-test/
```

版本信息体现在**文件名**中，不在文件夹名中。

### ★ 模板副本必须立即改名

从 `04-*-template-*.md` 拷贝出来填写时，**第一步必须立即改名**为 `{原型名}-vN_*.md`：

| 模板副本 | 改名为 |
|---------|--------|
| `03-design-doc-template-game.md` 副本 | `{原型名}-v{N}_design-doc.md` |
| `03-design-doc-template-product.md` 副本 | `{原型名}-v{N}_design-doc.md` |
| `03-design-doc-template-project.md` 副本 | `{原型名}-v{N}_design-doc-project.md` |

**不允许**：保留 `04-` 前缀的填好副本与正式产物混在同一目录——这是发布版二次复核发现的反模式（无框架版正是因此导致归档时无法判断"哪个是空模板/哪个是某个原型的产出"）。

## 文件夹内容

每个原型文件夹包含：

| 文件 | 命名规则 | 说明 |
|------|----------|------|
| HTML 原型 | `{原型名}-v{版本号}_mvp.html` | 单文件 HTML（零外部依赖），浏览器直接打开即玩 |
| MVP 设计文档 | `{原型名}-v{版本号}_mvp.md` | 功能概述 + 核心设计 + 配置项清单 + 验收清单 |
| Input 文档 | `{原型名}_input.md` | 基于 02 模板 + 00 问答补齐流程的填写结果，含跳过项清单与处理方式。仅生成一次（无版本号），后续迭代不改 |
| Spec 文档 | `{原型名}-v{版本号}_spec.md` | AI 按 03 映射规范从原型代码提取的结构化 Spec（生成时机：试玩验证通过后） |
| 目录说明 | `README.md` | 原型项目简介 + 文件导航（创建新原型时即生成；模板见下文） |
| 版本变更记录 | `CHANGELOG.md` | 版本迭代的结构化操作日志（创建新原型时即初始化，非迭代后才补建；格式见下文） |

## 版本迭代规则

同一原型名迭代时（如 `combat-demo` v1 → v2）：

1. **文件夹不变**：始终使用同一个 `{原型名}/` 文件夹，**不创建新的版本号文件夹**
2. **HTML 原型**：复制旧版 `.html`，改版本号命名（如 `combat-demo-v1_mvp.html` → `combat-demo-v2_mvp.html`），在新文件上修改
3. **MVP 文档**：直接修改，改标题版本号（如 `v1.0` → `v2.0`），具体变更记录在 `CHANGELOG.md` 中
4. **CHANGELOG.md**：追加本次迭代的操作记录

### 验收清单迭代规则

MVP 文档版本迭代时，**验收清单必须重新过一遍**：

| 操作 | 判据 | 动作 |
|------|------|------|
| **保留** | 验收项对应的功能/规则在新版本中仍然存在（无论旧版是否已完成验收） | 保留该项，已完成标记 `[X]` 继续保留 |
| **删除** | 验收项对应的功能已在新版本中被移除/替换/改掉 | 删除该项 |
| **新增** | 新版本引入了旧版没有的功能/规则 | 新增验收项。新增前先对照已有清单，**已有同类验收项的不新增**（避免重复） |

> **原则**：版本迭代不是"清空重来"——已通过的验收项仍有追溯价值（证明某个版本曾达成），但已不存在的功能对应的验收项应清理，避免腐化。

> **原则**：旧版 HTML 不删除（可回溯），MVP 文档持续更新（保持最新），CHANGELOG 记录每次迭代的 WHY。交付收口时，MVP 文档全部归档（见 §交付收口归档规则）。

## CHANGELOG 模板

每个 demo 文件夹内创建 `CHANGELOG.md`，格式：

```markdown
# {原型名} · 版本变更记录

> **本文是什么**：记录版本迭代的"为什么改"——重点在 WHY。
> **本文不是什么**：MVP 文档的替代品（MVP 管"现在是什么"）；git log 的替代品（git 管"改了什么文件"）。
> **维护**：每次迭代时追加一条记录。

---

## 格式

每行：`(YYYY-MM-DD) [类型] 一句话 + 原因`

类型：`增` / `删` / `改` / `修`（修 = bug 修复）

---

## 版本记录

### v1 (YYYY-MM-DD)
- `(日期)` `[增]` 初始版本：一句话描述 + 来源
```

> **创建时间**：YYYY-MM-DD
> **来源**：通过 FromDemoToSpec「开始原型验证」指令生成

---

## 项目简介

[1~2 句话：玩法类型 + 核心机制名称。**禁止写入**：技能数量、伤害公式、具体数值、规则细节。机制细节一律以 Spec 为准。]

## 文件导航

| 文件 | 用途 | 读者 |
|------|------|------|
| `{原型名}_input.md` | 输入文档：原始需求与问答补齐的完整记录 | 设计者回顾 / AI Spec 提取 |
| `{原型名}-v{N}_mvp.html` | 可玩原型：浏览器直接打开即玩（零外部依赖） | 设计者试玩验证 |
| `{原型名}-v{N}_mvp.md` | MVP 设计文档：功能概述 + 游戏设计 + 配置项 + 验收清单 | 设计者验收 / AI Spec 提取 |
| `{原型名}-v{N}_spec.md` | Spec 文档：AI 从原型代码提取的结构化技术规格（生成时机：试玩验证通过后） | 程序员实现参考 |
| `{原型名}-v{N}_spec-technical.md` | Technical Spec：状态机/数据流/localStorage 等技术实现细节 | 程序员实现参考 |
| `{原型名}-v{N}_design-doc.md` | 策划文档（WHY）：设计意图、体验目标、验证结论 | 策划 / 团队对齐 |
| `{原型名}-v{N}_design-doc-project.md` | 策划文档（WHAT+HOW）：页面/埋点/动效/音效等落地执行清单 | 程序/美术/音效/QA |
| `交付说明.md` | 交付说明：以哪个文件为准、实现范围、下游消费指引 | 下游接收方（第一份必读） |
| `CHANGELOG.md` | 版本变更记录：每次迭代的"为什么改" | 回溯决策历史 |
| `归档/` | 交付收口时归档的历史文件（旧版 mvp.md / input.md / 对话记录.txt 等） | 有问题时回溯 |

> **说明**：不同阶段原型包含的文件不同——早期只有 input + HTML + MVP + CHANGELOG + README；试玩验证通过并执行「开始 Spec 提取」后才会生成 spec / technical spec / design-doc 系列文件。
> **版本迭代规则**：旧版 HTML 不删除（可回溯），文件夹不含版本号，版本信息在文件名和 CHANGELOG 中。详见 `../README.md`。

## 交付收口归档规则

执行「准备交付」指令时，以下文件移入原型文件夹的 `归档/` 子目录：

| 文件类型 | 归档范围 | 说明 |
|---------|---------|------|
| `_mvp.md` | **全部版本**（不留最新版） | 设计意图由 `_design-doc.md` 替代；验收功能完成即结束 |
| `_mvp.html` | 除当前最新版外的所有旧版 | 保留最新版一份供下游试玩 |
| `_input.md` | 全部 | 原始需求记录，已完成使命 |
| `对话记录.txt` | 全部 | 沟通过程记录，已完成使命 |
| 旧版 `_spec.md` / `_spec-technical.md` / `_design-doc.md` / `_design-doc-project.md` | 若有多版则归档旧版 | 仅保留最新版 |

**归档后文件夹结构**：

```
{原型名}/
├── README.md
├── 交付说明.md
├── CHANGELOG.md
├── {原型名}-v{N}_mvp.html          ← 最新版
├── {原型名}-v{N}_spec.md
├── {原型名}-v{N}_spec-technical.md
├── {原型名}-v{N}_design-doc.md
├── {原型名}-v{N}_design-doc-project.md
└── 归档/
    ├── {原型名}-v1_mvp.html
    ├── {原型名}_input.md
    ├── 对话记录.txt
    └── ...
```

## 注意事项

- 原型 HTML 必须**单文件零依赖**，符合 `02-html-prototype-spec.md` 规范
- MVP 文档的验收清单**全部从游戏侧可验证**（不需要打开开发者工具）
- 本目录不 commit 临时/调试产物（`.tmp` / `draft` 等）
- 版本迭代时**不删除旧版 HTML**，保留在文件夹内作为历史参照
- README 的"项目简介"只写玩法类型+核心机制名称，**不写具体数值/技能列表/规则细节**
- 交付收口前不执行归档（保持工作态）；交付收口时按 §交付收口归档规则 执行

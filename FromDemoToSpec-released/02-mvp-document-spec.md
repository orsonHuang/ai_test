# 02 · MVP 设计文档生成规范

> 适用范围：已验证的 HTML 原型 → 结构化 MVP 设计文档
> 注入对象：AI（System Prompt）
> 版本：v1.0

---

## 一、文档定位

MVP（Minimum Viable Product）设计文档是**原型验证后的第一个正式输出物**，承担以下职责：

1. **沟通桥梁**：让未参与原型开发的人（程序、美术、策划/产品）快速理解"这个原型用起来什么样"
2. **验收基准**：定义"做到什么程度算 MVP 完成"，避免无限膨胀
3. **迭代锚点**：记录当前版本的已知问题和验证结论，为下一版提供上下文

MVP 文档**不是**最终策划文档，**不是**技术 Spec。它介于"可运行原型"和"正式文档"之间，侧重**可执行的功能描述 + 可验证的验收标准**。

> **通用性说明**：本规范以游戏原型为示例，但结构适用于所有原型类型——产品/工具/仪表板/表单等。非游戏类原型将"玩法"替换为"功能"，"胜负"替换为"完成目标"即可。

---

## 二、生成时机

MVP 文档与原型**同步生成**，不等待验证完成再生成：

```
AI 生成 HTML 原型
        │
        ▼（立即，同一轮次）
AI 读取：原型源码 + 输入模板
        │
        ▼
AI 按本规范生成 MVP 文档 V1（版本状态：V1 · 待试用验证）
        │
        ▼
设计者试用验证
        │
        ├─ 假设成立 / 部分成立 → 更新为 V2 · 已验证，补充验证结论
        │
        └─ 发现需调整数值/规则 → 更新为 V3+ · 已调整，记录调整项
```

> **不生成的情况**：输入模板（A 段验证目标）未填写，即没有可验证的假设。
>
> **版本状态约定**（在头部元信息中标注）：
> - `V1 · 待试用验证`：与原型同步生成的初稿，尚未有外部试用者反馈
> - `V2 · 已验证`：至少 1 位非设计者试用，核心假设被确认或证伪
> - `V3+ · 已调整`：基于验证反馈做了参数/规则调整
>
> **版本迭代时验收清单的处理**：详见 `Demo/README.md` §验收清单迭代规则。核心原则：已失效功能对应的验收项删除；仍存在的保留（已完成标记不丢）；新增项去重后再加。

## 四、强制结构（必须包含，缺一不可）

### 4.1 头部元信息

```markdown
# <demo-name> · MVP 设计文档 v<版本>

> **来源原型**：`Demo/<demo-name>/<demo-name>.html`
> **基于输入模板**：`input-examples/input-example-<类型>.md`
> **生成时间**：YYYY-MM-DD
> **版本状态**：V1 · 待试用验证 / V2 · 已验证 / V3 · 已调整 …
```

> 版本状态定义见「二、生成时机」。

---

### 4.2 一、功能概述

#### 1.1 原型名称 / 功能名称
- 一句话命名，包含核心机制/功能 + 关键参数
- 游戏示例：`战棋对抗（6×6 网格 · 回合制 · PvE）`
- 产品示例：`多步骤审批表单（3 步 · 条件字段 · 实时校验）`

#### 1.2 核心体验
- 3~5 句话描述"用户在这个原型里做什么、感受什么"
- 必须包含：用户动作、核心抉择、完成目标

#### 1.3 验证目标

| 项目 | 内容 |
|------|------|
| **核心假设** | 这个原型最想验证的假设（只能有 1 个，不可多） |
| **验证判据** | 可量化的"假设成立"标准 |
| **不验证** | 明确排除在本次 MVP 范围外的内容（美术、音效、联网、长期系统等） |

> **约束**：核心假设必须是"如果这条不成立，整个方案就站不住脚"级别的命题，不能是"好用不好用"这种主观描述。

---

### 4.3 二、核心设计

#### 2.1 场景/界面规格
- 类型（网格/卡牌/表单/图表/对话树/页面流程等）
- 尺寸/数量/层级/布局
- 特殊元素（如有）

#### 2.2 时序/流程结构
- 时序描述（阶段流转）
- 阶段流转图（`idle → userInput → processing → result → ...`）

#### 2.3 元素/实体/组件设计
- 表格形式：名称、类型、关键属性、特殊能力/行为
- 初始状态/默认值

#### 2.4 核心规则
- 用 `规则N：` 编号列出所有影响结果的业务规则
- 必须包含：边界条件（禁止发生的行为）
- 必须包含：特殊情况处理（平局、死锁、超时、空状态等）

#### 2.5 自动化行为规则（如适用）
- 优先级列表（从高到低）
- `[存疑]` 标注：设计者尚未拍板的部分

#### 2.6 完成/结束条件
- 表格：结果 → 条件
- 游戏类：胜负/平局/超时
- 产品类：提交成功/校验失败/流程中断

#### 2.7 交互流程
- 用户从打开原型到完成一次核心任务的操作链路
- 用箭头 `→` 串联，覆盖关键反馈节点（高亮、动画、数字变化、提示信息等）

---

### 4.4 三、配置项清单

> 所有在原型 `CONFIG` 对象中集中管理的参数必须列出。

| 参数名 | 默认值 | 可调范围 | 生效方式 | 说明 |
|--------|--------|----------|----------|------|
| `PARAM_NAME` | `value` | `min ~ max` | `即时/重置后` | 一句话说明 |

**必须包含**：
- 公式/计算逻辑详解（用代码块）
- 数值示例（至少 3 组，覆盖典型场景）

---

### 4.5 四、验收清单

> **验证方式**：纯用户侧验证，不需要打开浏览器开发者工具或查看代码。
> 每条验收项可独立执行，完成后标记 `[x]`。

分五类列出：

1. **基础交互**：用户能否顺利完成核心操作（选择、输入、确认、重置、返回）
2. **核心逻辑**：规则是否按预期生效（计算结果、状态变化、条件触发、完成判定）
3. **调试面板**：参数调整是否即时生效、面板信息是否准确
4. **边界条件**（用户视角）：异常操作是否被优雅处理（误触、空输入、时序错误、状态冲突）
5. **容错与韧性**（开发者视角）：性能上限、状态机非法转换、try-catch 兜底等"代码级防御"在用户操作下是否真正生效。该类对应原型 Spec 模板的「边界条件与容错」段，是对该段防御逻辑的实际验证

> **约束**：每条验收项必须是"不打开代码也能判断通过/失败"的操作描述。禁止出现"检查 `game.js` 第 N 行"这类要求。
> **第 4 vs 第 5 类的区别**：第 4 类是"用户能感知到的异常"（点了空白处、输入超长文本），第 5 类是"代码做了防御但用户未必知道"（连续触发 100 个特效是否仍流畅、断网后重连数据是否仍在）。两类互补，不重叠。

---

### 4.6 五、已知问题 ★必填

> **必填段**——即使本版本暂无已知问题，也必须写明"已扫描以下位置（列出 2-3 个重点扫描的模块），暂无发现"。
> **不允许留空或省略本段**。复核发现"自我诚实是工作流的隐性价值"——无工作流版完全没有已知问题列表，导致风险被掩盖。

| 位置 | 描述 | 类型 | 影响 |
|------|------|------|------|
| 模块/功能点 | 一句话描述 | `存疑`/`待定`/`改进项` | 对验证的影响程度 |

**类型定义**：
- `存疑`：设计层面未拍板，可能影响核心假设验证
- `待定`：已知但不在当前版本修复，不影响核心验证
- `改进项`：体验可优化，但不阻塞 MVP

**若无已知问题，填写格式**：

```
本版本已扫描的重点位置：[模块A] / [模块B] / [模块C]
扫描结论：暂无已知问题
扫描时间：YYYY-MM-DD
```

---

## 五、生成约束

1. **不可编造**：所有数值、规则、配置项必须与原型源码 `CONFIG` 对象一致
2. **不可遗漏**：`CONFIG` 中每一个可调参数都必须出现在"配置项清单"中
3. **必须标注存疑**：AI 对任何不确定的设计决策（如自动化行为优先级、边界条件处理）必须用 `[存疑]` 标注，不可假装确定
4. **验收项可执行**：每条验收清单条目必须能被非技术人员在 30 秒内判断是否通过
5. **版本状态诚实**：如果原型未经外部试用，版本状态必须是 `V1 · 待试用验证`，不可写 `已验证`

## 六、与 03 的衔接

MVP 文档生成后，下一步是 **03-prototype-to-spec-mapping.md** 的 Spec 提取：

- MVP 文档中的"核心规则"和"配置项清单"是 Spec 提取的主要输入
- MVP 文档中的"已知问题"应同步写入 Spec 的"风险/待决策"章节
- MVP 文档的"验收清单"是 Spec 中"测试用例"的前置素材

```
MVP 文档（面向"这个原型用起来什么样"）
        │
        ▼
03 映射规范 → Spec 文档（面向"程序怎么实现"）
        │
        ▼
04 策划文档模板 → 正式策划文档（面向"设计意图与验证结论"）
```

# 通用原型描述模板 v1.3

> **使用方式**：填写本模板 → 将完整模板 + 《HTML 原型编写规范》一同喂给 AI → AI 生成 HTML 原型
>
> **不知道怎么填？** 让 AI 走 `01-input-elicitation-protocol.md` 的问答补齐流程——通过开放题 + 针对性追问帮你补齐本模板，跳过的字段也会有兜底处理。
>
> **分层策略**：
> - 必填区：所有原型类型必须填写
> - 选填区：根据你的原型类型勾选对应模块填写（可多选）
> - 跳过项与处理方式：通过问答补齐时使用本段记录跳过项与最终处理

## 必填区：所有原型通用

### A. 验证目标

- **要验证的核心假设是什么？**
  > 例："玩家是否会在 A 和 B 两个战术选择之间产生有意义的取舍"
- **验证成功的判据是什么？**
  > 例："试玩 3 次后，至少有 2 种不同的有效策略出现"
- **不需要验证什么？（明确排除范围）**
  > 例："不需要验证美术风格和音效表现"

### B. 核心交互

- **用户做什么操作？**
  > 例：点击选择单位 → 点击选择目标格子 → 单位移动攻击
- **系统如何响应？**
  > 例：播放移动动画 → 计算伤害 → 更新 HP → 检查是否击败
- **交互循环（用箭头表示）**
  > 例：选择行动 → 执行行动 → AI 行动 → 回合结束 → 新回合开始

### C. 数据对象

| 对象名 | 关键属性 | 与其他对象的关系 |
|--------|----------|------------------|
| > 例：Unit | id, name, hp, atk, def, moveRange, team | 属于一个 Player / AI |
| | | |

### D. 状态与流程

- **系统有哪些状态？**
  > 例：等待开始 → 玩家回合 → 敌方回合 → 结算 → 结束
- **状态之间如何切换？（箭头 + 触发条件）**
  > 例：玩家回合 → （所有单位行动完毕） → 敌方回合
- **初始状态是什么？结束状态是什么？**

### E. 核心规则

```
- 规则 1：
- 规则 2：
- 规则 3：
- 边界条件（必须不能发生的事）：
- 特殊情况处理（如平局、死锁、溢出）：
```

### F. UI 骨架

- **整体布局描述**
  > 例：上方信息栏 + 中央 8×8 棋盘 + 下方操作区
- **关键视觉元素的位置**
  > 例：HP 条显示在单位上方，行动力用圆点表示
- **信息展示优先级**（玩家最需要看到什么，排前面）

### G. 可调参数（调试面板数据源）

> **最少填 3 个**（对应原型编写规范 §2.2 强制要求，少于 3 个 AI 会追问）

| 参数名 | 默认值 | 可调范围 | 说明 |
|--------|--------|----------|------|
| > 例：DAMAGE_MULTIPLIER | 1.0 | 0.5 ~ 3.0 | 全局伤害倍率 |
| | | | |
| | | | |

## 填写说明

1. **"不确定"比"硬填"更好**——在不确定的字段标注 `[待澄清]`，AI 会追问而非猜测；或直接走 00 问答补齐流程
2. **越具体越好**——"伤害高"不如"伤害约 20% HP，期望 5 次攻击击杀一个同级单位"
3. **不是写设计文档**——本模板是 AI 的输入，不需要完美措辞，需要明确约束
4. **可调参数是关键**——这直接影响调试面板的质量。多列几个，宁可列多了不用

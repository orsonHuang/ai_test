# 00 · Input 问答补齐协议

> **本文是什么**：把"填 02 模板"这件硬邦邦的事，变成 AI 跟设计者用大白话聊天聊出来的过程。
> **本文不是什么**：02 模板的替代品。02 还是终点，本协议只是到达 02 的"自然路径"。
> **使用时机**：设计者脑子里有想法但还没整理；或者打开 02 模板看了一眼觉得无从下手。
> **触发方式**：用户说「**开始原型验证**」或同义触发词（详见 `00-command-manual.md`）即启动本协议。AI 直接进入下文「阶段 1 · 开放钩子」。
> **行数预算**：360 行内（含三级降级策略，v1.1 起扩容）。

---

## 一、设计哲学

### 为什么需要这个协议

直接让设计者填 02 模板，会出现两种典型卡点：
1. **冷启动卡点**：脑子里有想法但是看到字段就懵——"验证目标？我也不知道我要验证啥啊"
2. **半成型卡点**：填了一半，剩下的字段（比如"AI 行为规则""通胀安全阀"）专业感太强，跳过不甘心，硬填又凭空捏造

本协议把模板填写过程包装成 AI 主导的**自然对话**——设计者只回答"人话级别的问题"，AI 在背后做字段映射 + 类型识别 + 完整度跟踪。

### 设计原则

1. **接地气**：不出现"验证目标""核心交互""数据对象"这类术语。AI 用例子和打比方
2. **小步快跑**：每轮 ≤3 个问题，避免压垮
3. **允许跳过**：用户说"不知道""跳过""[待澄清]"都接住，AI 记下，全部问完再回头处理
4. **AI 兜底有迹可循**：跳过项最终由 AI 用合理默认值处理时，必须把"处理方式"写进 input 文档，可追溯

---

## 二、三段式问答漏斗

```
阶段 1：开放钩子（5 题左右，AI 动态调整）
   │  目的：把设计者带入状态，撬出 WHY/WHAT
   │  产出：AI 内部映射出 A 验证目标 + B 核心交互 + E 关键规则的雏形
   ▼
阶段 2：类型识别 + 示例锚定（含三级降级兜底）
   │  目的：定原型类型（11 类通用 + 1 类 -sp 中选一）
   │  前置：【必须】扫描 input-examples/ → 读取候选 → 无匹配走三级降级
   │  产出：选定 input-example（或降级路径）+ spec-template，让设计者确认/纠偏
   ▼
阶段 3：针对性追问（按 gap 驱动）
   │  目的：补齐字段。【必先完整读取】锚定 input-example（或降级路径模板）作为"满血版"对照
   │  产出：完整或基本完整的 input 信息 + 跳过项清单
   ▼
阶段 4：跳过项回顾（如有跳过项才进入）
   │  目的：处理阶段 3 累积的跳过项
   │  产出：跳过项的最终处理方式（用户答 / AI 推断 / AI 兜底）
   ▼
生成 input 文档（含跳过项与处理方式记录）
```

## 三、阶段 1 · 开放钩子

### 标准 5 题（接地气版，AI 可根据上下文增删 1~2 题）

> AI 一次性把 5 题列出来，让设计者一口气回答（也允许只回答能答的）。
> 不强制设计者按编号回答——任何信息都先收着。

```
1. 你脑子里这个东西大概是个啥？玩起来 / 用起来什么感觉？
   （随便描述就行，比如"像 XXX 那样的塔防""一个填表单的工具但希望它别那么烦"）

2. 你最想验证的是什么？换句话说——你最不放心 / 最好奇 / 最想看看到底行不行的那个点。
   （想到几个写几个，越具体越好；也可以说"我自己都说不清"）

3. 玩家 / 用户在用的过程中，最关键的那一下"做选择"是什么？
   （比如塔防里的"在哪建塔"，表单里的"要不要勾这个选项"——找那个让人卡住一下、纠结一下的瞬间）

4. 如果这个东西做出来翻车了，你猜最可能在哪里翻？
   （可以是"玩家可能根本看不懂"、"参数一调就崩"、"做完没人想再玩第二次"，等等）

5. 如果你只能告诉做这个东西的程序员一件事，让他必须做对——你会说哪件事？
```

### AI 动态调整规则

| 情况 | 调整 |
|------|------|
| 设计者第一句话已经把 1+2 都说清了 | 跳过 1+2，直接问 3+4+5 |
| 设计者答得很短（每题 < 10 字） | 增加 1~2 题挖深，比如"玩家会什么时候掏钱 / 流失？" |
| 设计者一上来就开始讲数值/规则 | 先收着，但要追问 1+2，避免一头扎进实现细节 |
| 设计者明显在描述非游戏类（工具/UX/仪表板） | 把 3 改成"用户在哪一步最容易放弃 / 最容易出错" |

### 输出（AI 内部，不给用户看）

阶段 1 结束时，AI 心里有一张草稿表：
```
A.验证目标：[已大致明确 / 还很虚]
B.核心交互：[已大致明确 / 还很虚]
E.关键规则：[已大致明确 / 还很虚]
原型类型猜测：[XXX 类，置信度 中/高/存疑]
建议 input-example：[input-example-XXX.md / 无明确候选，需走降级]
```

## 四、阶段 2 · 类型识别 + 示例锚定

### 阶段 2 前置动作（AI 必须先执行，再向用户陈述）

进入阶段 2 时，AI **不能跳过文件 I/O 直接判型**，必须按顺序完成：

1. **扫描 input-examples/ 目录**：列出所有可用 `.md` 文件名
2. **读取候选**：基于阶段 1 的"建议 input-example"，打开并完整读取 1~2 个候选文件
3. **对照确认**：对照候选文件的实际内容（必填区怎么填的、选填模块勾了哪些），确认类型判断与候选文件的一致性
4. **完成以上三步后**，才向用户陈述判型结果

> 若扫描后发现无匹配候选（文件名对不上 / 内容明显不对口），不跳步——走下方「锚定失败：三级降级策略」。

### AI 主动判型

基于阶段 1 回答，AI 用接地气的话告诉设计者类型判断：

```
听下来你这个像是【即时制塔防类】的东西。我能找到一个挺像的填好的样板：
input-example-realtime-towerdefense.md。我打算照着那个的结构来跟你聊后面的细节。
你觉得这个判断靠谱吗？还是说我理解偏了？
```

### 类型识别决策辅助

参考 `03-prototype-to-spec-mapping.md` 的选型表。11 类通用 + `match3-rpg-sp` 定制。

**注意**：`-sp` 后缀（special-purpose）表示模板作者主观判定"该组合不够通用"，**不影响 AI 的类型判断和推荐逻辑**。AI 判断类型匹配后应正常推荐 `-sp` 示例，`-sp` 仅作为信念度调低的辅助信号（例如标注"中"而非"强"）。

### 输出（AI 给用户）

- 锚定的 input-example 文件名（或降级路径：`spec-template-XXX.md` / `01-prototype-input-template.md` 通用模板）
- 信念度标注（强 / 中 / 存疑 / 存疑-无模板）
- 一句话给用户校准的机会

如果用户说"不像"或"我想做不一样的"，AI 让用户自己描述差异在哪，然后回到阶段 1 的开放题再聊一轮。

### 锚定失败：三级降级策略

若阶段 2 前置动作完成后判定**无匹配的 input-example**（无同类文件名 / 内容类型明显不对口），AI **不能跳过锚定步骤直接判型**，按下表三级降级：

| 级别 | 动作 | 命中条件 | 命中后 |
|------|------|----------|--------|
| **第一级** | 扫描 `spec-templates/` 目录，寻找最接近的 spec-template | 有文件名风格相近的模板（如 `spec-template-realtime.md`） | 以该 spec-template 为锚定骨架，结合通用模板字段结构推进阶段3。锚定标注："spec-template-XXX.md（信念度：中，无直接 input-example）" |
| **第二级** | 向用户明确报告，寻求确认 | 第一级也找不到相近模板 | 报告原文："🔍 我扫描了 `input-examples/` 和 `spec-templates/`，没有找到跟你的描述匹配的现成模板。我准备用通用模板 `01-prototype-input-template.md` 继续后面的问答补齐流程。你觉得可以吗？还是你想手动指定一个方向？" |
| **第三级** | 通用模板兜底 | 用户确认 / 用户无异议 | 使用 `01-prototype-input-template.md` 通用模板作为基础框架。阶段 3 的 gap 检测改为逐段对照通用模板的必填区（A~G），不依赖"满血版"对照模式。锚定标注："无匹配，使用通用模板 01-prototype-input-template.md（信念度：存疑-无模板）" |

> **原则**：宁可低信念度锚定一个模板，也不在"猜"的状态下错判类型。三级走完后仍未锚定 = 不允许进入阶段 3。

## 五、阶段 3 · 针对性追问

### gap 检测

**【必须】** AI 先完整读取锚定的 input-example（或降级路径选定的 spec-template / 通用模板）文件全部内容，将其作为"满血版"对照基准，然后对照阶段 1 收到的信息，列出哪些字段还空着。**未读取文件前不得开始 gap 检测。**

若走第三级降级路径（通用模板兜底），则逐段对照 `01-prototype-input-template.md` 通用模板的必填区（A~G），替代"满血版"对照模式。

每轮追问遵守：
- **批量 ≤3 题**
- **带建议选项**：尽量让设计者"选"而不是"想"
- **接地气**：不直接说字段名，用日常语言

### 追问示例（塔防）

❌ 不要这样：
```
请填写 G 段可调参数（最少 3 个）：
- 参数1：
- 参数2：
- 参数3：
```

✅ 要这样：
```
原型里我准备给你做一个调试面板，可以现场调几个数字看效果。
塔防类常见会调这几个，你觉得哪些重要：
- 怪物的血量倍数（让怪更肉 / 更脆）
- 塔的伤害倍数
- 怪物刷新的快慢
- 怪物每波数量
- 玩家初始金币
你挑 3~5 个，没在列表里也可以加。
```

### 跳过的接收方式

设计者说以下任一即视为"跳过"：
- "不知道"
- "你看着办"
- "[待澄清]"
- "跳过"
- 沉默两轮（用户隐式跳过）

**AI 必须显式确认**：
```
好，「XX」这个我先记下，等所有问题问完了我们一起回头看一遍。
```

### AI 内部维护：跳过项清单

```
跳过项：
1. AI 行为规则（阶段3-第2轮）
2. 通胀安全阀（阶段3-第3轮）
3. 异常高亮规则（阶段3-第4轮）
```

## 六、阶段 4 · 跳过项回顾（关键机制）

> 这一步用户拍板的，是本协议的核心差异点。

### 触发条件

阶段 3 全部追问完成 → 检查跳过项清单 → 非空就进入本阶段。

### 三段处理

每个跳过项依次走以下流程：

#### 第 1 段 · AI 自查能否直接推断

```
对照阶段 1~3 已收集的所有信息，我能不能从已有信息推断出这一项的合理值？
```

能推断 → 直接落定，但在 input 文档里**标注 "AI 推断（基于 X+Y+Z）"**

#### 第 2 段 · 带建议的二轮追问

不能推断 → 再问一次，但这次必须带 2~3 个具体建议 + 信念度：

```
回头看下「AI 行为规则」这个我们之前跳过了，我有几个建议你挑一下：

1. （强信念）近战优先打血最少的，远程优先打离自己最近的——这是塔防里最常见的
2. （中信念）所有 AI 一律打血最少的——简单粗暴，验证波次节奏够用
3. （存疑）随机选目标——只在你想测"运气因素"时用

你选哪个？还是有别的想法？
```

#### 第 3 段 · AI 兜底 + 处理方式备注

设计者依然说不知道 / 还是跳过 → AI **自己拍板**，但必须：
1. 选第 2 段建议中信念度最高的
2. 在 input 文档「跳过项与处理方式」段写清楚：
   - 哪一项被跳过
   - 三段处理走到第几段
   - 最终用什么值
   - 为什么选这个（依据是什么）

```
好，这一项我用方案 1（近战优先打血最少 + 远程优先打最近的）兜底。
原因：强信念 + 跟主流塔防一致，原型出来后你试玩有不对劲再跟我说。
我把这个处理方式记到 input 文档里了。
```

## 七、终止条件

任一满足即终止问答，进入生成阶段：

1. **完整度达标**：必填区 A~G 每段 ≥60% 完整 + 选填区至少 1 个模块完整
2. **用户主动收束**：连续 2 轮用户回答都是"够了 / 可以了 / 你看着办"
3. **达到轮次上限**：阶段 3 追问轮次 ≥5 轮（防止无限问）

满足任一 → AI 进入阶段 4（如有跳过项）→ 生成 input 文档 + HTML 原型 + MVP 文档 + README.md + CHANGELOG.md + 保存完整对话记录为 `对话记录.txt`（位于 `Demo/{原型名}/`）。

## 八、输出物：input 文档结构

> 这是本协议的**最终交付物之一**，与 HTML 原型、MVP 文档并列。

文件名：`{原型名}_input.md`，存放路径：`Demo/{原型名}/`

文档结构：

```markdown
# {原型名} · Input 文档 v{版本号}

> 本文档由 00 问答补齐协议生成。基于 02 模板 + 对话补齐。

## 一、原型类型

- 锚定示例：input-example-XXX.md
- 锚定 Spec 模板：spec-template-XXX.md
- 类型判定信念度：[强 / 中 / 存疑]

## 二、必填区

### A. 验证目标
[填写内容]

### B. 核心交互
[填写内容]

... (按 02 模板顺序)

## 三、选填区（按勾选模块）

### [✓] XX 类
[填写内容]

## 四、跳过项与处理方式

| 字段 | 三段处理走到 | 最终处理 | 依据/原因 |
|------|------------|---------|----------|
| 例：AI 行为规则 | 第3段（AI 兜底） | 近战打血最少+远程打最近 | 强信念主流方案，待试玩验证 |
| 例：通胀安全阀 | 第1段（AI 推断） | 资源上限设 9999 | 基于已填的"验证窗口=10分钟"推断 |

## 五、附录：AI 备注（可选）

> AI 在问答过程中观察到的、值得记录但不影响原型生成的信息。
> 例如：用户在哪个问题上犹豫最久 / 用户表达过哪些"虽然不在模板里但很重要"的信息。

[AI 自由记录，没有可省略本段]
```

## 十、AI 执行检查清单

每次走本协议时，AI 自检：

- [ ] 阶段 1 用了**接地气的话**问，没出现"验证目标/核心交互/数据对象"等术语
- [ ] 阶段 1 结束时已扫描 `input-examples/` 目录，内部草稿表已标注"建议 input-example"
- [ ] 阶段 2 前置：已打开并完整读取 1~2 个候选 input-example 文件（内容对照确认，不是猜的）
- [ ] 阶段 2 无匹配时已走三级降级策略，未跳过锚定步骤
- [ ] 阶段 3 前置：已完整读取锚定 input-example（或降级路径模板）全部内容作为 gap 检测基准
- [ ] 阶段 2 主动判型 + 给信念度 + 给用户校准机会
- [ ] 阶段 3 每轮 ≤3 题 + 带建议选项 + 显式接住"跳过"
- [ ] 阶段 4 对每个跳过项严格走三段处理
- [ ] 终止条件检查（不无限问）
- [ ] 生成 input 文档，且「跳过项与处理方式」段写清楚

# 通用原型描述模板 · 表单/流程类填写示例

> **用途**：本文件是 `01-prototype-input-template.md` 的表单/流程类完整填写参考。
> **对应 Spec 模板**：`spec-templates/spec-template-form.md`
> 参照场景：玩家创建游戏房间的多步表单（3 步骤，含条件字段和实时校验）。
> 读者可对照本文件理解"填到什么程度 AI 才能动"。
> **填写难度信号**：本示例填写完整度 = 100%。如果你看完觉得「自己填不到这个程度」，建议改走 `01-input-elicitation-protocol.md` 的问答补齐流程——AI 会通过对话帮你补齐。

## 必填区：所有原型通用

### A. 验证目标

- **要验证的核心假设是什么？**
  玩家在创建房间时，当前 3 步表单设计是否让他们在"步骤 2（对局配置）"产生明显困惑或放弃——即步骤分割方式和字段顺序是否合理，信息量是否超载。

- **验证成功的判据是什么？**
  3 位从未见过此界面的玩家独立完成创建流程，步骤 2 停留时间 ≤ 90 秒，且无人在步骤 2 中途退出。如有人退出，记录退出节点。

- **不需要验证什么？**
  不需要验证：后端接口实现、权限控制、房间真实对战逻辑、支付流程。只验证表单流程本身的摩擦点。

---

### B. 核心交互

- **用户做什么操作？**
  步骤1：填写房间名称+选择游戏模式 → 步骤2：配置对局参数（根据模式不同显示不同字段）→ 步骤3：确认预览+提交

- **系统如何响应？**
  字段失焦时立即校验并显示错误提示；"下一步"按钮在当前步骤所有必填项通过前禁用；步骤3展示所有已填信息的只读预览；提交后跳转成功页

- **交互循环（用箭头表示）**
  ```
  步骤1（基础信息）→ 校验通过 → 步骤2（对局配置）→ 校验通过 → 步骤3（确认预览）→ 提交 → 成功页
  任意步骤 ← 点击"上一步"按钮（保留已填内容）
  ```

---

### C. 数据对象

| 对象名 | 关键属性 | 与其他对象的关系 |
|--------|----------|------------------|
| FormState | currentStep, data{}, errors{}, isDirty | 全局单例；data 保存所有字段值，errors 保存校验结果 |
| FieldConfig | id, label, type, required, validation, conditionalOn | 描述单个字段的配置；conditionalOn 指向父字段ID+触发值 |
| StepConfig | stepIndex, title, fields[] | 描述单个步骤的字段列表；fields 引用 FieldConfig |
| ValidationResult | fieldId, isValid, errorMessage | 由校验函数返回；写入 FormState.errors |

---

### D. 状态与流程

- **系统有哪些状态？**
  `step1`（基础信息）→ `step2`（对局配置）→ `step3`（确认预览）→ `submitting`（提交中）→ `success`（创建成功）

- **状态之间如何切换？**
  ```
  step1      → （当前步骤所有必填项通过，点"下一步"）  → step2
  step2      → （当前步骤所有必填项通过，点"下一步"）  → step3
  step3      → （点"提交"）                           → submitting
  submitting → （模拟成功）                            → success
  任意步骤   → （点"上一步"）                          → 前一步骤（保留数据）
  success    → （点"再创建一个"）                      → step1（清空数据）
  ```

- **初始状态**：`step1`，所有字段空值，所有错误清空
- **结束状态**：`success`，展示"房间 [name] 已创建"+ 房间码

---

### E. 核心规则

```
- 规则1：必填字段在失焦时触发实时校验（不等到"下一步"才统一报错）
- 规则2：条件字段（conditionalOn）在父字段值不满足时隐藏，隐藏时不参与校验
- 规则3："下一步"按钮在本步骤有任意未通过必填项时禁用（灰色，不可点击）
- 规则4：步骤3为只读预览，不可修改，点击可跳回对应步骤编辑
- 规则5：所有步骤数据在前进/后退时保留（不清空）
- 边界条件（必须不能发生的事）：
    - 不允许跳过步骤直接提交
    - 不允许"下一步"在校验未通过时触发状态切换
    - 数字字段不允许输入非数字字符（输入时直接过滤）
- 特殊情况处理：
    - 提交失败（模拟网络错误）：回到 step3，显示红色错误提示，不清空数据
    - 字段值合法但业务规则冲突（如人数超过模式上限）→ 在"下一步"时统一检查，显示跨字段错误
```

---

### F. UI 骨架

- **整体布局**：
  顶部步骤进度条（1→2→3）+ 中央表单区（单列，最大宽 480px，居中）+ 底部操作区（上一步/下一步/提交）+ 右下调试面板

- **关键视觉元素位置**：
  - 步骤进度条：顶部固定，当前步骤高亮圆圈+文字
  - 表单字段：单列排列，字段标签在上，输入框在下；必填项标签后显示红色 *
  - 错误提示：紧贴输入框下方，红色小字（不是弹窗）
  - 下一步按钮：底部右对齐，禁用时灰色不可点
  - 预览页（步骤3）：所有字段以 label: value 列表形式展示；旁边有"编辑"链接跳回对应步骤

- **信息展示优先级**：
  1. 当前步骤的字段（最大占比）
  2. 错误提示（立即可见，不需要滚动找）
  3. 步骤进度感（顶部进度条）
  4. 操作按钮（底部固定）

### G. 可调参数（调试面板数据源）

| 参数名 | 默认值 | 可调范围 | 说明 |
|--------|--------|----------|------|
| VALIDATE_ON_BLUR | true | true/false | 是否在失焦时触发实时校验（关闭=只在点下一步时校验） |
| SUBMIT_DELAY_MS | 800 | 0~3000 | 模拟提交延迟（ms），用于测试 loading 状态展示 |
| SIMULATE_SUBMIT_FAIL | false | true/false | 强制模拟提交失败，测试错误恢复流程 |
| SHOW_STEP_NUMBERS | true | true/false | 进度条是否显示步骤编号 |
| PRESERVE_DATA_ON_BACK | true | true/false | 后退时是否保留已填数据 |

## 选填区：表单/流程类

### 字段清单

#### 步骤1：基础信息

| 字段 ID | 标签 | 类型 | 必填 | 校验规则 | 条件显示 |
|---------|------|------|------|---------|---------|
| room_name | 房间名称 | text | 是 | 2~20字，不含特殊字符 | 无 |
| game_mode | 游戏模式 | select | 是 | 枚举：标准/竞技/休闲 | 无 |
| is_private | 私密房间 | checkbox | 否 | 无 | 无 |
| password | 房间密码 | password | 是（条件） | 4~8位数字 | is_private = true |

#### 步骤2：对局配置

| 字段 ID | 标签 | 类型 | 必填 | 校验规则 | 条件显示 |
|---------|------|------|------|---------|---------|
| max_players | 最大人数 | number | 是 | 2~8（标准/休闲），2~4（竞技） | 无 |
| time_limit | 每回合时限（秒）| number | 是 | 15~300 | 无 |
| enable_spectate | 允许观战 | checkbox | 否 | 无 | game_mode ≠ 竞技 |
| ranked_match | 记录天梯分 | checkbox | 否 | 无 | game_mode = 竞技 |

#### 步骤3：确认预览

所有字段只读展示，无新增输入。

### 关键摩擦点假设

```
预期高摩擦点（本次验证目标）：
  步骤2 - max_players 字段：约束来自 game_mode，但步骤切换后玩家可能已忘记自己选了什么模式
  步骤2 - time_limit 字段：单位不直观（秒），玩家可能填 3 而非 180

验证方式：记录玩家在哪个字段停留最久 / 触发了哪些校验错误
```

### 步骤划分逻辑

```
步骤1（What）：这个房间是什么 → 名字、模式、是否私密
步骤2（How）：这场对局怎么玩 → 人数、时限、特殊规则
步骤3（Confirm）：确认后不可改 → 提交前的全量预览

分割原则：信息量 ≤ 4 个字段/步骤，认知负荷可控
```

# 通用原型描述模板 · 构筑/卡组类填写示例

> **用途**：本文件是 `01-prototype-input-template.md` 的构筑/卡组类完整填写参考。
> **对应 Spec 模板**：`spec-templates/spec-template-deckbuilding.md`
> 参照场景：卡牌对战游戏的起始卡池草案（Deckbuilding Draft，从公共牌池轮抽构组）。
> 读者可对照本文件理解"填到什么程度 AI 才能动"。
> **填写难度信号**：本示例填写完整度 = 100%。如果你看完觉得「自己填不到这个程度」，建议改走 `01-input-elicitation-protocol.md` 的问答补齐流程——AI 会通过对话帮你补齐。

## 必填区：所有原型通用

### A. 验证目标

- **要验证的核心假设是什么？**
  当前卡池设计是否存在"必摘路线"——即某一套协同组合（如全攻/全防/特定触发链）强度远超其他，导致有经验的玩家总是走同一路线，新手探索欲被提前掐灭。

- **验证成功的判据是什么？**
  3 位试玩者的最终卡组中，没有超过 2 人选了相同的 5 张卡（共 10 张卡组）；或试玩后玩家能说出"我选这张是因为和 X 能连动"的至少 3 种不同连动理由。

- **不需要验证什么？**
  不需要验证：联网对战、卡牌美术、完整卡池（只验证核心子集 15~20 张）、排行榜机制、长期卡牌获取路线。

---

### B. 核心交互

- **用户做什么操作？**
  看到当前轮的 3 张候选卡 → 选择 1 张加入卡组 → 其余 2 张放回牌池（或废弃）→ 进入下一轮，重复 10 轮 → 展示最终卡组 + 协同分析

- **系统如何响应？**
  选卡后：高亮选中卡，淡出其他两张 → 更新卡组预览区（右侧） → 刷新协同提示（当前卡组已触发的标签连动）→ 展示下一轮 3 张

- **交互循环（用箭头表示）**
  ```
  展示 3 张候选 → 玩家选 1 张 → 更新卡组 + 协同提示 → 是否第 10 轮？
      → 否：进入下一轮
      → 是：展示最终卡组总览 + 协同强度评分 + 重新草稿按钮
  ```

---

### C. 数据对象

| 对象名 | 关键属性 | 与其他对象的关系 |
|--------|----------|------------------|
| Card | id, name, cost, type(攻击/防御/触发/增益), tags[], baseEffect, synergyTriggers[] | 属于 CardPool；被 Draft 抽出；进入 Deck |
| CardPool | cards[], drawPile[] | 持有全部卡牌；每轮从 drawPile 抽 3 张 |
| Draft | currentRound, picks[], cardPool(ref) | 全局草稿进程；picks 是玩家已选卡列表 |
| Deck | cards[], synergyMap{} | 由 Draft.picks 生成；synergyMap 记录当前触发的协同标签 |
| Synergy | tagPair[2], description, bonusEffect | 描述两个标签触发的协同效果 |

---

### D. 状态与流程

- **系统有哪些状态？**
  `init`（初始化牌池）→ `drafting`（轮抽进行中）→ `picked`（本轮已选，等待下一轮）→ `result`（10 轮完成，展示结果）→ `reset`（重置）

- **状态之间如何切换？**
  ```
  init     → （初始化完毕）              → drafting
  drafting → （玩家点击选卡）            → picked
  picked   → （round < 10）             → drafting（自动进入下一轮）
  picked   → （round = 10）             → result
  result   → （玩家点击"重新草稿"）      → reset → init
  ```

- **初始状态**：`init`，CardPool 包含 18 张卡；Deck 为空；round = 1
- **结束状态**：`result`，展示 10 张卡组 + 协同分析 + 协同强度总分

---

### E. 核心规则

```
- 规则1：每轮从 drawPile 随机抽 3 张展示，玩家选 1 张，其余 2 张废弃（不放回）
- 规则2：同一张卡可出现多次（多副本），同名卡叠加时触发"复数效果"（由卡定义）
- 规则3：协同检测：每次选卡后扫描 Deck，统计已激活的 synergyTriggers（需要卡 A 和卡 B 同时在组）
- 规则4：协同强度分 = 所有已激活协同的 bonusEffect 分值之和（展示用，不影响草稿逻辑）
- 边界条件（必须不能发生的事）：
    - drawPile 不足 3 张时：不允许崩溃，从废弃堆重新洗入（或提前结束草稿）
    - 不允许玩家在 result 状态继续选卡
    - 协同检测不允许误报（需要双卡都在 Deck 才计分）
- 特殊情况处理：
    - 牌池耗尽：若 round < 10 且 drawPile < 3，提前进入 result，当前轮跳过
    - 全废弃路线（玩家强行选残次卡）：允许，协同分显示 0，不做强制引导
```

---

### F. UI 骨架

- **整体布局**：
  左侧候选区（3 张卡横排，中央大展示）+ 右侧卡组预览（已选 10 格，展示卡名+标签）+ 底部协同提示区 + 右下调试面板

- **关键视觉元素位置**：
  - 候选卡：居中横排，卡面显示名称/费用/标签/效果描述；悬停高亮边框
  - 已激活协同：底部以标签对显示，绿色字体，旁边显示协同效果描述
  - 卡组预览：右侧列表，已选位格亮显，未选位显示"?"占位
  - 轮次指示器：顶部进度条（1/10 → 10/10）

- **信息展示优先级**：
  1. 当前 3 张候选卡（核心决策区）
  2. 已激活协同提示（帮助识别路线）
  3. 当前卡组预览（已选方向感）
  4. 牌池剩余数量（稀缺感知）

---

### G. 可调参数（调试面板数据源）

| 参数名 | 默认值 | 可调范围 | 说明 |
|--------|--------|----------|------|
| POOL_SIZE | 18 | 10~30 | 卡池总卡数（影响稀缺度和重复出现率） |
| PICKS_PER_DRAFT | 10 | 5~15 | 每次草稿选几张 |
| SHOW_SYNERGY_HINT | true | true/false | 是否显示实时协同提示（关闭可测试"盲选"难度） |
| SYNERGY_SCORE_VISIBLE | true | true/false | 是否显示协同强度总分 |
| DRAFT_SEED | 0 | 0~9999 | 随机种子（0=纯随机，固定值=可复现的测试场景） |

## 选填区：构筑/卡组类

### 卡池定义（核心子集，18 张）

| 卡 ID | 名称 | 费用 | 类型 | 标签 | 基础效果 | 副本数 |
|-------|------|------|------|------|---------|--------|
| C01 | 烈焰斩 | 2 | 攻击 | [火] | 造成 3 点伤害 | 3 |
| C02 | 护盾术 | 1 | 防御 | [护] | 获得 2 点护盾 | 3 |
| C03 | 燃烧印记 | 1 | 触发 | [火][印] | 持有时，[火]牌效果+1 | 2 |
| C04 | 铁壁 | 3 | 防御 | [护][重] | 获得 5 点护盾 | 2 |
| C05 | 连锁闪电 | 3 | 攻击 | [雷] | 造成 2 点伤害，若已有[雷]牌则再造成 2 点 | 2 |
| C06 | 雷芒附魔 | 2 | 增益 | [雷][印] | 持有时，[雷]牌附加 1 点穿甲 | 2 |
| C07 | 反弹甲 | 2 | 防御 | [护][反] | 获得 1 点护盾；受攻击时反弹 1 点 | 2 |
| C08 | 急速咒 | 1 | 增益 | [速] | 本回合下一张牌费用-1 | 2 |

### 协同设计（关键标签对）

| 标签对 | 协同描述 | 触发条件 |
|--------|---------|---------|
| [火] × [印] | "燃烧印记"激活：所有[火]牌效果+1 | 同时持有 C03 + 任意[火]牌 |
| [雷] × [印] | "雷芒附魔"激活：所有[雷]牌穿甲+1 | 同时持有 C06 + 任意[雷]牌 |
| [护] × [反] | "反弹甲"强化：护盾量≥4 时反弹+1 | 同时持有 C07 + 总护盾来源≥2张 |
| [火] × [雷] | 无直接协同（设计意图：两路互斥，测试是否存在混搭最优解） | — |

### 预期测试路线

```
路线A：全火路（C01×2 + C03×2 + C08）→ 高爆发，低容错
路线B：全护路（C02×3 + C04 + C07）→ 低输出，高生存
路线C：雷印路（C05×2 + C06×2 + C08）→ 中等爆发，穿甲优势
验证目标：三路是否都"有人选、有赢面"，无单一压倒性路线
```

### 稀有度/抽取权重

```
本原型不设稀有度（验证协同张力优先），所有卡均权重相同。
后续版本若需要引入稀有度，在 POOL_SIZE × 权重表扩展。
```

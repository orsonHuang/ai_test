# 通用原型描述模板 · 叙事/剧情类填写示例

> **用途**：本文件是 `01-prototype-input-template.md` 的叙事/剧情类完整填写参考。
> **对应 Spec 模板**：`spec-templates/spec-template-narrative.md`
> 参照场景：RPG 事件卡的选择分支系统（单次事件，3 个关键选择节点，多结局）。
> 读者可对照本文件理解"填到什么程度 AI 才能动"。
> **填写难度信号**：本示例填写完整度 = 100%。如果你看完觉得「自己填不到这个程度」，建议改走 `01-input-elicitation-protocol.md` 的问答补齐流程——AI 会通过对话帮你补齐。

## 必填区：所有原型通用

### A. 验证目标

- **要验证的核心假设是什么？**
  玩家在关键选择节点是否会感受到"这个选择有代价"——即每个选项不是"一眼最优"，玩家会犹豫、权衡，甚至在结局后想"如果当时选另一个会怎样"。

- **验证成功的判据是什么？**
  3 位不同试玩者中，至少 2 位在同一节点停顿 ≥5 秒；或至少 2 位试玩后主动询问"其他选项是什么结果"。

- **不需要验证什么？**
  不需要验证：美术表现、音效、语音、角色立绘、完整剧情世界观。只验证选择的重量感和分支差异的可感知性。

---

### B. 核心交互

- **用户做什么操作？**
  阅读事件描述文字 → 点击选项按钮（2~3 个） → 看到结果文字 → 继续阅读下一段 → 重复直到结局

- **系统如何响应？**
  点选后：隐藏选项区 → 显示结果文字 → 更新状态变量（HP/资源/关系值）→ 短暂停顿 → 呈现下一节点 / 结局

- **交互循环（用箭头表示）**
  ```
  展示节点文字 → 等待玩家选择 → 播放结果文字 → 更新变量 → 进入下一节点（或触发结局）→ 展示结局页
  ```

---

### C. 数据对象

| 对象名 | 关键属性 | 与其他对象的关系 |
|--------|----------|------------------|
| Node | id, title, bodyText, choices[], isEnding | 构成故事树；每个 Node 持有若干 Choice |
| Choice | id, label, resultText, nextNodeId, effects[] | 属于 Node；effects 写入 PlayerState |
| PlayerState | hp, gold, reputation, flags{} | 全局唯一；被 Choice.effects 读写；flags 用于条件分支 |
| Effect | type(stat/flag), key, delta | 属于 Choice；执行时修改 PlayerState |

---

### D. 状态与流程

- **系统有哪些状态？**
  `start`（事件开始）→ `reading`（展示节点，等待选择）→ `resolving`（播放结果文字）→ `transition`（加载下一节点）→ `ending`（展示结局）

- **状态之间如何切换？**
  ```
  start        → （渲染第一个节点完毕）           → reading
  reading      → （玩家点击选项）                 → resolving
  resolving    → （结果文字展示完毕）              → transition
  transition   → （nextNodeId = ENDING_X）        → ending
  transition   → （nextNodeId = 普通节点）         → reading
  ending       → （玩家点击"重来"）               → start
  ```

- **初始状态**：`start`，PlayerState = { hp: 100, gold: 50, reputation: 0, flags: {} }
- **结束状态**：`ending`，展示结局标题 + 结局文字 + 关键状态变量最终值

---

### E. 核心规则

```
- 规则1：每个节点至少 2 个、最多 4 个选项
- 规则2：条件选项（需要 flags.X = true 才显示）允许存在，不满足条件时隐藏
- 规则3：结局类型共 3 种，由哪些关键选择决定（不是累加数值决定），减少"数值堆最优"
- 规则4：同一节点不允许两个选项产生完全相同的 effects（避免"伪选择"）
- 边界条件（必须不能发生的事）：
    - 不允许出现死节点（nextNodeId 不存在）
    - 不允许 HP ≤ 0 时游戏还能继续（自动触发 ENDING_DEAD）
    - 不允许选项区在 resolving 状态仍可点击
- 特殊情况处理：
    - 条件选项全部被隐藏时，至少保留 1 个无条件选项
    - flags 冲突（同时设置互斥 flag）→ 后写覆盖前写，记录日志
```

---

### F. UI 骨架

- **整体布局**：
  居中单栏（最大宽 600px），上方状态栏（HP/金币/声望）+ 中央文字区（事件描述）+ 下方选项按钮列表 + 右下角调试面板

- **关键视觉元素位置**：
  - 状态栏：顶部固定，3 个资源用图标+数字呈现
  - 事件描述：白底卡片，衬线字体，适中行距
  - 选项按钮：纵向排列，每个按钮显示选项文字；鼠标悬停时显示预期效果提示（可调参数控制是否显示）
  - 结果文字：斜体，灰色背景，与事件描述区区分

- **信息展示优先级**：
  1. 事件描述/结果文字（核心内容）
  2. 选项按钮（行动入口）
  3. 当前状态变量（决策依据）
  4. 节点路径追踪（调试用）

---

### G. 可调参数（调试面板数据源）

| 参数名 | 默认值 | 可调范围 | 说明 |
|--------|--------|----------|------|
| SHOW_EFFECT_HINT | true | true/false | 悬停选项时是否显示效果预览（影响"犹豫感"验证） |
| TEXT_REVEAL_SPEED | 30 | 0~100 ms/字 | 文字逐字出现速度，0=立即全显 |
| INITIAL_HP | 100 | 10~200 | 初始生命值 |
| INITIAL_GOLD | 50 | 0~200 | 初始金币 |
| AUTO_ADVANCE_MS | 0 | 0~3000 | 结果文字展示后自动进入下一节点的延迟（0=手动点击） |

## 选填区：叙事/剧情类

### 故事树结构

- **节点总数**：约 8~10 个（含 3 个结局节点）
- **树形 or DAG**：DAG（部分节点可从不同路径到达）
- **结局数量与类型**：
  - ENDING_HERO：声望 ≥ 3，最终感化首领
  - ENDING_EXILE：gold ≥ 80，用金钱买通离开
  - ENDING_DEAD：HP ≤ 0，战斗失败

### 关键选择节点

| 节点编号 | 选择场景描述 | 选项数 | 后续影响维度 |
|----------|------------|--------|------------|
| N01 | 遭遇强盗拦路——谈判/战斗/逃跑 | 3 | reputation ± / hp - / gold - |
| N04 | 村民请求帮助——答应/拒绝/敷衍 | 3 | reputation + / flags.helped_village |
| N07 | 首领最终对峙——说服/贿赂/交战 | 3（条件：flags.helped_village 解锁第4项） | 决定结局走向 |

### 情感曲线设计

```
节点1（平静入场）→ 节点2（首次危机，轻度）→ 节点3-4（信息堆积，压力上升）
→ 节点5（假缓解）→ 节点6-7（最终对峙，高压）→ 结局（释放）
```

### 变量设计说明

- **数值型变量**（hp, gold, reputation）：量化后果，让玩家感知选择代价
- **旗帜型变量**（flags）：解锁/封锁选项，制造"前因后果"感
- **不使用评分制结局**：避免玩家把故事变成"刷分游戏"，用关键旗帜决定结局路径

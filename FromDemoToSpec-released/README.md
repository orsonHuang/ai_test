# AI 驱动原型验证与 Spec 沉淀工作流

> 副标题：把"从不验证"变成"低成本高频验证"

---

## 导语 · 这是什么

**FromDemoToSpec（F2S）** 是一套面向游戏/产品设计者的 AI 协作工作流，把"从一个想法到可玩原型再到结构化设计文档"的全过程标准化。

**能做什么**：

- 把口述的玩法/功能想法 → AI 分钟级出可玩 HTML 原型 + MVP 设计文档
- 试玩调参后 → AI 自动提取为结构化 Spec 文档，可直接对接开发排期
- 适用 11 类通用原型（战斗/即时制/三消/经济/构筑/叙事/活动/工具/UX/数据看板/表单流程）

**当前可用指令**（详见 `00-command-manual.md`）：

| 指令 | 用途 |
|------|------|
| **开始原型验证** | 通过自然对话补齐输入信息 → 生成原型 + MVP 文档 + Input 文档 |
| **修改Demo** | 对 `Demo/` 目录内现有原型文件进行修改/迭代 |
| **开始 Spec 提取** | 原型验证通过后 → AI 自动提取 Spec 文档 + 预填策划文档框架 |
| **准备交付** | Spec 提取完成后 → 冲突扫描 + 归档 + 生成交付说明，交下游实现 |

**推荐起步**：直接说「**开始原型验证**」即可（或类似表达，如"我想做个原型"、"我有个想法想验证下"）。AI 会自动进入问答流程。

## AI 接手路由

> AI 在此项目协作时的文件读取顺序。

**新会话启动**：

1. `00-ai-collaboration-principles.md` — 协作原则（澄清优先、信念度标注等） + 输出约定速查（§6 产物命名）
2. `00-command-manual.md` — 可用指令清单（确保识别用户指令）
3. 按用户意图定位到对应规范文档

| 时机 | 动作 |
|------|------|
| 生成原型前 | `02-html-prototype-spec.md` 全文 — 逐条对照检查 |
| 生成后 | `00-file-naming-convention.md` — 确认文件命名 + 路径 |
| 提取 Spec 前 | `03-prototype-to-spec-mapping.md` + MVP 文档 + 对应 `spec-templates/` 模板 |
| 询问输入信息时 | `01-input-elicitation-protocol.md` — 问答补齐协议 |

**按需深入**：`00-ai-collaboration-principles.md`（协作原则）、`00-file-naming-convention.md`（文件命名规范）、`01-prototype-input-template.md`（输入模板）、`02-mvp-document-spec.md`（MVP 文档生成规范）、`03-design-doc-template-game.md`（游戏类设计文档模板）、`03-design-doc-template-product.md`（产品类设计文档模板）、`03-design-doc-template-project.md`（落地策划案模板）、`input-examples/`（填写示例）、`spec-templates/`（Spec 模板）。

## 一、三层架构

```
┌──────────────────────────────────────────────────┐
│  第一层：规范层 · 约束 AI + 收束设计者              │
│                                                  │

│  ├─ 03 原型→Spec 映射规范 + 配置表                  │
│  ├─ 00 AI 协作原则（降低沟通摩擦）                   │
│  └─ 00 Input 问答补齐协议（自然对话补齐 02 输入）    │
│                                                  │
│  （已有）策划文档模板（公司现有）                    │
└─────────────────────┬────────────────────────────┘
                      │ 作为 System Prompt / Context 注入 AI
                      ▼
┌──────────────────────────────────────────────────┐
│  第二层：生成层 · AI 按规范出原型                   │
│                                                  │
│  输入：玩法/功能描述（按输入模板填写 或 走 00 问答补齐）│
│  输出：├─ 可玩 HTML 原型                           │
│        │   ├─ 核心逻辑（玩法/系统/交互）            │
│        │   ├─ 调试面板（状态查看/参数热调/日志）    │
│        │   └─ 符合编写规范（结构统一/可迭代）       │
│        ├─ Input 文档（问答结果+跳过项与处理方式）   │
│        └─ MVP 设计文档（验证后生成）                │
└─────────────────────┬────────────────────────────┘
                      │ 试玩验证后

┌──────────────────────────────────────────────────┐
│  第三层：沉淀层 · 原型 → 文档自动提取 → 交付收口    │
│                                                  │
│  ┌─ [沉淀]                                       │
│  │  输入：├─ 已验证的 HTML 原型                    │
│  │        ├─ MVP 设计文档（设计意图+调参记录+验收结论）│
│  │        └─ 试玩调整记录                          │
│  │  输出：├─ Spec 文档（AI 自动提取：规则/数值/机制）│

│      前置检查：冲突扫描 → 填写补齐 → 状态升级 → 范围确认 │
│      收口动作：归档过期 → 生成交付说明 → 更新 README   │
│      输出：└─ 交付说明.md（下游唯一入口）            │
└──────────────────────────────────────────────────┘
```

## 三、完整工作流

```
玩法想法（自然语言/要点描述）
        │
        ▼
A 路径：设计者按 02 填写原型描述模板（参考 input-examples/）
B 路径：AI 走 00 问答补齐协议（开放题→类型识别→gap追问→跳过项处理）
        │
        ▼
得到 Input 文档（含跳过项与处理方式记录）
        │
        ▼
AI 按 01+02+00 生成 HTML 原型 V1 + MVP 文档 V1
        │
        ▼
设计者试玩 + 调试面板调参 + 记录问题
        │
        ▼
AI 按 00 给调整建议（带信念度标注）→ 设计者拍板
        │
        ▼
AI 生成 V2 原型 + 同步更新 MVP 文档（循环至验证假设成立）
        │
        ▼
AI 按 03 自动提取 → 填入 spec-templates/ 对应模板 → 生成 Spec 文档
   + 同步预填 04 策划文档（结构信息 + 参数表 + 已知问题，标注"待策划填写"段）
        │
        ▼
策划补填设计意图 + 验证结论 + 体验目标 → 策划文档完成
        │
        ▼
执行「准备交付」指令
  ├─ 收口前置检查（冲突扫描/填写补齐/状态升级/范围确认）
  ├─ 归档过期文件（mvp.md 全部 / input.md / 对话记录 → 归档/）
  └─ 生成交付说明.md → 交下游实现
```

## 四、泛用性说明

本工作流不限于战斗原型，通过通用输入模板的类型扩展模块，同样适用于：

**游戏类**
- **战斗/对抗类**（回合制）：战棋、卡牌对战等
- **即时制/实时驱动类**：塔防、自动战斗（Auto-battler）、无限滚动等
- **三消类**：含回合制（步数限制）和即时制（计时器）两种变体
- **活动/小玩法类**：限时活动、小游戏、积分赛等
- **叙事/剧情类**：RPG 事件卡、视觉小说分支、选择型对话树等
- **经济/资源流类**：城建/放置类资源循环、货币通胀测试、升级曲线验证等
- **构筑/卡组类**：轮抽草稿、卡组协同验证、稀有度分布测试等

**产品/工具类**
- **系统/工具类**：配置工具、数据处理、流程工具等
- **产品/UX 类**：交互原型、页面流程、导航结构等
- **数据可视化/仪表板类**：运营看板、指标异常检测、决策效率验证等
- **表单/流程类**：多步骤表单、条件字段、注册/创建/审批流程等

## 五、与现有流程的关系

本工具包的输出物是**正式开发流程的输入**，不是替代品：

- Spec 文档 → 可作为程序需求排期的依据
- 策划文档 → 与团队现有文档模板兼容
- HTML 原型 → 可作为程序理解需求的参考实现

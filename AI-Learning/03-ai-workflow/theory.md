# 03 — AI工作流：Agent + 工具调用 + 游戏事件系统

> 目标：从"会说话"的NPC升级到"会做事"的NPC/系统

## 1. 前两个阶段的局限

| 阶段 | NPC能力 | 还缺什么 |
|------|---------|---------|
| 01 | 能对话，有性格 | 只能"说"，不能"做" |
| 02 | 能查知识库回答 | 还是只能"说"，回答基于已有文档 |
| **03** | **能行动** | 调用游戏函数、触发事件、改变世界状态 |

**一句话：** 01和02的NPC是顾问，03的NPC是执行官。

## 2. Function Calling（工具调用）

### 核心概念

LLM 本身只能生成文本。但游戏里的NPC需要"做事"——查资源、给物品、触发任务。

**Function Calling** 就是让 LLM 输出一个"调用指令"，而不是纯文本回复。

### 工作流程

```
玩家说："帮我查一下基地还有多少铁矿石"
    ↓
LLM 判断：这是一个查询请求，应该调用 check_resource 函数
    ↓
LLM 输出（不是文字回复，而是函数调用指令）：
    {
        "function": "check_resource",
        "arguments": {"resource_type": "iron", "location": "base"}
    }
    ↓
你的代码执行这个函数，返回结果：
    "铁矿石库存：23 单位"
    ↓
LLM 拿到结果，生成自然语言回复：
    "指挥官，铁矿石库存23单位。建议暂停采矿站扩建，先修通讯天线。"
```

### 关键设计决策

**给NPC哪些工具？**

不是越多越好。每个工具都需要：
1. 清晰的描述（LLM才知道什么时候该调用）
2. 明确的参数格式（LLM才知道怎么传参）
3. 可预测的返回值（LLM才知道怎么解读结果）

**《遥远行星》的工具设计示例：**

| 工具名 | 功能 | 参数 | 返回 |
|--------|------|------|------|
| check_resource | 查资源库存 | resource_type, location | 数量+状态 |
| get_facility_status | 查设施状态 | facility_name | 运行状态+效率 |
| trigger_event | 触发世界事件 | event_type, intensity | 事件ID+影响描述 |
| generate_mission | 生成交线任务 | difficulty, theme | 任务详情JSON |

## 3. ReAct：思考→行动→观察→循环

### 为什么需要循环？

单次调用不够。比如：
- 玩家说"帮我准备一次远征"
- NPC需要：先查资源 → 发现铁不够 → 建议先采矿 → 确认采矿站状态 → 才能给出完整方案

这就是 **ReAct**（Reason + Act）模式：

```
思考(Reason)：玩家要远征，我需要先确认资源情况
行动(Act)：调用 check_resource("all", "base")
观察(Observe)：铁矿石23，冰晶5，暗能量碎片0
思考：暗能量碎片为0，远征需要至少2碎片作为能源
行动：调用 generate_mission("采矿", "dark_shard")
观察：生成任务"深入陨石坑采集暗能量碎片"
思考：现在可以给出完整建议了
最终回复："指挥官，远征准备需要暗能量碎片..."
```

### 跟01阶段的区别

01阶段：玩家问 → LLM直接回答（一轮）
03阶段：玩家问 → LLM思考 → 调工具 → 看结果 → 再思考 → 可能再调工具 → 最终回答（多轮）

## 4. Agent 工作流在游戏中的三种形态

### 形态A：单Agent + 工具（我们这次做的）

一个NPC（阿尔法）带几个工具，能查状态、触发事件。
- 简单、可控、适合独立游戏
- 阿尔法既是"嘴"又是"手"

### 形态B：多Agent协作

```
规划Agent：分析世界状态，决定今天发生什么
执行Agent：根据规划触发具体事件
审核Agent：检查事件是否符合世界观、难度合理
```

- 更精细、更像"游戏导演"
- 成本更高（3次API调用才出一个事件）
- 04阶段会接触

### 形态C：自动工作流（无玩家输入）

后台定时运行，驱动世界演化：
- 每5分钟检查世界状态 → 自动生成事件
- NPC按日程行动 → 产出变化 → 市场价格波动
- 类似 RimWorld 的 Storyteller

## 5. 本次实战项目：动态事件生成器

**功能：** 玩家描述当前游戏状态 → Agent自动查询+思考 → 生成合适的支线任务或世界事件

**流程：**
1. 玿家输入当前进度描述
2. Agent调用 check_world_status 获取世界状态
3. Agent调用 check_player_progress 获取玩家进度
4. Agent综合分析，调用 generate_event 生成事件
5. 返回完整的事件描述（标题、触发条件、奖励、后果、NPC对话预览）

# 02 — 游戏知识库：RAG + NPC记忆与世界观

> 目标：让AI-NPC能"读"游戏设定文档，基于官方世界观回答问题

---

## 1. 为什么需要 RAG？

### 现在的问题

阿尔法现在只知道 System Prompt 里写的几行设定。如果你问他：
- "这个星球的原住民是什么？"
- "基地附近有什么资源？"
- "通讯天线的修复需要什么材料？"

他要么编一个（幻觉），要么说"不知道"。

### RAG 的作用

```
玩家问问题
    ↓
从游戏设定文档中检索相关段落
    ↓
把段落塞进 Prompt
    ↓
LLM 基于检索到的内容回答
```

**一句话理解：** RAG = 给 AI 开卷考试，让它基于"教材"答题，而不是凭记忆瞎编。

---

## 2. RAG 三步走

### Step 1：Embedding（向量化）

把每段文本变成一个数字向量（比如1536维的数组），语义相近的文本向量距离近。

```
"星球大气含氧量低" → [0.12, -0.34, 0.56, ...] 
"空气中氧气稀薄"   → [0.11, -0.32, 0.55, ...]  ← 语义相近，向量也近
"基地有氧气发生器" → [0.78, 0.21, -0.43, ...]  ← 语义不同，向量远
```

### Step 2：检索

玩家问"星球上能呼吸吗？"→ 也变成向量 → 跟所有文档段落比距离 → 取最相近的几段。

### Step 3：增强生成

把检索到的段落 + 玩家问题一起发给 LLM：
```
System: 你是阿尔法。以下是相关设定资料：
【检索到的文档段落1】
【检索到的文档段落2】
请基于以上资料回答玩家问题。如果资料中没有答案，说"我的数据库中没有相关信息"。

User: 星球上能呼吸吗？
```

## 3. 关键技术点

### 文档分块（Chunking）
- 不能把整篇文档扔进去（太长、太贵、检索不准）
- 按段落/标题切分，每块 200-500 字
- 保留上下文（比如标题层级）

### 向量数据库
- **ChromaDB**：轻量、本地、Python 友好（我们用这个）
- FAISS：Facebook 出的，更快但配置复杂
- Milvus：生产级，适合大规模

### NPC 记忆（进阶）
- 短期记忆：最近 N 轮对话
- 长期记忆：把旧对话总结成摘要存起来
- 关系记忆："玩家曾经帮过我" → 影响态度
